## Що ви зможете зробити

Після цього модуля ви зможете:
- **Створювати та редагувати** файли за допомогою `nano` прямо з термінала без графічного редактора
- **Написати** простий bash-скрипт, який об’єднує кілька команд в один файл
- **Зробити** скрипт виконуваним за допомогою `chmod +x` та пояснити, навіщо потрібен цей крок
- **Обирати** між `nano` та `vim` і пояснювати, коли варто використовувати кожен із них

---

## Чому це важливо

У минулому модулі ви створювали файли за допомогою `touch` — але вони були порожніми. Порожні файли подібні до чистих бланків замовлень на кухні ресторану. Вони корисні для резервування місця, але не приносять багато користі, поки хтось у них щось не запише.

У реальному світі вам доведеться постійно редагувати файли:
- **Конфігураційні файли** вказують програмам, як поводитися (як правила внутрішнього розпорядку на кухні)
- **Скрипти** — це файли, що містять послідовність команд (як картка з рецептом)
- **Kubernetes manifests** — це файли, які вказують Kubernetes, що саме запускати (ви до цього ще дійдете!)

Розглянемо випадок із життя великої технологічної компанії: молодший інженер мав оновити просте налаштування в load balancer. Він відкрив файл `nginx.conf` у `nano`, випадково ввів кілька зайвих символів під час спроби збереження і зберіг файл. За лічені хвилини весь сайт «ліг», що призвело до втрати тисяч доларів прибутку. Чому? Тому що він редагував файл безпосередньо на робочому (production) сервері, не усвідомлюючи цього, а одна друкарська помилка в конфігураційному файлі може вивести сервіс із ладу.

Вам потрібно вміти відкрити файл, написати в ньому щось, зберегти та закрити його — і все це з термінала. Без миші. Без графічного текстового редактора. Тільки ви та клавіатура. Навчитися безпечно редагувати файли в терміналі — це не просто про написання тексту; це про те, як точно керувати «пультом управління» ваших серверів.

## Навіщо редагувати файли в терміналі?

«Хіба я не можу просто використовувати Блокнот чи TextEdit?»

Можете — на власному комп’ютері. Але згадайте Модуль 0.1: майже кожен сервер працює на Linux, і сервери зазвичай не мають графічного інтерфейсу. Коли ви підключаєтеся до віддаленого сервера (про що ви дізнаєтеся в Модулі 0.7), там немає ні миші, ні робочого столу, ні Блокнота. Є лише термінал.

Текстовий редактор у терміналі — це ніж шеф-кухаря у світі обчислень. Це не найвишуканіший інструмент, але ви будете використовувати його всюди.

## Знайомство з nano: ваш перший термінальний редактор

Існує багато текстових редакторів, які працюють у терміналі. Два найвідоміші — це **vim** та **nano**.

Ми починаємо з **nano**, тому що:

| nano | vim |
|------|-----|
| Працює так, як ви очікуєте | Має складну криву навчання |
| Ви просто вводите текст | Потрібно натиснути `i`, перш ніж почати вводити текст |
| Меню внизу підказує, як зберегти та вийти | Люди часто «застрягають» і не можуть зрозуміти, як вийти |
| Ідеальний для початківців | Потужний, але спочатку приголомшує |

Використовувати nano не соромно. Багато досвідчених інженерів використовують його для швидких правок. Ви вивчите vim пізніше, коли будете готові до більших можливостей.

> **Цікавий факт**: «Як вийти з vim» — одне з найпопулярніших запитань про програмування в інтернеті. Понад 2 мільйони людей переглянули це питання на Stack Overflow. З nano у вас ніколи не буде такої проблеми.

## Відкриття nano

Давайте створимо та відредагуємо файл. Спочатку переконайтеся, що ви перебуваєте в домашній директорії:

```bash
cd ~
```

Тепер відкрийте nano з новим файлом:

```bash
nano hello.txt
```

Ваш екран повністю зміниться. Ви побачите щось на кшталт цього:

```
  GNU nano 6.2                    hello.txt







^G Help    ^O Write Out  ^W Where Is   ^K Cut       ^C Location
^X Exit    ^R Read File  ^\ Replace    ^U Paste     ^T Execute
```

Давайте розберемо це:

- **Верхній рядок**: Назва редактора та назва вашого файлу
- **Середня область**: Тут ви вводите текст (вона порожня, бо файл новий)
- **Нижні два рядки**: Меню з доступними командами

---

## Символ ^ означає Ctrl

Це те, що спочатку плутає всіх:

> **Символ `^` означає «утримуйте клавішу Ctrl».**

Отже, коли ви бачите `^O Write Out`, це означає: «Натисніть одночасно Ctrl та O, щоб зберегти файл».

| Те, що ви бачите | Що натиснути | Що це робить |
|------------------|--------------|--------------|
| `^O` | Ctrl + O | Зберегти файл |
| `^X` | Ctrl + X | Вийти з nano |
| `^K` | Ctrl + K | Вирізати поточний рядок |
| `^U` | Ctrl + U | Вставити вирізаний рядок |
| `^W` | Ctrl + W | Пошук тексту |
| `^G` | Ctrl + G | Показати довідку |

Ось і все. Шість комбінацій клавіш — і ви можете робити все необхідне.

---

## Введення тексту

Це найпростіша частина: **просто пишіть.**

Відкривши nano, введіть наступне (натискайте Enter в кінці кожного рядка):

```
Welcome to the Kitchen!
Today's special: Learning to edit files.
Chef says: You're doing great.
```

Ваш екран має виглядати так:

```
  GNU nano 6.2                    hello.txt                    Modified

Welcome to the Kitchen!
Today's special: Learning to edit files.
Chef says: You're doing great.


^G Help    ^O Write Out  ^W Where Is   ^K Cut       ^C Location
^X Exit    ^R Read File  ^\ Replace    ^U Paste     ^T Execute
```

Зверніть увагу на слово **Modified** у верхньому рядку. Це означає, що ви внесли зміни, які ще не збережені.

---

## Збереження: Ctrl + O

Давайте збережемо файл. Натисніть:

```
Ctrl + O
```

nano попросить підтвердити назву файлу:

```
File Name to Write: hello.txt
```

Натисніть **Enter** для підтвердження. Індикатор «Modified» зникне. Ваш файл збережено.

---

## Вихід: Ctrl + X

Щоб вийти з nano та повернутися в термінал:

```
Ctrl + X
```

Якщо ви внесли зміни після останнього збереження, nano запитає:

```
Save modified buffer?  Y Yes  N No  ^C Cancel
```

Натисніть `Y` (Yes), щоб зберегти та вийти, `N` (No), щоб вийти без збереження, або `Ctrl + C`, щоб скасувати вихід і залишитися в nano.

---

## Перевірка вашого файлу: `cat`

Тепер, коли ви знову в терміналі, давайте підтвердимо, що файл зберігся. Команда `cat` виводить вміст файлу на екран:

```bash
cat hello.txt
```

Очікуваний результат:

```
Welcome to the Kitchen!
Today's special: Learning to edit files.
Chef says: You're doing great.
```

`cat` — це скорочення від «concatenate» (вона може об’єднувати кілька файлів), але більшість людей використовують її, щоб швидко зазирнути всередину файлу. Уявіть, що ви читаєте записку, не беручи її в руки — просто кидаєте швидкий погляд.

---

## Редагування існуючого файлу

Щоб відредагувати файл, який уже існує, просто відкрийте його так само:

```bash
nano hello.txt
```

Ваш текст на місці. Використовуйте клавіші зі стрілками для переміщення. Друкуйте, щоб додати текст. Використовуйте Backspace/Delete, щоб видалити текст.

Додайте новий рядок внизу:

```
PS: The pantry is fully stocked.
```

Збережіть за допомогою `Ctrl + O`, Enter. Вийдіть за допомогою `Ctrl + X`.

## Вирізання та вставка рядків

nano має прості команди для вирізання та вставки цілих рядків:

1. Наведіть курсор на рядок, який хочете вирізати
2. Натисніть **Ctrl + K**, щоб вирізати його (рядок зникне)
3. Перемістіть курсор туди, де хочете вставити рядок
4. Натисніть **Ctrl + U**, щоб вставити його

Ви можете вирізати кілька рядків, натиснувши `Ctrl + K` кілька разів — вони накопичуються. Потім `Ctrl + U` вставить їх усі разом.

Це схоже на те, як шеф-кухар змінює порядок кроків у рецепті.

---

## Пошук тексту

Працюєте з довгим файлом і потрібно щось знайти? Натисніть **Ctrl + W**, введіть те, що шукаєте, і натисніть Enter.

```
Ctrl + W
```

nano виведе запит:

```
Search:
```

Введіть `special` і натисніть Enter. Курсор перескочить до першої появи слова «special» у файлі.

Натисніть `Ctrl + W`, а потім знову Enter (нічого не вводячи), щоб знайти наступний збіг.

## Ваш перший скрипт

Тепер давайте зробимо дещо потужне: напишемо скрипт. Скрипт — це просто текстовий файл, який містить команди, які комп’ютер може виконати. Це картка з рецептом для термінала.

### Крок 1: Створіть скрипт

```bash
nano my-first-script.sh
```

> **Зупиніться та подумайте**: Цей скрипт починається з дивного рядка: `#!/bin/bash`. Як ви думаєте, що він робить? Він називається «shebang» (шебанг) — це спосіб скрипта сказати комп’ютеру: «Я написаний на bash, використовуй програму bash, щоб мене запустити». Без нього комп’ютер не знав би, як інтерпретувати файл. Ви бачитимете цей рядок на початку кожного shell-скрипта.

Введіть точно наступне:

```bash
#!/bin/bash

echo "Welcome to the kitchen!"
echo "Today's date is: $(date)"
echo "You are logged in as: $(whoami)"
echo "Your current directory is: $(pwd)"
echo ""
echo "Great job, chef! Your first script works!"
```

Давайте розберемо кожен рядок:

- `#!/bin/bash` — Це «shebang». Він каже комп’ютеру: «використовуй програму bash для запуску цього файлу». Кожному скрипту це потрібно в першому рядку.
- `echo` — Команда, яка виводить текст на екран. Уявіть, що на кухні вигукують: «Замовлення готове!»
- `$(date)` — Запускає команду `date` і вставляє результат. Синтаксис `$()` означає «запусти цю команду і дай мені результат».
- `$(whoami)` — Запускає `whoami`, яка повідомляє ваше ім’я користувача.
- `$(pwd)` — Ви вже це знаєте — виводить вашу поточну директорію.

### Крок 2: Збережіть та вийдіть

Натисніть `Ctrl + O`, Enter, потім `Ctrl + X`.

### Крок 3: Зробіть його виконуваним

> **Зупиніться та подумайте**: Перед запуском `chmod` спробуйте запустити скрипт напряму: `./my-first-script.sh`. Що сталося? Ви маєте отримати «Permission denied». Це зроблено навмисно — за замовчуванням нові файли сприймаються лише як дані. Комп’ютеру потрібен явний дозвіл, щоб поводитися з файлом як з програмою. Це функція безпеки, а не помилка.

Зараз цей файл — просто текст. Комп’ютер не запустить його, бо він не має *прав* на виконання. Давайте це виправимо:

```bash
chmod +x my-first-script.sh
```

`chmod` означає **ch**ange **mod**e (змінити режим). `+x` означає «додати право на виконання». Ви кажете менеджеру ресторану: «Це не просто документ, це рецепт, за яким треба готувати».

### Крок 4: Запустіть його

```bash
./my-first-script.sh
```

`./` означає «запусти цей файл із поточної директорії». Очікуваний результат:

```
Welcome to the kitchen!
Today's date is: Sun Mar 23 14:30:00 UTC 2026
You are logged in as: yourname
Your current directory is: /Users/yourname

Great job, chef! Your first script works!
```

Ви щойно написали та виконали свою першу програму. Це не дрібниця — це великий крок.

## Кілька слів про vim

Рано чи пізно ви зустрінетеся з vim. Він неймовірно потужний, ним користуються багато досвідчених інженерів, і він встановлений практично на кожній системі Linux. Але vim відомий своєю крутою кривою навчання, тому що він працює в «режимах» — ви не можете просто відкрити його і почати писати.

Ми не ігноруємо vim. Ми приберігаємо його на потім, коли у вас буде більше досвіду. Наразі nano справляється із завданнями, а вчитися ходити перед тим, як бігати — це те, як тренуються і шеф-кухарі.

Коли ви будете готові, vim буде чекати на вас. А поки що nano — ваш друг.

## Чи знали ви?

- **nano — це клон клона.** Він був створений як вільна заміна редактора під назвою `pico`, який був вбудований у поштовий клієнт Pine у 1990-х роках. `nano` розшифровується як «nano's ANOther editor» — рекурсивна абревіатура, що є ботанічною традицією у програмуванні.

- **Найстаріший текстовий редактор, який досі використовується — це `ed`, створений у 1969 році.** Його написав Кен Томпсон, один із творців Unix. `ed` — це «рядковий редактор» (line editor), у ньому ви не можете бачити весь файл одночасно. Ви редагуєте один рядок за раз. Інженери у 1970-х писали цілі операційні системи за допомогою `ed`. Ваш досвід із nano — це розкіш у порівнянні з цим.

- **Конфігураційні файли керують світом.** Майже кожна програма зчитує текстовий конфігураційний файл під час запуску. Ваш вебсервер, ваша база даних, сам Kubernetes — усі вони читають текстові файли, щоб знати, як поводитися. Вміння редагувати ці файли з термінала — одна з найпрактичніших навичок у всій комп’ютерній сфері.

## Типові помилки

| Помилка | Чому це проблема | Що робити замість цього |
|---------|------------------|-------------------------|
| Натискання Ctrl+Z замість Ctrl+X для виходу | Ctrl+Z призупиняє nano (ховає його), а не закриває. Файл залишається відкритим у фоні | Якщо ви випадково призупинили, введіть `fg`, щоб повернути nano. Потім використайте Ctrl+X для правильного виходу |
| Забули зберегти перед виходом | Ваші зміни будуть втрачені | Завжди натискайте Ctrl+O для збереження перед Ctrl+X для виходу. Або просто натисніть Ctrl+X і введіть Y на запит про збереження |
| Відсутність `#!/bin/bash` у скриптах | Система не знає, як запускати файл | Завжди робіть перший рядок bash-скрипта `#!/bin/bash` |
| Забули `chmod +x` перед запуском скрипта | Ви отримаєте «Permission denied» | Запустіть `chmod +x filename.sh`, щоб зробити файл виконуваним |
| Використання `nano`, коли мали на увазі `cat` | Ви випадково відкриваєте файл для редагування, хоча хотіли лише прочитати його | Використовуйте `cat filename`, щоб переглянути файл без редагування |
| Редагування файлу на віддаленому сервері, вважаючи його локальним | Ви можете випадково змінити конфігурацію production замість локальних тестових файлів, що призведе до простою | Завжди перевіряйте ваш prompt (наприклад, `user@server`) або запустіть `hostname`, щоб підтвердити, на якій машині ви редагуєте файли |
| Відкриття бінарного файлу (як зображення чи скомпілована програма) в nano та його збереження | Nano спробує прочитати бінарні дані як текст, і збереження назавжди зіпсує файл | Використовуйте nano тільки для звичайних текстових файлів (скрипти, конфіги, логи). Використовуйте `file filename`, щоб перевірити тип файлу перед відкриттям |

   <details>
   <summary>Відповідь</summary>
   Ви сприйняли символ `^` буквально, замість того щоб використати його як клавішу-модифікатор. У термінальних програмах, таких як nano, символ `^` представляє клавішу Ctrl (Control). Таким чином, `^O` означає, що ви повинні утримувати клавішу Ctrl і одночасно натиснути літеру O. Це стандартна конвенція в Unix-подібних системах для позначення комбінацій клавіш. Вводячи символи окремо, ви просто додавали текст у документ, замість того щоб виконати команду збереження.
   </details>

2. **Ви витратили 15 хвилин, ретельно пишучи bash-скрипт у nano. Ви натискаєте Ctrl+X для виходу, але випадково натискаєте «N» на запит «Save modified buffer?». Що станеться з вашим скриптом, і що варто було зробити інакше?**
   <details>
   <summary>Відповідь</summary>
   Зміни у вашому скрипті будуть назавжди втрачені, оскільки натискання «N» каже nano відкинути всі модифікації, зроблені після останнього збереження. На відміну від сучасних графічних редакторів із функціями автозбереження чи історії, термінальні редактори роблять саме те, що ви їм кажете в конкретний момент. Щоб запобігти цьому, варто виробити звичку вручну зберігати файл перед виходом. Ви робите це, натискаючи Ctrl+O (Write Out) і Enter для підтвердження назви файлу, що негайно записує ваші зміни на диск. Після збереження індикатор «Modified» угорі зникне, і ви зможете безпечно вийти за допомогою Ctrl+X.
   </details>

3. **Ви завантажили від колеги скрипт, що закінчується на `.sh`. Ви робите його виконуваним і намагаєтеся запустити, але система видає помилку, що не знає, як виконати файл. Відкривши його в nano, ви бачите, що першим рядком є просто `echo "Starting backup"`. Якого важливого елемента не вистачає і чому він потрібен системі?**
   <details>
   <summary>Відповідь</summary>
   У скрипті відсутній рядок «shebang» (наприклад, `#!/bin/bash`) у самому верху файлу. Без цього рядка завантажувач програм операційної системи не знає, яку програму-інтерпретатор слід використовувати для читання та виконання наступних інструкцій. Shebang діє як сувора директива, що вказує системі, чи передавати вміст файлу до bash, python, node чи іншого інтерпретатора. Оскільки його не було, система спробувала вгадати або використати метод за замовчуванням, що завершилося невдачею, бо контекст не був чітко визначений.
   </details>

4. **Ви написали ідеальний скрипт для автоматизації бекапів сервера, зберегли його як `backup.sh` і ввели `./backup.sh` для запуску. Замість того щоб почати бекап, термінал суворо відповів: `Permission denied`. Чому система заблокувала ваш скрипт і як це виправити?**
   <details>
   <summary>Відповідь</summary>
   Система заблокувала ваш скрипт, тому що за замовчуванням новостворені текстові файли мають лише права на читання та запис, але не на виконання. Це фундаментальна функція безпеки в Linux, розроблена для запобігання випадковому або зловмисному запуску будь-яких текстових файлів чи завантажених даних як програм. Щоб виправити це, ви повинні явно надати файлу право на виконання, запустивши `chmod +x backup.sh`. Це змінить режим файлу, сигналізуючи операційній системі, що ви, як власник, довіряєте цьому файлу і дозволяєте йому працювати як програмі.
   </details>

5. **Ви написали скрипт під назвою `backup.sh` і спробували запустити його за допомогою `./backup.sh`. Ви отримали «Permission denied». Потім ви запустили `chmod +x backup.sh` і спробували знову — тепер він каже «line 1: syntax error». Які ДВІ речі пішли не так і в якому порядку їх слід виправляти?**
   <details>
   <summary>Відповідь</summary>
   Перша проблема полягала в тому, що ваш файл не мав прав на виконання, що є стандартним для новостворених файлів із міркувань безпеки; це було правильно виправлено за допомогою `chmod +x`. Друга проблема полягає в тому, що сам скрипт містить невірні інструкції, швидше за все, це помилка в shebang у першому рядку (наприклад, `#bin/bash` замість `#!/bin/bash`). Ви завжди повинні спочатку вирішувати проблеми з правами доступу, тому що система навіть не намагатиметься перевірити синтаксис файлу, який їй не дозволено виконувати. Як тільки права надано, інтерпретатор може прочитати файл, знайти синтаксичну помилку і видати корисне повідомлення, вказуючи безпосередньо на рядок 1. Щоб виправити проблему, що залишилася, вам слід відкрити файл у nano і виправити друкарську помилку в першому рядку.
   </details>

## Практична вправа: Дошка оголошень на кухні

Створіть «дошку оголошень» для кухні ресторану, попрактикуйтеся в редагуванні та напишіть скрипт.

### Частина 1: Створення та редагування замітки

```bash
cd ~
nano kitchen-memo.txt
```

Введіть наступні п’ять рядків:

```
=== KITCHEN MEMO BOARD ===
1. Morning prep starts at 6 AM
2. New menu items arriving Thursday
3. Remember: clean as you go
4. Staff meeting at 3 PM Friday
```

Збережіть за допомогою `Ctrl + O`, Enter. Вийдіть за допомогою `Ctrl + X`.

Перевірте свою роботу:

```bash
cat kitchen-memo.txt
```

Ви повинні побачити всі рядки, надруковані саме так, як ви їх ввели.

### Частина 2: Редагування замітки

Відкрийте її знову:

```bash
nano kitchen-memo.txt
```

Додайте шостий рядок внизу:

```
5. Chef says: great work today, team!
```

Збережіть та вийдіть (`Ctrl + O`, Enter, `Ctrl + X`).

Перевірте:

```bash
cat kitchen-memo.txt
```

Усі шість рядків мають бути на місці.

### Частина 3: Написання скрипта очищення

```bash
nano kitchen-report.sh
```

Введіть:

```bash
#!/bin/bash

echo "=== Kitchen Status Report ==="
echo "Date: $(date)"
echo "Chef on duty: $(whoami)"
echo ""
echo "--- Memo Board Contents ---"
cat kitchen-memo.txt
echo ""
echo "--- Files in current directory ---"
ls
echo ""
echo "Report complete. Kitchen is running smoothly!"
```

Збережіть та вийдіть.

Зробіть його виконуваним та запустіть:

```bash
chmod +x kitchen-report.sh
./kitchen-report.sh
```

Очікуваний результат (деталі будуть відрізнятися):

```
=== Kitchen Status Report ===
Date: Sun Mar 23 14:45:00 UTC 2026
Chef on duty: yourname

--- Memo Board Contents ---
=== KITCHEN MEMO BOARD ===
1. Morning prep starts at 6 AM
2. New menu items arriving Thursday
3. Remember: clean as you go
4. Staff meeting at 3 PM Friday
5. Chef says: great work today, team!

--- Files in current directory ---
hello.txt    kitchen-memo.txt    kitchen-report.sh    my-first-script.sh
...

Report complete. Kitchen is running smoothly!
```

### Частина 4: Очищення

```bash
rm hello.txt kitchen-memo.txt kitchen-report.sh my-first-script.sh
```

**Критерії успіху**: Ви створили файл у nano, відредагували його, написали bash-скрипт, який зчитує дані з іншого файлу, зробили його виконуваним та запустили. Ви більше не просто користуєтеся терміналом — ви *програмуєте* в ньому.
