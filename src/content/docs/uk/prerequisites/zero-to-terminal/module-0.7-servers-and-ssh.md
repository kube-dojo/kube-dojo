## Чому цей модуль важливий

Все, що ви робили досі, було на *вашому* комп'ютері. Ваша кухня, ваші файли, ваш термінал. Але в реальному світі технологій більша частина дій відбувається на комп'ютерах, які знаходяться *десь інде* — у центрі обробки даних, у хмарі, у стійці з машинами, яких ви ніколи фізично не торкнетесь.

Ці інші комп'ютери називаються **серверами**, і спосіб спілкуватися з ними — через **SSH**.

Коли ви працюватимете з Kubernetes, ви керуватимете серверами (багатьма). Розуміння того, що таке сервери й як до них підключатися, — не опція, а фундамент.

---

## Що таке сервер?

Сервер — це просто комп'ютер. Ось і все. Ніякої магії, ніякої таємниці.

Слово «сервер» описує те, що комп'ютер *робить*, а не те, чим він *є*. Сервер — це комп'ютер, чия робота — **обслуговувати** (serve) інші комп'ютери.

```
Ваш ноутбук:
  - Має екран, клавіатуру, тачпад
  - Розрахований на ОДНУ людину, що сидить перед ним
  - Має графічний інтерфейс (вікна, іконки, миша)
  - Ви переглядаєте вебсторінки, пишете документи, дивитесь відео

Сервер:
  - Часто НЕ має екрана, клавіатури чи миші
  - Розрахований на обслуговування БАГАТЬОХ користувачів/комп'ютерів одночасно
  - Зазвичай має ЛИШЕ інтерфейс терміналу (без робочого столу)
  - Хостить вебсайти, запускає бази даних, обробляє дані
```

### Кухонна аналогія

Уявіть так:

- **Ваш ноутбук** — це домашня кухня. Ви готуєте для себе. Ви їсте в тій самій кімнаті, де готуєте. «Шеф-кухар» і «клієнт» — одна й та сама людина.

- **Сервер** — це кухня ресторану. Вона існує, щоб готувати їжу й *відправляти* її в обідню залу, повну клієнтів. На кухні немає обідніх столів — це не її робота. Її робота — готувати й подавати.

Коли ви відвідуєте вебсайт, ваш браузер (клієнт в обідній залі) надсилає запит до сервера (кухня на задньому дворі). Сервер готує відповідь (готує страву) й надсилає її назад до вашого браузера (подає страву).

## Ваш ноутбук проти сервера

Під капотом вони мають однакові базові компоненти:

| Компонент | Ваш ноутбук | Типовий сервер |
|-----------|-------------|----------------|
| CPU | 4–8 ядер | 16–128 ядер |
| RAM | 8–32 ГБ | 64–512 ГБ |
| Сховище | 256 ГБ – 2 ТБ | 1 ТБ – 100+ ТБ |
| Екран | Так | Зазвичай ні |
| Клавіатура | Так | Зазвичай ні |
| Операційна система | macOS/Windows | Linux (майже завжди) |
| Призначення | Одна людина, багато завдань | Багато клієнтів, конкретні завдання |

Сервер — це, по суті, потужний комп'ютер, оптимізований для обробки багатьох запитів одночасно, роботи 24/7 та віддаленого керування.

## Локальний та віддалений

Два слова, які ви постійно чутимете:

- **Локальний** = ваш комп'ютер, прямо тут, перед вами
- **Віддалений** = інший комп'ютер, десь інде, до якого ви підключаєтесь через мережу

```
Локальний:   Ваша кухня. Ви стоїте в ній.
Віддалений:  Кухня в іншому місті. Ви телефонуєте їм, щоб дати вказівки.
```

Коли ви вводите `ls` у терміналі зараз, ця команда виконується **локально** — на вашому комп'ютері.

Коли ви підключаєтесь до сервера й вводите `ls`, ця команда виконується **віддалено** — на сервері. Але у вашому терміналі вона виглядає абсолютно так само. У цьому краса.

---

## SSH: Ваш захищений тунель до віддалених кухонь

**SSH** означає **S**ecure **Sh**ell (захищена оболонка). Це програма, яка дає змогу відкрити сесію терміналу на віддаленому комп'ютері, безпечно.

Уявіть SSH як **захищену телефонну лінію до віддаленої кухні**. Ви берете слухавку (відкриваєте SSH), набираєте номер (підключаєтесь до сервера) й починаєте давати вказівки (вводити команди). Шеф-кухар у віддаленій кухні виконує їх, і ви чуєте результати через телефон.

«Захищена» частина важлива: все, що ви надсилаєте через SSH, зашифровано. Ніхто не може підслухати вашу розмову. Це як мати приватну, закодовану телефонну лінію.

### Коротко про змінні середовища

Іноді ви бачитимете в командах або документації речі на кшталт `$USER` або `$HOME`. Це **змінні середовища** — іменовані контейнери, які ваша система автоматично заповнює корисними значеннями.

- `$USER` містить ваше ім'я користувача
- `$HOME` містить шлях до вашої домашньої директорії

Спробуйте:

```bash
echo $USER
echo $HOME
```

Вам не потрібно їх встановлювати — ваш комп'ютер робить це за вас при вході. Ми згадуємо це зараз, бо команди SSH часто використовують `$USER`, і ви бачитимете змінні середовища протягом усієї кар'єри.

### Команда SSH

Базова команда SSH виглядає так:

```bash
ssh username@ip-address
```

Розберімо:

- `ssh` — програма, яку ви запускаєте
- `username` — ваше ім'я облікового запису на віддаленому сервері (як ваш бейджик у віддаленій кухні)
- `@` — просто роздільник (знак «на»)
- `ip-address` — адреса віддаленого сервера (як номер телефону кухні)

Реальний приклад може виглядати так:

```bash
ssh chef@192.168.1.100
```

Або з доменним ім'ям замість IP-адреси:

```bash
ssh chef@kitchen.example.com
```

### Що відбувається при підключенні

```
1. Ви вводите: ssh chef@kitchen.example.com
2. SSH зв'язується з віддаленим сервером
3. Сервер запитує: "Хто ви? Доведіть."
4. Ви надаєте доказ (пароль або ключ — детальніше нижче)
5. Сервер каже: "Ласкаво просимо, шеф-кухарю. Ось ваш термінал."
6. Рядок запрошення змінюється і показує назву віддаленого сервера
7. Кожна команда, яку ви вводите, тепер виконується на ВІДДАЛЕНОМУ сервері
8. Введіть "exit", щоб від'єднатися й повернутися до локального терміналу
```

Рядок запрошення може змінитися з:

```
yourname@your-laptop ~ $
```

на:

```
chef@remote-server ~ $
```

Ось як ви знаєте, що підключені до іншої машини.

## Паролі проти ключів SSH

Є два способи довести свою особу віддаленому серверу:

### Паролі

Простий спосіб. Сервер запитує пароль, ви його вводите.

```bash
ssh chef@kitchen.example.com
# Сервер запитує: chef@kitchen.example.com's password:
# Ви вводите пароль (він не відображається на екрані — це нормально)
```

Паролі працюють, але мають проблеми:
- Потрібно вводити щоразу
- Їх можна вгадати або вкрасти
- Вони незручні для автоматизованих систем

### Ключі SSH (кращий спосіб)

Ключі SSH працюють як система замка й ключа:

```
У вас є КЛЮЧ (приватний ключ) — зберігається на вашому комп'ютері, ніколи не передається.
На сервері є ЗАМОК (публічний ключ) — він знає, який ключ підходить.

При підключенні:
1. Ви пред'являєте свій ключ
2. Сервер перевіряє: "Цей ключ підходить до мого замка?"
3. Якщо так: "Заходьте!" (пароль не потрібен)
4. Якщо ні: "Доступ заборонено."
```

Це як мати фізичний ключ від дверей віддаленої кухні. Вам не потрібно нікому повідомляти пароль — ви просто відмикаєте двері.

**Генерація ключів SSH** (вам не потрібно робити це прямо зараз, просто знайте як):

```bash
ssh-keygen -t ed25519
```

Це створює два файли:
- `~/.ssh/id_ed25519` — Ваш **приватний ключ** (ключ). НІКОЛИ не діліться ним з кимось.
- `~/.ssh/id_ed25519.pub` — Ваш **публічний ключ** (замок). Ним можна ділитися вільно.

Ви копіюєте публічний ключ на сервер, і відтоді можете підключатися без пароля.

> **Уявіть так**: Приватний ключ — це ключ від вашого дому. Публічний ключ — це копія замка на ваших дверях. Ви можете дати замок будь-кому й сказати «поставте це на свої двері». Тепер ваш ключ відкриває їхні двері. Але ніхто не може виготовити ключ, просто подивившись на замок.

## Поширені параметри SSH

| Параметр | Що робить | Приклад |
|----------|-----------|---------|
| `-p` | Підключитися через інший порт (стандартний — 22) | `ssh -p 2222 chef@server.com` |
| `-i` | Використати конкретний файл ключа | `ssh -i ~/.ssh/mykey chef@server.com` |
| `-v` | Детальний режим (показує, що відбувається — корисно для налагодження) | `ssh -v chef@server.com` |

## Життєвий цикл з'єднання

```
Ваш комп'ютер                        Віддалений сервер
    |                                       |
    |  --- ssh chef@server.com ---------->  |
    |                                       |  "Запит на з'єднання отримано"
    |  <--- "Доведіть свою особу" --------  |
    |                                       |
    |  --- надсилає ключ/пароль --------->  |
    |                                       |  "Особу підтверджено"
    |  <--- "Ласкаво просимо! Ось оболонка" |
    |                                       |
    |  --- ls, pwd, nano тощо ---------->  |  (команди виконуються ТУТ, на сервері)
    |  <--- результати надсилаються назад   |
    |                                       |
    |  --- exit --------------------------->|
    |                                       |  "До побачення"
    |  (повернення до локального терміналу)  |
```

**Ключове розуміння**: Коли ви підключені через SSH, ваш термінал *виглядає* так само, але кожна команда виконується на віддаленій машині. Якщо ви створите файл — він створюється на сервері, а не на вашому ноутбуці. Якщо ви виконаєте `pwd` — він покаже директорію сервера, а не вашу.

## Чому це важливо для Kubernetes

Kubernetes працює на серверах — зазвичай на багатьох. Типовий кластер Kubernetes може мати:

- 3 сервери для «площини керування» (офіс управління ресторану)
- 10–100+ серверів як «робочі вузли» (кухні, що виконують фактичне приготування)

Ви використовуватимете SSH для підключення до цих серверів, діагностики проблем, перевірки логів і керування конфігураціями. Команди, які ви вивчили в попередніх модулях — `ls`, `cd`, `nano`, `cat` — працюють абсолютно так само на цих віддалених серверах.

## Чи знали ви?

- **SSH був винайдений у 1995 році фінським дослідником** на ім'я Тату Ілонен, після атаки з перехопленням паролів у мережі його університету. До SSH люди використовували `telnet` для підключення до віддалених комп'ютерів, що передавав все (включно з паролями) відкритим текстом, який будь-хто в мережі міг прочитати. SSH зашифрував з'єднання, зробивши його безпечним. Ідея була настільки вдалою, що стала глобальним стандартом майже миттєво.

- **Порт 22 був призначений особисто.** Коли Тату Ілонен створив SSH, йому потрібен був номер порту. Він написав електронного листа організації, що керує номерами портів (IANA), і вони просто призначили йому порт 22. Він отримав його протягом дня. Сьогодні порт 22 — один із найбільш впізнаваних номерів портів у комп'ютерній справі.

- **Міжнародна космічна станція використовує SSH.** Астронавти та центр керування польотами використовують SSH для безпечного керування комп'ютерними системами станції. Той самий інструмент, про який ви дізнаєтесь у цьому модулі, використовується для керування комп'ютерами на орбіті Землі.

## Типові помилки

| Помилка | Чому це проблема | Що робити натомість |
|---------|-----------------|---------------------|
| Ділитися приватним ключем | Будь-хто з вашим приватним ключем може отримати доступ до ваших серверів | Ніколи не діліться `id_ed25519` (файл БЕЗ `.pub`). Діліться лише файлом `.pub` |
| Забуваєте, що ви на віддаленому сервері | Можете видалити файли або змінити речі на неправильній машині | Завжди перевіряйте рядок запрошення й використовуйте `hostname`, щоб впевнитись, на якій машині ви |
| Вводите пароль у неправильному полі | SSH приховує введення пароля (ні крапок, ні зірочок). Люди думають, що щось зламалось, і вводять пароль десь видимо | Коли SSH просить пароль, просто вводьте його наосліп і натискайте Enter. Система ОТРИМУЄ ваші натискання |
| Не від'єднуєтесь, коли закінчили | Залишені відкритими SSH-сесії витрачають ресурси сервера й можуть бути ризиком безпеки | Введіть `exit` або натисніть `Ctrl + D`, коли закінчили |
| Панікуєте, коли з'єднання обривається | Перебої мережі трапляються — це не означає, що щось зламалось | Просто перепідключіться тією самою командою SSH. Ваші файли на сервері все ще на місці |

## Тест

1. **Яка різниця між вашим ноутбуком і сервером?**
   <details>
   <summary>Відповідь</summary>
   Фізично вони дуже схожі — обидва мають CPU, RAM і сховище. Різниця — у призначенні. Ваш ноутбук розрахований на одну людину з екраном і клавіатурою. Сервер розрахований на одночасне обслуговування багатьох клієнтів, часто не має екрана чи клавіатури і зазвичай працює на Linux лише з інтерфейсом терміналу. Сервер зазвичай потужніший (більше ядер CPU, більше RAM), бо обробляє багато запитів одночасно.
   </details>

2. **Що означає SSH і що він робить?**
   <details>
   <summary>Відповідь</summary>
   SSH означає Secure Shell (захищена оболонка). Він дає змогу відкрити сесію терміналу на віддаленому комп'ютері через зашифроване з'єднання. Ви вводите команди на локальній машині, вони виконуються на віддаленій, і результати надсилаються назад вам. «Захищена» частина означає, що все спілкування зашифровано, тож ніхто не може підслухати.
   </details>

3. **Яка різниця між приватним і публічним ключем?**
   <details>
   <summary>Відповідь</summary>
   Приватний ключ залишається на вашому комп'ютері й ніколи не має передаватися — це як ключ від вашого дому. Публічний ключ можна вільно поширювати й розміщувати на серверах, до яких ви хочете мати доступ — це як замок. При підключенні сервер перевіряє, чи ваш приватний ключ відповідає збереженому публічному ключу. Якщо відповідає — ви всередині. Знаючи публічний ключ, ніхто не зможе видати себе за вас — для цього потрібен приватний ключ.
   </details>

4. **Коли ви підключені до віддаленого сервера через SSH і вводите `touch recipe.txt`, де створюється файл?**
   <details>
   <summary>Відповідь</summary>
   На віддаленому сервері, а не на вашій локальній машині. Коли ви підключені через SSH, кожна введена команда виконується на віддаленому сервері. Файл `recipe.txt` створюється у файловій системі сервера. Ваша локальна машина не отримує копії. Щоб перевірити, на якій машині ви, подивіться на рядок запрошення або введіть `hostname`.
   </details>

## Практична вправа: SSH до localhost

Ми попрактикуємо SSH, підключившись до власного комп'ютера. Це може здатися безглуздим («Я і так тут!»), але це демонструє саме те, як працює SSH — і досвід ідентичний підключенню до справжнього віддаленого сервера.

### На macOS:

Спочатку увімкніть Remote Login (SSH), якщо він ще не увімкнений:

1. Відкрийте **System Settings** (або System Preferences на старших macOS)
2. Перейдіть до **General > Sharing**
3. Увімкніть **Remote Login**

Тепер у терміналі:

```bash
ssh localhost
```

Ви побачите щось на кшталт:

```
The authenticity of host 'localhost (127.0.0.1)' can't be established.
ED25519 key fingerprint is SHA256:AbCdEf123456...
Are you sure you want to continue connecting (yes/no)?
```

Введіть `yes` і натисніть Enter. (Це з'являється лише при першому підключенні до нового сервера. Комп'ютер каже «Я ніколи не спілкувався з цим сервером раніше — ви впевнені, що він справжній?»)

Введіть пароль вашого Mac, коли попросять.

Тепер ви «підключені». Спробуйте:

```bash
hostname
pwd
ls
echo "Hello from SSH!"
```

Все виглядає так само, бо ви підключені до себе. Але механізм ідентичний підключенню до сервера на іншому кінці світу.

Щоб від'єднатися:

```bash
exit
```

### На Linux:

Більшість систем Linux мають SSH увімкнений за замовчуванням. Якщо ні:

```bash
sudo apt install openssh-server    # Debian/Ubuntu
sudo systemctl start sshd          # Start the SSH service
```

Потім:

```bash
ssh localhost
```

Виконуйте ті самі кроки, що й для macOS вище.

### На Windows:

SSH до localhost складніший на Windows. Натомість просто попрактикуйте синтаксис команди:

```
# Ось як виглядало б підключення до справжнього сервера:
ssh yourname@192.168.1.100

# Ви б побачили:
yourname@192.168.1.100's password:

# Після введення пароля:
yourname@remote-server:~$

# Тепер кожна команда виконується на віддаленому сервері.
# Введіть "exit", щоб від'єднатися.
```

### Що було б зі справжнім сервером

Якби у вас був хмарний сервер (ви отримаєте його, коли почнете трек Kubernetes), досвід був би:

```bash
# Підключитися до сервера в хмарі
ssh ubuntu@54.123.45.67

# Ви б побачили:
ubuntu@ip-54-123-45-67:~$

# Тепер ви всередині Linux-сервера в центрі обробки даних десь
# Кожна команда виконується там:
ls          # Показує файли на сервері
pwd         # Показує ваш шлях на сервері
nano        # Відкриває nano на сервері
cat /etc/os-release    # Показує операційну систему сервера

# Від'єднайтесь, коли закінчите
exit
```

Це виглядало б, відчувалось і працювало б точно так само, як вправа з localhost. Єдина різниця — фізичне розташування комп'ютера.

**Критерії успіху**: Ви успішно (або концептуально розумієте, як) відкрили SSH-з'єднання, перевірили, що підключені до іншої машини (або до тієї самої через SSH), виконали команди й від'єдналися. Ви розумієте, що SSH дає вам віддалений термінал і що команди виконуються на сервері, а не на вашому ноутбуці.
