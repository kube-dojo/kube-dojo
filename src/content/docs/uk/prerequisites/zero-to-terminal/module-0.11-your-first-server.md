## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Розгорнути** простий веб-сервер з терміналу і зрозуміти, що він робить під капотом
- **Простежити** HTTP-запит від браузера до сервера та назад, пояснюючи кожен крок
- **Використовувати** `curl` для тестування сервера з командного рядка
- **Зв'язати** концепції з усіх попередніх модулів: файли, мережі, порти, SSH — все з'єднується тут

## Чому цей модуль важливий

Це фінальний іспит. Підсумковий проєкт. Момент, коли все стає на свої місця.

Ви збираєтесь **розгорнути вебсайт, який може відвідати будь-хто. Використовуючи лише термінал.**

Жодних модних конструкторів сайтів із перетягуванням. Жодного WordPress. Жодного Squarespace. Лише ви, термінал і навички, які ви набували з Модуля 0.1.

Подумайте, де ви починали. У Модулі 0.1 ви дізнались, що таке комп'ютер *взагалі*. А зараз ви збираєтесь використати його, щоб розмістити щось в інтернеті. Це не дрібниця. Саме це професіонали роблять щодня — і ви зараз це зробите теж.

Цей модуль має **два варіанти**:

- **Варіант A: Локальний (безплатно, реєстрація не потрібна)** — Запустити вебсервер на власній машині за допомогою Docker
- **Варіант B: Хмарний (безплатний рівень, потрібна реєстрація)** — Розгорнути на справжньому хмарному сервері, доступному всьому інтернету

Варіант A швидший і простіший. Варіант B ближчий до реальної практики. Обидва правильні. Оберіть той, що вас захоплює, або зробіть обидва.

## Навички, які ви набули

Перед початком підіб'ємо підсумки. Кожен пройдений модуль відіграє тут свою роль:

| Модуль | Навичка | Як ви її використаєте |
|--------|---------|----------------------|
| 0.1 | Як працюють комп'ютери | Розуміння того, що сервер фактично робить |
| 0.2 | Термінал | Ваш єдиний інтерфейс для всього проєкту |
| 0.3 | Команди | Навігація, створення файлів, перевірка стану |
| 0.4 | Файли та директорії | Створення HTML-файлу вашого вебсайту |
| 0.5 | Редагування файлів | Написання вебсторінки за допомогою nano |
| 0.6 | Мережі | Розуміння портів, IP-адрес та як браузери знаходять сервери |
| 0.7 | Сервери та SSH | Знання того, що таке сервер (та підключення до нього у Варіанті B) |
| 0.8 | Пакети | Встановлення програм на сервер |
| 0.9 | Хмара | Розуміння того, де живе ваш сервер (Варіант B) |

Якщо ви пропустили якийсь із цих модулів, поверніться й пройдіть його. Цей підсумковий проєкт передбачає, що всі дев'ять навичок готові.

## Що таке вебсервер?

Перш ніж щось розгортати, переконаймося, що ми чітко розуміємо одну концепцію.

**Вебсервер** — це програма, що слухає запити й надсилає назад вебсторінки. Ось і все. Коли ви вводите `google.com` у браузері, ваш браузер надсилає запит до вебсервера Google, і сервер надсилає назад HTML, який ваш браузер відображає.

Найпопулярніший вебсервер у світі називається **nginx** (вимовляється «енджін-ікс»). Він обслуговує приблизно третину всіх вебсайтів в інтернеті. Ми будемо використовувати його сьогодні.

У нашій аналогії з кухнею ресторану: nginx — це **офіціант**. Він приймає замовлення (HTTP-запити від браузерів) і доставляє страви (HTML-сторінки) клієнту.

## Варіант A: Локальний сервер із Docker (безплатно, без реєстрації)

Цей варіант використовує **Docker** — інструмент, що запускає застосунки в ізольованих «контейнерах». Вам не потрібно глибоко розуміти Docker зараз (це тема Cloud Native 101). Наразі просто уявіть, що це спосіб запустити програму, не встановлюючи її на постійно на вашу машину.

### Крок 1: Встановіть середовище для контейнерів

Вам потрібен інструмент для запуску контейнерів. Оберіть **будь-який** з цих — всі працюють однаково для нашої вправи:

| Інструмент | Найкраще для | Ліцензія |
|------------|-------------|----------|
| [Docker Desktop](https://www.docker.com/products/docker-desktop/) | Найпопулярніший, найбільша спільнота | Безкоштовний для особистого/малого бізнесу |
| [OrbStack](https://orbstack.dev/) | macOS — найшвидший, найлегший, найкращий UX | Безкоштовний для особистого використання |
| [Podman Desktop](https://podman-desktop.io/) | Без демона, безпечніший за замовчуванням | Вільне ПЗ |
| [Rancher Desktop](https://rancherdesktop.io/) | Включає K8s з коробки | Вільне ПЗ |

- **macOS/Windows**: Завантажте та встановіть будь-який з наведених вище
- **Linux**:
  ```bash
  # Варіант A: Docker
  sudo apt update && sudo apt install docker.io -y
  sudo systemctl start docker
  sudo usermod -aG docker $USER

  # Варіант B: Podman (без демона, без root)
  sudo apt update && sudo apt install podman -y
  ```
  (Якщо обрали Docker, вийдіть і зайдіть знову після команди usermod.)

> **Примітка**: Якщо встановили Podman, команди ідентичні — просто пишіть `podman` замість `docker`. Можна навіть створити аліас: `alias docker=podman`

Перевірте, що Docker працює:

```bash
docker --version
```

Ви маєте побачити щось на кшталт `Docker version 24.x.x` або новіше. Якщо бачите «command not found» — Docker ще не встановлений.

### Крок 2: Запустіть nginx

Ось вона. Одна команда для запуску вебсервера:

```bash
docker run -d -p 8080:80 --name my-website nginx
```

> **Подумайте**: Ви прив'язуєте порт 8080 до порту 80. Якщо інша програма на вашому комп'ютері вже використовує порт 8080 (наприклад, інший вебсервер або інструмент локальної розробки), як на вашу думку поведеться ця команда Docker? Вона перезапише наявну програму чи видасть помилку?

Розберімо кожну частину (бо розуміння важливіше за запам'ятовування):

| Частина | Що робить |
|---------|-----------|
| `docker run` | Запустити новий контейнер |
| `-d` | Запустити у фоні (detached), щоб термінал повернувся |
| `-p 8080:80` | З'єднати порт 8080 вашого комп'ютера з портом 80 контейнера |
| `--name my-website` | Дати контейнеру зручне ім'я |
| `nginx` | Використати образ nginx (Docker завантажує його автоматично) |

Пам'ятаєте Модуль 0.6 про мережі? Порт 80 — стандартний порт для вебтрафіку. Ми прив'язуємо його до 8080 на вашій машині, щоб не конфліктувати з чимось іншим.

> **Зв'яжіть концепції**: Прапорець `-p 8080:80` — це Модуль 0.6 (порти) у дії. Ваш браузер надсилає запит на порт 8080 вашої машини. Docker перенаправляє його на порт 80 всередині контейнера, де слухає nginx. Відповідь повертається тим самим шляхом. Кожна концепція з цих модулів зараз працює разом.

### Крок 3: Подивіться, що працює

Відкрийте веббраузер і перейдіть за адресою:

```
http://localhost:8080
```

Ви маєте побачити сторінку з написом **«Welcome to nginx!»**

Це вебсервер, що працює на вашій машині. Ви щойно це зробили. Однією командою.

### Крок 4: Створіть свою власну вебсторінку

> **Зупиніться та подумайте**: Як ви гадаєте, що станеться, якщо ви заміните стандартну сторінку nginx власним HTML-файлом всередині контейнера? Вона відобразиться миттєво чи потрібно щось перезапустити? Спробуйте вгадати, а потім виконайте кроки й перевірте, чи ви мали рацію.

Тепер замінімо стандартну сторінку на щось, створене вами. Відкрийте термінал і створіть HTML-файл:

```bash
nano ~/index.html
```

Введіть (або вставте) це:

```html
<!DOCTYPE html>
<html>
<head>
    <title>My First Server</title>
    <style>
        body {
            font-family: Arial, sans-serif;
            max-width: 600px;
            margin: 80px auto;
            text-align: center;
            background-color: #1a1a2e;
            color: #eee;
        }
        h1 { color: #00d4ff; }
        p { font-size: 1.2em; line-height: 1.6; }
        .badge {
            display: inline-block;
            background: #00d4ff;
            color: #1a1a2e;
            padding: 8px 20px;
            border-radius: 20px;
            font-weight: bold;
            margin-top: 20px;
        }
    </style>
</head>
<body>
    <h1>Hello, Internet!</h1>
    <p>This page is running on a server that I set up myself,
       using nothing but the terminal.</p>
    <p>I went from "what is a computer" to "I deployed a website"
       in ten modules.</p>
    <div class="badge">Zero to Terminal: Complete</div>
</body>
</html>
```

Збережіть і вийдіть (`Ctrl + O`, Enter, `Ctrl + X`).

### Крок 5: Скопіюйте сторінку в сервер

Пам'ятаєте, вебсервер працює всередині контейнера Docker. Потрібно скопіювати файл туди:

```bash
docker cp ~/index.html my-website:/usr/share/nginx/html/index.html
```

Ця команда каже: «Скопіюй `index.html` з моєї домашньої директорії в контейнер з назвою `my-website`, розмістивши його за шляхом `/usr/share/nginx/html/index.html`.»

Шлях `/usr/share/nginx/html/` — це місце, де nginx шукає вебсторінки для обслуговування. Це просто директорія — так само як директорії, з якими ви працювали в Модулі 0.4.

### Крок 6: Подивіться ВАШУ сторінку

Поверніться до браузера й оновіть `http://localhost:8080`.

Ви маєте побачити свою сторінку — темний фон, блакитний заголовок, ваші слова.

**Ви щойно розгорнули вебсайт.**

Ви створили файл (Модуль 0.4), відредагували його за допомогою nano (Модуль 0.5), зрозуміли, що означають порт і localhost (Модуль 0.6), і подали його з працюючого серверного процесу (Модуль 0.7). Все з'єдналося.

### Прибирання

Коли надивитесь на свою роботу:

```bash
docker stop my-website
docker rm my-website
```

Це зупиняє й видаляє контейнер. Ваш файл `~/index.html` все ще на вашій машині.

## Варіант B: Хмарний сервер (безплатний рівень)

Цей варіант розміщує ваш вебсайт на **справжньому сервері в інтернеті** з публічною IP-адресою. Будь-хто у світі може його відвідати. Саме так працюють справжні вебсайти.

Вам знадобиться обліковий запис безплатного рівня у хмарного провайдера. Інструкції нижче використовують загальний підхід, що працює з AWS, GCP або Oracle Cloud.

### Крок 1: Отримайте безплатну хмарну VM

Зареєструйтесь на безплатному рівні одного з цих провайдерів:

- **Oracle Cloud** (найщедріший безплатний рівень — завжди безплатні VM): [cloud.oracle.com/free](https://cloud.oracle.com/free)
- **Google Cloud** ($300 безплатного кредиту на 90 днів): [cloud.google.com/free](https://cloud.google.com/free)
- **AWS** (750 годин/місяць t2.micro на 12 місяців): [aws.amazon.com/free](https://aws.amazon.com/free)

Створіть найменшу доступну Linux VM (Ubuntu найпростіший для початківців). Під час налаштування:

1. Оберіть **Ubuntu** як операційну систему
2. Виберіть **найменший безплатний екземпляр** (наприклад, t2.micro на AWS, e2-micro на GCP)
3. **Завантажте ключ SSH** коли запропонують — він знадобиться для підключення
4. Переконайтесь, що група безпеки / файрвол дозволяє **порт 22 (SSH)** та **порт 80 (HTTP)**

Запишіть **публічну IP-адресу** вашого нового сервера. Вона виглядатиме приблизно як `34.123.45.67`.

### Крок 2: Підключіться через SSH

Пам'ятаєте Модуль 0.7? Ось де SSH стає реальним:

```bash
chmod 400 ~/Downloads/my-key.pem
ssh -i ~/Downloads/my-key.pem ubuntu@YOUR_PUBLIC_IP
```

Замініть `YOUR_PUBLIC_IP` на фактичну IP-адресу вашої VM. Замініть шлях до ключа на місце, куди ви його зберегли.

Якщо все налаштовано правильно, ви побачите привітальне повідомлення Linux і рядок запрошення. Тепер ви всередині комп'ютера в центрі обробки даних десь — можливо, на іншому континенті.

### Крок 3: Встановіть nginx

Тепер використайте навички керування пакетами з Модуля 0.8:

```bash
sudo apt update
sudo apt install nginx -y
```

Ось і все. nginx встановлений і працює. На Ubuntu він запускається автоматично після встановлення.

Перевірте, що працює:

```bash
sudo systemctl status nginx
```

Ви маєте побачити `active (running)` зеленим кольором.

> **Зупиніться та подумайте**: Ви щойно встановили nginx, і він запустився автоматично. Перш ніж відкривати веббраузер, яку команду ви могли б запустити прямо тут, у терміналі, щоб перевірити, чи сервер справді відповідає на запити на вашій VM?

### Крок 4: Перевірте стандартну сторінку

Відкрийте браузер на своєму комп'ютері й перейдіть за адресою:

```
http://YOUR_PUBLIC_IP
```

Ви маєте побачити стандартну сторінку nginx. Ця сторінка обслуговується з машини в центрі обробки даних, через інтернет, до вашого браузера. Приділіть хвилину, щоб це оцінити.

### Крок 5: Створіть свою сторінку

> **Подумайте**: Стандартна сторінка nginx знаходиться за шляхом `/var/www/html/index.html`. Якби ви створили другий файл з назвою `about.html` у тій самій директорії, яку саме URL-адресу ви б ввели у браузері, щоб переглянути його?

Все ще підключені через SSH, відредагуйте стандартну вебсторінку:

```bash
sudo nano /var/www/html/index.html
```

> **Зверніть увагу**: Шлях — `/var/www/html/` на nginx в Ubuntu, а не `/var/share/`. Різні системи розміщують вебфайли в дещо різних місцях.

Видаліть усе у файлі (`Ctrl + K` кілька разів) і введіть свій HTML:

```html
<!DOCTYPE html>
<html>
<head>
    <title>My First Cloud Server</title>
    <style>
        body {
            font-family: Arial, sans-serif;
            max-width: 600px;
            margin: 80px auto;
            text-align: center;
            background-color: #1a1a2e;
            color: #eee;
        }
        h1 { color: #00d4ff; }
        p { font-size: 1.2em; line-height: 1.6; }
        .badge {
            display: inline-block;
            background: #00d4ff;
            color: #1a1a2e;
            padding: 8px 20px;
            border-radius: 20px;
            font-weight: bold;
            margin-top: 20px;
        }
    </style>
</head>
<body>
    <h1>Hello, Internet!</h1>
    <p>This page is running on a real cloud server that I set up myself,
       using nothing but SSH and the terminal.</p>
    <p>I went from "what is a computer" to "I deployed a website
       on the internet" in ten modules.</p>
    <div class="badge">Zero to Terminal: Complete</div>
</body>
</html>
```

Збережіть і вийдіть (`Ctrl + O`, Enter, `Ctrl + X`).

### Крок 6: Подивіться свою сторінку в інтернеті

Оновіть `http://YOUR_PUBLIC_IP` у браузері.

Ваша сторінка тепер **доступна в інтернеті**. Ви можете надіслати цю IP-адресу другу, і він побачить вашу сторінку. З телефону, з іншої країни — звідусіль.

Ви зробили це за допомогою SSH (Модуль 0.7), керування пакетами (Модуль 0.8), редагування файлів (Модуль 0.5) та розуміння мереж (Модуль 0.6) і хмарних обчислень (Модуль 0.9).

### Важливо: Попередження про безплатний рівень

Хмарні VM можуть коштувати грошей, якщо ви перевищите ліміти безплатного рівня. Коли закінчите цю вправу:

- **Зупиніть або видаліть VM** через консоль хмарного провайдера
- Або залиште працювати, якщо ваш безплатний рівень це дозволяє (наприклад, завжди-безплатний рівень Oracle)
- **Ніколи не залишайте хмарний ресурс увімкненим, про який забули** — це одна з найпоширеніших (і найдорожчих) помилок початківців

Щоб від'єднатися від SSH:

```bash
exit
```

- **Перший вебсайт у світі досі працює.** Тім Бернерс-Лі створив його у 1991 році в CERN. Він обслуговувався з комп'ютера NeXT, на якому була приклеєна записка: «This machine is a server. DO NOT POWER IT DOWN!!» (Ця машина — сервер. НЕ ВИМИКАЙТЕ ЇЇ!!). Ви й досі можете відвідати його за адресою [info.cern.ch](http://info.cern.ch). Ваше сьогоднішнє налаштування сервера було більш досконалим, ніж те, що запустило Всесвітню мережу.

- **nginx був створений, щоб виграти парі.** У 2002 році Ігор Сисоєв поставив собі за мету розв'язати «проблему C10K» — обробку 10 000 одночасних з'єднань на одному сервері. На той час Apache (домінуючий вебсервер) з цим не справлявся. Сисоєв витратив два роки на написання nginx, і він не просто розв'язав C10K — сучасний nginx може обробляти понад мільйон одночасних з'єднань. Тепер він обслуговує приблизно 34% усіх вебсайтів в інтернеті.

- **Ваш вебсайт обслуговується так само, як Netflix.** Серйозно. Netflix, Airbnb і Dropbox — усі використовують nginx як свій вебсервер. Різниця між вашим налаштуванням і їхнім — у масштабі (у них тисячі серверів) та конфігурації (у них команди інженерів, що налаштовують параметри). Але фундаментальна технологія — процес, що слухає порт 80 і повертає HTML — ідентична.

## Типові помилки

| Помилка | Чому це проблема | Що робити натомість |
|---------|-----------------|---------------------|
| Забули відкрити порт 80 у хмарному файрволі | Сервер працює, але ніхто не може до нього дістатися | Перевірте групи безпеки / правила файрвола; дозвольте вхідний HTTP на порту 80 |
| Використовуєте `http://localhost` для хмарного варіанту | `localhost` означає *вашу* машину, а не віддалений сервер | Використовуйте публічну IP-адресу вашої хмарної VM |
| Редагуєте неправильний шлях `index.html` | nginx не обслуговуватиме ваш файл, якщо він у неправильній директорії | Ubuntu використовує `/var/www/html/`, Docker — `/usr/share/nginx/html/` |
| Забуваєте `sudo` при редагуванні файлів на сервері | Файли вебсервера належать root; отримаєте «Permission denied» | Використовуйте `sudo nano /var/www/html/index.html` |
| Залишаєте хмарну VM працювати після вправи | Безплатні рівні мають ліміти; вас можуть нарахувати оплату | Зупиніть або видаліть VM, коли закінчите експериментувати |
| Не завантажуєте ключ SSH під час створення VM | Ви не зможете підключитися до сервера без нього | Завжди зберігайте файл ключа одразу; деякі провайдери дозволяють завантажити його лише один раз |

## Тест

1. **Ви пояснюєте роль вебсервера колезі, який налаштовує нову програму. Він запитує: "Я написав свої HTML-файли, чому мені потрібен цей nginx, що працює на сервері?" Як ви поясните специфічну роль, яку відіграє nginx у доставці цих файлів користувачам?**
   <details>
   <summary>Відповідь</summary>
   nginx діє як "офіціант" або посередник між файловою системою сервера та зовнішнім інтернетом. Хоча у вас є HTML-файли на жорсткому диску, браузер не може просто зайти у ваш комп'ютер і прочитати їх. nginx активно слухає на певному мережевому порту (зазвичай 80 або 443) вхідні HTTP-запити. Коли надходить запит, nginx інтерпретує його, знаходить відповідний HTML-файл у файловій системі, упаковує його в дійсну HTTP-відповідь і відправляє його назад через мережу до браузера користувача. Без цього активного механізму прослуховування та відповіді ваші HTML-файли повністю недоступні для веб.
   </details>

2. **Ви успішно запустили команду \`docker run -d -p 9090:80 nginx\` на своїй локальній машині. Однак, за звичкою, ви відкриваєте браузер і переходите за адресою \`http://localhost:8080\`. Що саме відбувається, і чому зміна першого числа у прапорці \`-p\` призвела до такого результату?**
       <details>
       <summary>Відповідь</summary>
       Ваш браузер відобразить помилку "connection refused" (з'єднання відхилено) або "site can't be reached" (сайт недоступний). Прапорець \`-p 9090:80\` наказує Docker зіставити порт 9090 на вашій фізичній машині (хості) з портом 80 всередині ізольованого контейнера, де насправді слухає nginx. Відвідуючи \`localhost:8080\`, ваш браузер стукає у мережеві двері (порт 8080), які наразі жодна програма не слухає. nginx щасливо працює всередині контейнера і чекає на трафік на своєму внутрішньому порту 80, але цей трафік тепер ексклюзивно маршрутизується з порту 9090 на вашій локальній машині, а не з 8080.
       </details>
3. **Ви розгортаєте вебсайт за допомогою методу хмарної VM (Варіант B) і успішно копіюєте свій користувацький \`index.html\` до \`/var/www/html/index.html\`. Пізніше ви спробуєте метод Docker (Варіант A) і скопіюєте свій ідентичний HTML-файл до \`/var/www/html/index.html\` всередині контейнера, але браузер все ще показує стандартну сторінку "Welcome to nginx!". Що пішло не так, і чого це вчить вас щодо конфігурації програмного забезпечення?**
       <details>
       <summary>Відповідь</summary>
       Контейнер ігнорує ваш користувацький файл, тому що офіційний образ Docker для nginx налаштований його творцями на пошук веб-файлів в іншій директорії — а саме, \`/usr/share/nginx/html/\`. Програмне забезпечення, таке як nginx, не має єдиного універсального, магічного місця, де воно знаходить файли; натомість, воно покладається на конфігураційний файл, який диктує точний шлях файлової системи, з якого воно повинно обслуговувати. Розробники пакунків для Ubuntu (Варіант B) обрали \`/var/www/html/\` як свій стандарт, тоді як розробники образу Docker обрали \`/usr/share/nginx/html/\`. Це вчить нас, що шляхи є довільними конфігураційними виборами, зробленими системними адміністраторами або розробниками пакунків, і ви завжди повинні адаптуватися до конфігурації конкретного середовища, а не припускати універсальні значення за замовчуванням.
       </details>
4. **Покроково опишіть, що відбувається, коли ви вводите `http://YOUR_PUBLIC_IP` у браузері, а ваш сервер nginx повертає вашу власну сторінку. Включіть DNS, TCP, порт, nginx та файлову систему.**
   <details>
   <summary>Відповідь</summary>
   Коли ви вводите URL-адресу, ваш браузер перевіряє, чи потрібно розв'язувати доменне ім'я через DNS (хоча тут ми використовуємо пряму IP-адресу, пропускаючи розв'язання DNS). Далі ваш комп'ютер ініціює TCP-з'єднання з цією IP-адресою саме через порт 80, який є стандартним портом для HTTP-трафіку. Після завершення TCP-рукостискання браузер надсилає запит HTTP GET із запитом кореневого документа (`/`). Вебсервер nginx, що слухає порт 80, отримує цей запит, переглядає свою конфігурацію, щоб знайти відповідну директорію у файловій системі (наприклад, `/var/www/html/`), і читає знайдений там файл `index.html`. Нарешті, nginx надсилає вміст цього файлу назад через TCP-з'єднання як HTTP-відповідь, яку ваш браузер перетворює на видиму вебсторінку.
   </details>

5. **Ваш браузер показує «connection refused» (з'єднання розірвано) при відвідуванні `localhost:8080`. Назвіть три можливі причини та спосіб діагностики кожної.**
   <details>
   <summary>Відповідь</summary>
   «Connection refused» зазвичай означає, що на цьому порті нічого не слухає, що вказує на кілька поширених проблем. По-перше, контейнер Docker міг зупинитися; ви можете діагностувати це, запустивши `docker ps`, щоб перевірити, чи ваш контейнер `my-website` все ще активний. По-друге, ви могли вказати неправильні порти у вашій команді запуску, наприклад, `-p 8080:8080` замість `-p 8080:80`; перевірте це, подивившись відображення портів у виводі `docker ps`. По-третє, інша програма вже може використовувати порт 8080 на вашій машині, не даючи Docker прив'язатися до нього; ви можете перевірити це за допомогою команди на кшталт `lsof -i :8080` або `netstat`, щоб побачити, який процес займає порт.
   </details>

6. **Ви можете підключитися до хмарної VM через SSH, але не через браузер на порті 80. Яка найбільш імовірна причина та яку команду ви б запустили для перевірки?**
   <details>
   <summary>Відповідь</summary>
   Найімовірнішою причиною є файрвол, що блокує вхідний вебтрафік, оскільки хмарні провайдери зазвичай блокують порт 80 за замовчуванням, дозволяючи порт 22 для SSH. Оскільки SSH працює, ми знаємо, що сервер у мережі й доступний, тому проблема стосується саме HTTP-трафіку. Щоб перевірити, чи сам сервер працює правильно всередині, ви можете підключитися через SSH і запустити `curl http://localhost`. Якщо `curl` повертає HTML-контент локально, це підтверджує, що nginx працює ідеально, а проблема однозначно у зовнішньому файрволі хмарного провайдера або налаштуваннях груп безпеки, які блокують зовнішній доступ до порту 80.
   </details>

7. **Ви розгорнули свій сервер і хочете перевірити, чи повертає він правильний HTML, перш ніж відкривати браузер. Як би ви використали `curl` для такої перевірки, і на що саме ви дивитесь у виводі?**
   <details>
   <summary>Відповідь</summary>

Цей інструмент надсилає запит HTTP GET і виводить тіло відповіді прямо на екран, минаючи будь-яке графічне відображення. Ви дивитесь, чи виводить термінал необроблений HTML-код вашої власної сторінки, наприклад, теги `<h1>Hello, Internet!</h1>`. Якщо він повертає очікуваний HTML, ви знаєте, що сервер успішно обробляє запити й видає правильний файл, що доказує працездатність серверної частини ще до залучення браузера.
   </details>

8. **Назвіть щонайменше п'ять навичок із попередніх модулів, які ви використали в цьому підсумковому проєкті.**

### Шлях B: Cloud Native

**Хочете створювати й розгортати застосунки у масштабі? Саме туди рухається індустрія.**

Трек Cloud Native починається саме там, де ви зупинились. Ви дізнаєтесь про контейнери (технологію за тією командою Docker, яку ви щойно виконали), потім Docker у деталях, потім Kubernetes — систему, що керує тисячами контейнерів на сотнях серверів автоматично.

Якщо ви хочете стати хмарним інженером, DevOps-інженером чи платформним інженером — це ваш шлях.

> Почніть тут: [Cloud Native 101](../cloud-native-101/module-1.1-what-are-containers/)

### Шлях C: Обидва

**Більшість досвідчених інженерів знають обидва.** Вони розуміють внутрішню будову Linux *і* хмарно-нативні інструменти. Почніть з того, що захоплює більше. Інший шлях нікуди не подінеться.
