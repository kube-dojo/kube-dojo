## Чому це важливо

Усе на вашому комп'ютері — кожне фото, кожна пісня, кожна програма, кожне налаштування — зберігається як **файл**. І ці файли організовані в **директорії** (які також називають папками).

Коли ви використовуєте графічний інтерфейс (GUI), ви бачите різнокольорові іконки папок і файлів. Ви двічі клацаєте по них, щоб відкрити. Але за лаштунками ваш комп'ютер сприймає файли та директорії як дерево текстових шляхів, наприклад `/home/yourname/Documents/report.txt`.

У Kubernetes та хмарній інженерії ви постійно працюватимете з файлами: конфігураційними файлами, скриптами, логами, маніфестами. Якщо ви не вмієте навігувати файлами та директоріями через термінал, ви загубитеся. Цей модуль виправить це.

## Що таке файл?

**Файл** — це контейнер для інформації, що зберігається на вашому комп'ютері.

Уявіть його як аркуш паперу з текстом. У аркуша є:

- **Назва** (щоб ви могли його знайти): `grocery-list.txt`
- **Вміст** (сама інформація): "яйця, молоко, хліб"
- **Місцезнаходження** (де він зберігається): на вашому столі, у шухляді, у шафі

Файли можуть містити будь-що:
- Текст (як-от нотатка або конфігурація)
- Код (інструкції для комп'ютера)
- Зображення (ваші фото)
- Музику, відео, бази даних та багато іншого

> В нашій аналогії з ресторанною кухнею: файл — це як **картка з рецептом**. Вона має назву ("Томатний суп"), вміст (сам рецепт) і десь зберігається (коробка з рецептами, полиця, шухляда).

### Усе є файлом (Справді!)

Ось дещо незвичайне про Linux (операційну систему, на якій працює Kubernetes):

**Майже все представлено як файл.**

- Ваші реальні файли? Файли.
- Ваша клавіатура? Система бачить її як файл (`/dev/input/event0`).
- Ваш жорсткий диск? Файл (`/dev/sda`).
- Запущені програми? Вони мають записи в `/proc/`, які виглядають як файли.

Зараз вам не потрібно турбуватися про файли пристроїв. Просто знайте, що концепція "файлів" у Linux набагато глибша за документи та фото. Ця філософія дизайну є однією з причин потужності Linux — усе можна читати, записувати та маніпулювати за допомогою одного й того самого набору інструментів.

## Що таке директорія?

**Директорія** — це контейнер для файлів (та інших директорій). Це те саме, що й **папка** — буквально та сама концепція, просто інша назва.

- Користувачі GUI кажуть "папка" (бо іконка схожа на паперову папку).
- Користувачі термінала кажуть "директорія" (технічний термін).

> На кухні: директорія — це як **шухляда** або **полиця**. Ваші картки з рецептами (файли) потрапляють у шухляди (директорії). Шухляди можуть містити інші шухляди (піддиректорії).

---

## Дерево директорій

Ваш комп'ютер організовує всі файли та директорії в структуру **дерева**. Воно починається з однієї точки вгорі і розгалужується донизу.

```
/                          <-- "Корінь" (root) — сама вершина, перший поверх
├── home/                  <-- Де живуть акаунти користувачів
│   └── yourname/          <-- ВАША домашня директорія
│       ├── Documents/
│       ├── Downloads/
│       ├── Desktop/
│       └── projects/
├── etc/                   <-- Системні конфігураційні файли
├── var/                   <-- Змінні дані (логи, бази даних)
├── tmp/                   <-- Тимчасові файли
└── usr/                   <-- Програми та утиліти користувачів
```

### Ключові локації

| Шлях | Що це таке | Аналогія з кухнею |
|------|-----------|-----------------|
| `/` | **Коренева директорія** — сама вершина дерева | Сама будівля — все знаходиться всередині неї |
| `/home/yourname/` | **Ваша домашня директорія** — ваш особистий простір | Ваша особиста робоча станція на кухні |
| `~` | **Скорочення для вашої домашньої директорії** | Прізвисько вашої робочої станції |
| `/etc/` | Системні конфігураційні файли | Посібник з політики ресторану та стандарти рецептів |
| `/tmp/` | Тимчасові файли (видаляються при перезавантаженні) | Стіл для приготування — використовується під час готування, прибирається після |
| `/var/log/` | Файли логів (записи про те, що сталося) | Книга історії замовлень |

> Символ `~` (тильда) — це ярлик. Замість того, щоб вводити `/home/yourname/`, ви можете просто ввести `~`. Ваш термінал зрозуміє, що ви маєте на увазі. Це як прізвисько — простіше використовувати, ніж повне ім'я.

> **Чому це важливо в K8s**: Як інженер Kubernetes, ви будете постійно взаємодіяти з конкретними файлами саме в цих директоріях. Ви будете налаштовувати доступ до кластера, редагуючи `~/.kube/config`. Ви будете налагоджувати системні компоненти, читаючи маніфести в `/etc/kubernetes/manifests/`. А коли щось зламається, ви шукатимете підказки у `/var/log/pods/` або `/var/log/containers/`.

### На macOS

macOS дещо відрізняється. Ваша домашня директорія знаходиться у `/Users/yourname/` замість `/home/yourname/`. Але `~` все одно працює як скорочення, тому вам рідко потрібно про це думати.

## Навігація: де я?

Коли ви відкриваєте термінал, ви "стоїте" в певній директорії — зазвичай у вашій домашній директорії. Термінал не показує вам візуальну карту; вам потрібно запитати.

### `pwd` — Print Working Directory

"Де я зараз перебуваю?"

```bash
$ pwd
/home/yourname
```

`pwd` розшифровується як **Print Working Directory** (вивести робочу директорію). Вона показує повний шлях до місця, де ви зараз знаходитесь.

> Аналогія з кухнею: "В якій кімнаті я стою?" — `pwd` відповідає на це питання.

### `ls` — List Contents

"Що є в цій кімнаті?"

```bash
$ ls
Desktop    Documents    Downloads    Music    Pictures
```

`ls` розшифровується як **list** (список). Вона показує файли та директорії у вашому поточному місці.

Ви можете додати опції, щоб побачити більше деталей:

```bash
$ ls -l
```

Прапор `-l` означає "довгий формат" (long format). Тепер ви побачите щось на кшталт:

```
drwxr-xr-x  2 yourname yourname 4096 Mar 23 10:00 Desktop
drwxr-xr-x  3 yourname yourname 4096 Mar 23 09:45 Documents
-rw-r--r--  1 yourname yourname  220 Mar 23 08:30 notes.txt
```

Не панікуйте — ми розкодуємо це незабаром.

### `cd` — Change Directory

"Перейти в іншу кімнату."

```bash
$ cd Documents
$ pwd
/home/yourname/Documents
```

`cd` розшифровується як **change directory** (змінити директорію). Ви кажете їй, куди йти, і вона вас туди переносить.

Деякі основні скорочення `cd`:

| Команда | Куди вона вас переносить | Аналогія |
|---------|-------------------|---------|
| `cd ~` або просто `cd` | Ваша домашня директорія | "Повернутися до моєї робочої станції" |
| `cd ..` | На один рівень вгору (батьківська директорія) | "Перейти в кімнату, яка містить цю кімнату" |
| `cd -` | Остання директорія, в якій ви були | "Повернутися туди, де я щойно був" |
| `cd /` | Коренева директорія | "Перейти на перший поверх" |

## Абсолютні та відносні шляхи

Це важлива концепція. Є два способи описати, де знаходиться файл або директорія:

### Абсолютний шлях (повна адреса)

Абсолютний шлях починається від кореня (`/`) і вказує повне місцезнаходження:

```
/home/yourname/Documents/report.txt
```

Це як надання повної поштової адреси: **вул. Головна 123, Київ, 01001, Україна**. Незалежно від того, де ви знаходитесь у світі, ця адреса вказує саме на одне місце.

### Відносний шлях (напрямки звідси)

Відносний шлях описує місцезнаходження відносно того, де ви зараз перебуваєте:

```
Documents/report.txt
```

Це як сказати: **"Два квартали ліворуч, потім один квартал вгору"**. Ці вказівки працюють лише тоді, коли ви знаєте точку старту.

### Приклад

```bash
$ pwd
/home/yourname

# Ці дві команди роблять те саме:
$ cd /home/yourname/Documents       # Абсолютний шлях
$ cd Documents                       # Відносний шлях (працює, бо ми в /home/yourname)
```

### Спеціальні символи шляху

| Символ | Значення | Приклад |
|--------|---------|---------|
| `/` | Коренева директорія (на початку) або роздільник шляху | `/home/yourname` |
| `~` | Ваша домашня директорія | `~/Documents` = `/home/yourname/Documents` |
| `.` | Поточна директорія ("тут") | `./script.sh` = "script.sh у цій директорії" |
| `..` | Батьківська директорія ("на один рівень вгору") | `../Downloads` = "перейти вгору, потім у Downloads" |

## Читання файлів

Тепер зазирнемо всередину файлів. У вас є кілька інструментів, кожен з яких корисний у різних ситуаціях.

> **Замисліться**: Вам потрібно перевірити останні кілька рядків лог-файлу довжиною 10 000 рядків. Чи використаєте ви команду, яка показує весь файл, чи ту, що показує лише кінець? Майте це на увазі, читаючи про `cat`, `head` та `tail` нижче — кожен з них існує, тому що вирішує певну проблему.

### `cat` — Показати весь файл

```bash
$ cat notes.txt
This is my first note.
I wrote it in the terminal!
```

`cat` розшифровується як "concatenate" (з'єднання), але більшість людей використовують її для відображення вмісту файлу. Вона виводить весь файл на ваш екран.

> Аналогія з кухнею: `cat` — це як прочитати вголос весь рецепт від початку до кінця.

**Коли використовувати**: Маленькі файли (до 50 рядків).

### `head` — Показати перші 10 рядків

```bash
$ head long-file.txt
```

Це показує лише перші 10 рядків. Ви можете змінити кількість:

```bash
$ head -n 5 long-file.txt    # Показати перші 5 рядків
```

> Аналогія з кухнею: "Просто прочитай мені назву та інгредієнти — мені не потрібен повний рецепт".

### `tail` — Показати останні 10 рядків

```bash
$ tail log-file.txt
```

Показує останні 10 рядків. Надзвичайно корисно для читання лог-файлів, де найновіші записи знаходяться внизу.

```bash
$ tail -n 20 log-file.txt    # Показати останні 20 рядків
```

> Аналогія з кухнею: "Які були останні кілька замовлень?"

## Створення файлів та директорій

### `mkdir` — Створити директорію

```bash
$ mkdir recipes
```

Це створює нову директорію під назвою `recipes` у вашому поточному місці.

Щоб створити вкладені директорії (директорію всередині директорії всередині директорії):

```bash
$ mkdir -p recipes/italian/pasta
```

Прапор `-p` означає "створити батьківські директорії за потреби" (parents). Без нього ви отримаєте помилку, якщо `recipes/` або `recipes/italian/` ще не існують.

### `touch` — Створити порожній файл

```bash
$ touch shopping-list.txt
```

Це створює новий порожній файл. (Якщо файл уже існує, це оновить позначку часу файлу без зміни вмісту.)

> Аналогія з кухнею: `mkdir` — це створення нової шухляди. `touch` — це розміщення в ній порожньої картки для рецепта.

## Приховані файли (dot-файли)

Деякі файли та директорії починаються з крапки (`.`). Вони називаються **прихованими файлами** або **dot-файлами**.

```bash
$ ls
Documents    Downloads    Music

$ ls -a
.   ..   .bashrc   .config   Documents   Downloads   Music
```

Прапор `-a` означає "all" (все) — включаючи приховані файли.

Приховані файли зазвичай містять конфігурації та налаштування. Деякі поширені:

| Файл | Що він робить |
|------|-------------|
| `.bashrc` | Налаштування для вашого термінала bash |
| `.zshrc` | Налаштування для вашого термінала zsh (стандарт у macOS) |
| `.config/` | Директорія, що містить конфігурації програм |
| `.ssh/` | SSH-ключі (використовуються для безпечних з'єднань) |
| `.gitconfig` | Налаштування Git |

> Ці файли приховані, тому що вам не потрібно бачити їх щодня, а випадкове видалення може зіпсувати ваші налаштування. Вони як електрична проводка за стінами кухні — важливі, але зазвичай не на виду.

## Основи прав доступу до файлів

> **Зупиніться та подумайте**: Коли ви запускаєте `ls -l`, ви бачите щось на кшталт `-rw-r--r--` поруч із кожним файлом. Як ви думаєте, що означають ці літери? `r` може нагадувати "read" (читання), `w` — "write" (запис)... а дефіси? Спробуйте вгадати, перш ніж читати далі.

Пам'ятаєте, як ми запускали `ls -l` і бачили це?

```
drwxr-xr-x  2 yourname yourname 4096 Mar 23 10:00 Desktop
-rw-r--r--  1 yourname yourname  220 Mar 23 08:30 notes.txt
```

Давайте розкодуємо перший стовпець: `-rw-r--r--`

### Рядок прав доступу

```
-  rw-  r--  r--
|  |    |    |
|  |    |    └── Права інших (Others — всі інші)
|  |    └── Права групи (Group)
|  └── Права власника (Owner — ви)
└── Тип файлу (- = файл, d = директорія)
```

### Три права доступу

| Літера | Право | Для файлів | Для директорій |
|--------|-----------|-----------|-----------------|
| `r` | Читання (Read) | Можна бачити вміст | Можна переглядати список вмісту |
| `w` | Запис (Write) | Можна змінювати вміст | Можна додавати або видаляти файли |
| `x` | Виконання (Execute) | Можна запускати як програму | Можна входити в директорію (`cd` в неї) |
| `-` | Немає прав | Неможливо виконати дію | Неможливо виконати дію |

### Читання прикладу

```
-rw-r--r--  notes.txt
```

- `-` : Це звичайний файл (не директорія)
- `rw-` : Власник може **читати** та **писати** (але не виконувати)
- `r--` : Група може лише **читати**
- `r--` : Інші можуть лише **читати**

```
drwxr-xr-x  Desktop
```

- `d` : Це **директорія**
- `rwx` : Власник може читати, писати та входити
- `r-x` : Група може читати та входити (але не додавати/видаляти файли)
- `r-x` : Інші можуть читати та входити (але не додавати/видаляти файли)

> Аналогія з кухнею: Права доступу — це як **у кого є який ключ**. Шеф-кухар (власник) має ключ від усього. Су-шефи (група) можуть відкрити більшість шухляд. Офіціанти (інші) можуть лише зазирнути у вікно.

> **Історія з життя**: Один розробник якось встановив права доступу до файлу конфігурації бази даних, що містив паролі, як "доступний для читання всім" (`-rw-rw-rw-`). Зловмисник, який отримав низькорівневий доступ до сервера, зміг просто за допомогою `cat` прочитати конфігураційний файл, отримати облікові дані та викрасти всю базу даних клієнтів. Належні права доступу (`-rw-------`) зупинили б цю атаку в самому зародку.

Поки що не турбуйтеся про зміну прав — просто знайте, як їх читати. Ми розберемо `chmod`, коли він вам знадобиться.

## Чи знали ви?

> 1. **Коренева директорія `/` називається "root" (корінь), тому що дерево директорій росте вниз.** Як і у справжнього дерева, корінь знаходиться вгорі, а гілки розходяться нижче. Кожен окремий файл на вашому комп'ютері знаходиться десь на гілці, яка з'єднується з `/`.
>
> 2. **Скорочення `~` було обрано через розкладку клавіатури.** На терміналі ADM-3A (виготовленому в 1970-х роках, одному з перших відеотерміналів), клавіша "Home" була на тій самій клавіші, що й символ `~`. Так `~` стало скороченням для домашньої директорії, і це збереглося на понад 50 років.
>
> 3. **Linux ставиться до всього як до файлу — включаючи вашу клавіатуру.** Коли ви натискаєте клавішу, ядро записує це натискання у файлоподібний інтерфейс. Програми читають із цього інтерфейсу, щоб знати, що ви надрукували. Ця філософія "усе є файлом" — причина, чому Linux можна повністю керувати з термінала — завжди є файл для читання або запису.

## Типові помилки

| Помилка | Що відбувається | Як виправити | Реальний вплив |
|---------|-------------|-----|-------------------|
| Спроба `cd` у файл замість директорії | Помилка `Not a directory` | Використовуйте `cat` або `head` для читання файлів; використовуйте `cd` для директорій | Втрата часу під час критичного інциденту, коли ви намагаєтеся знайти логи. |
| Забутий пробіл між командою та шляхом | Помилка `cdDocuments: command not found` | Завжди ставте пробіл: `cd Documents` | Незначне роздратування, але скрипти з пропущеними пробілами не зможуть виконуватися в автоматизованих пайплайнах. |
| Використання зворотних слешів `\` замість прямих `/` | Шлях не знайдено | Linux/macOS використовують прямі слеші: `/home/you/Documents` | Скрипт, написаний на Windows, може повністю зламатися при розгортанні на вузлі Kubernetes з Linux. |
| Створення файлу, коли ви мали на увазі директорію | Ви отримуєте файл з назвою "recipes" замість папки | Використовуйте `mkdir` для директорій, `touch` для файлів | Програми, що очікують директорію для запису логів, впадуть, якщо замість неї там буде файл. |
| Втрата орієнтації у вкладених директоріях | Ви не знаєте, де ви | Запустіть `pwd`, щоб побачити ваше місцезнаходження, або `cd ~`, щоб повернутися додому | Ви можете випадково видалити або перезаписати файли не в тому середовищі (наприклад, у продакшені замість девелопменту). |
| Друкарська помилка у назві шляху | `No such file or directory` | Спочатку використовуйте `ls`, щоб побачити, що там є, потім уважно друкуйте (або використовуйте Tab!) | Скрипти автоматичного резервного копіювання мовчки не зможуть зберегти дані, якщо цільовий шлях написаний з помилкою. |

> **Порада професіонала: Tab-автодоповнення.** Почніть вводити назву файлу або директорії та натисніть **Tab**. Термінал автоматично допише її за вас. Якщо є кілька збігів, натисніть **Tab** двічі, щоб побачити всі варіанти. Це економить час і допомагає уникнути друкарських помилок.

## Контрольні запитання

**Питання 1**: Ви читаєте документацію, яка каже вам скопіювати ліцензійний ключ у `~/.kube/config`. Проте ваша поточна робоча директорія — `/var/log/pods/`. Де саме символ `~` вказує системі шукати цей файл, і чому це скорочення використовується замість повного шляху?

<details>
<summary>Показати відповідь</summary>

Система шукатиме в домашній директорії вашого користувача (наприклад, `/home/yourname/.kube/config` у Linux або `/Users/yourname/.kube/config` у macOS). Символ `~` діє як динамічне скорочення, яке завжди вказує на домашню директорію поточного користувача. Це неймовірно корисно в документації та скриптах, оскільки воно працює бездоганно незалежно від того, яке ваше конкретне ім'я користувача або яку операційну систему ви використовуєте. Використовуючи `~`, розробники можуть написати одну команду, яка працюватиме на особистих машинах усіх користувачів без потреби в налаштуванні.

</details>

**Питання 2**: Ви щойно завантажили інструмент, який містить файл `.env` із секретними ключами API. Коли ви вводите `ls` у директорії, файл не відображається. Чому це відбувається і яку команду ви повинні запустити, щоб переконатися, що файл насправді там є?

<details>
<summary>Показати відповідь</summary>

Ви повинні запустити `ls -a` (або `ls --all`), щоб побачити його. Файл не з'являється при звичайній команді `ls`, тому що його назва починається з крапки (`.`), що робить його прихованим файлом. Операційні системи приховують dot-файли за замовчуванням, щоб підтримувати візуальну чистоту директорій, оскільки ці файли зазвичай містять конфігураційні дані або дані середовища, з якими вам не потрібно взаємодіяти під час звичайної щоденної роботи з файлами. Прапор `-a` спеціально скасовує цю поведінку за замовчуванням, щоб показати все, що є в директорії.

</details>

**Питання 3**: Молодший розробник засмучений тим, що не може легко знайти свій файл конфігурації термінала `.bashrc` у домашній папці за допомогою графічного провідника. Чому конфігураційні файли на кшталт `.bashrc` приховані за замовчуванням, і що може піти не так, якби вони були повністю видимі поруч зі звичайними документами?

<details>
<summary>Показати відповідь</summary>

Конфігураційні файли приховані за замовчуванням, щоб захистити їх від випадкової зміни або видалення. Якби файли на кшталт `.bashrc` були видимі поруч із повсякденними документами, користувач міг би помилково видалити їх під час очищення старих файлів або випадково змінити їх, намагаючись відкрити звичайний текстовий документ. Видалення або пошкодження цих файлів може миттєво зіпсувати середовище вашого термінала, зламати налаштування програм або заблокувати доступ до певних інструментів. Приховування гарантує, що з ними взаємодіятимуть лише ті користувачі, які спеціально мають намір змінити конфігурації.

</details>

**Питання 4**: Ви зараз займаєтеся пошуком несправностей у програмі, і ваш термінал знаходиться в `/home/user/projects/app/src`. Ви розумієте, що вам потрібно прочитати інструкції, розташовані в `/home/user/projects/app/README.md`. Напишіть команду для читання цього файлу за допомогою відносного шляху та поясніть, чому тут може бути кращим саме відносний шлях.

<details>
<summary>Показати відповідь</summary>

Команда: `cat ../README.md` (або `head ../README.md`). Символ `..` каже системі піднятися на один рівень вгору в директорію `app`, а потім шукати там файл `README.md`. Відносний шлях тут кращий, тому що його набагато швидше надрукувати, ніж повний абсолютний шлях (`/home/user/projects/app/README.md`). Під час активної сесії пошуку несправностей навігація за допомогою відносних шляхів економить час, зменшує ймовірність друкарських помилок і полегшує переміщення всередині локалізованої структури проєкту.

</details>

**Питання 5**: Скрипт розгортання не спрацьовує. Всередині скрипту він намагається отримати доступ до конфігураційного файлу, викликаючи `cd config/`. Однак скрипт працює лише тоді, коли його запускають із конкретної папки, і ламається, якщо його запускати звідкись ще. У чому принципова різниця між використанням `config/` проти `/etc/app/config/` і чому скрипт зламався?

<details>
<summary>Показати відповідь</summary>

Скрипт зламався, тому що використовував відносний шлях (`config/`), який повністю залежить від вашої поточної робочої директорії на момент виконання скрипту. Якщо ви не перебуваєте в батьківській директорії `config/`, система не знайде її, і скрипт не спрацює. Навпаки, `/etc/app/config/` — це абсолютний шлях. Абсолютні шляхи починаються від кореневої директорії (`/`) і вказують на точне, однозначне місцезнаходження цілі, гарантуючи, що скрипт знайде папку незалежно від того, звідки його було запущено.

</details>

**Питання 6**: Ви розслідуєте інцидент безпеки. Чутливий файл, що містить електронні адреси клієнтів, має рядок прав доступу `-rw-r-----`. Власником файлу є користувач `admin`, а файл належить групі `support`. Якщо новий користувач приєднується до групи `support`, які саме дії він може виконувати з цим файлом і що заважає йому змінювати його?

<details>
<summary>Показати відповідь</summary>

Користувач у групі `support` може лише **читати** вміст файлу, оскільки сегмент прав групи — `r--`. Він не може змінювати або видаляти вміст, оскільки для групи відсутнє право на запис (`w`). Лише власник файлу (`admin`) має права і на читання, і на запис (`rw-`). Такий поділ гарантує, що персонал підтримки може переглядати необхідну інформацію для допомоги клієнтам, не маючи можливості випадково змінити або пошкодити чутливі дані.

</details>

**Питання 7**: Вузол Kubernetes у продакшені падає, а файл системного логу `/var/log/syslog` виріс до понад 500 000 рядків. Вам потрібно швидко визначити помилку, яка сталася безпосередньо перед падінням. Яку команду ви повинні використати для перегляду файлу і чому запуск `cat /var/log/syslog` був би катастрофічним вибором у цьому сценарії?

<details>
<summary>Показати відповідь</summary>

Ви повинні використати `tail -n 50 /var/log/syslog` (або подібну), щоб переглянути лише кінець файлу. Використання `cat` було б катастрофічним вибором, оскільки вона спробувала б вивести всі 500 000 рядків у ваш термінал одночасно. Це б завалило ваш екран, заморозило сесію термінала і зробило б абсолютно неможливим пошук критичних повідомлень про помилки, прихованих у самому низу файлу, де записані найновіші події. Команда `tail` існує саме для такого випадку, дозволяючи вам ефективно перевіряти найсвіжіші системні логи.

</details>

## Практична вправа: Створення вашої першої структури директорій

### Мета

Створити структуру директорій, додати файли, прочитати їх та перевірити права доступу — і все це через термінал.

### Кроки

1. **Перейдіть до своєї домашньої директорії:**

```bash
$ cd ~
$ pwd
```

Підтвердьте, що ви бачите шлях до своєї домашньої директорії.

2. **Створіть структуру директорій проєкту:**

```bash
$ mkdir -p kubedojo-practice/recipes/appetizers
$ mkdir -p kubedojo-practice/recipes/main-courses
$ mkdir -p kubedojo-practice/recipes/desserts
```

3. **Перейдіть всередину:**

```bash
$ cd kubedojo-practice
$ ls
```

Ви повинні побачити: `recipes`

4. **Створіть кілька файлів:**

```bash
$ touch recipes/appetizers/bruschetta.txt
$ touch recipes/main-courses/pasta-carbonara.txt
$ touch recipes/desserts/tiramisu.txt
```

5. **Додайте вміст у файл** (ми використаємо `echo` з `>` для запису у файл):

```bash
$ echo "Ingredients: bread, tomatoes, basil, olive oil" > recipes/appetizers/bruschetta.txt
$ echo "Ingredients: pasta, eggs, pancetta, parmesan" > recipes/main-courses/pasta-carbonara.txt
$ echo "Ingredients: coffee, mascarpone, ladyfingers, cocoa" > recipes/desserts/tiramisu.txt
```

> Символ `>` означає "направити вивід у цей файл" замість відображення на екрані. Сприймайте це як переспрямування відповіді шеф-кухаря: замість того, щоб бути промовленою вголос, вона записується на картку з рецептом.

6. **Прочитайте файли:**

```bash
$ cat recipes/appetizers/bruschetta.txt
$ head recipes/main-courses/pasta-carbonara.txt
$ tail recipes/desserts/tiramisu.txt
```

7. **Перевірте права доступу:**

> **Зупиніться та подумайте**: Перш ніж запустити наступну команду, який рядок прав доступу ви очікуєте побачити для файлу `bruschetta.txt`? (Підказка: Ви його створили, отже, ви — власник. Чи можете ви читати та писати в нього?)

```bash
$ ls -l recipes/appetizers/
```

Ви повинні побачити рядок прав доступу для вашого файлу bruschetta.

8. **Перевірте наявність прихованих файлів у вашій домашній директорії:**

> **Зупиніться та подумайте**: Якби ви запустили просто `ls ~` без прапора `-a`, чи побачили б ви файли на кшталт `.bashrc` або `.config`? Чому так або чому ні?

```bash
$ ls -a ~
```

Шукайте файли, що починаються з `.` — це ваші dot-файли!

9. **Попереміщуйтеся в системі:**

```bash
$ cd recipes/desserts
$ pwd                          # Де ви?
$ cd ..                        # Перейти на один рівень вгору
$ pwd                          # Тепер де?
$ cd ~                         # Перейти додому
$ pwd                          # Знову вдома
```

### Критерії успіху

Ви завершили цю вправу, якщо ви можете:

- [ ] Переходити до своєї домашньої директорії за допомогою `cd ~`
- [ ] Створювати вкладені директорії за допомогою `mkdir -p`
- [ ] Створювати файли за допомогою `touch`
- [ ] Записувати вміст у файли за допомогою `echo "text" > file`
- [ ] Читати файли за допомогою `cat`, `head` та `tail`
- [ ] Перевіряти права доступу до файлів за допомогою `ls -l`
- [ ] Переглядати приховані файли за допомогою `ls -a`
- [ ] Навігувати за допомогою `cd`, `cd ..` та `cd ~`

> Ви щойно використали інструмент, яким досвідчені інженери користуються щодня. Ви на своєму місці.

## Наступний модуль

Тепер ви вмієте навігувати файловою системою, створювати файли та директорії, читати файли та розуміти права доступу. Кухня починає здаватися знайомою.

**Наступний модуль**: [Модуль 0.5: Редагування файлів](../module-0.5-editing-files/) — Навчіться по-справжньому наповнювати файли вмістом за допомогою текстового редактора, який працює прямо у вашому терміналі.
```
