## Що ви зможете зробити

Після цього модуля ви зможете:
- **Пояснити**, що таке сервер і чому йому не потрібні екран або клавіатура
- **Підключитися** до віддаленої машини за допомогою SSH та керувати нею через термінал
- **Розрізняти** автентифікацію за паролем та на основі ключів SSH і пояснити, чому ключі безпечніші
- **Перевірити**, чи перебуваєте ви на потрібній машині після підключення (використовуючи `hostname`, `whoami`, `pwd`)

## Чому це важливо

Все, що ви робили до цього часу, відбувалося на *вашому* комп'ютері. Ваша "кухня", ваші файли, ваш термінал. Але в реальному світі технологій більшість подій відбувається на комп'ютерах, які знаходяться *десь в іншому місці* — у центрі обробки даних, у хмарі або в стійці з обладнанням, до якого ви ніколи не торкнетеся фізично.

Ці інші комп'ютери називаються **серверами**, а спосіб взаємодії з ними — через **SSH**.

Працюючи з Kubernetes, ви будете керувати серверами (їх буде багато). Розуміння того, що таке сервери і як до них підключатися, не є факультативним — це фундамент.

---

## Що таке сервер?

Сервер — це просто комп'ютер. Це все. Ніякої магії чи таємниць.

Слово "сервер" описує те, що комп'ютер *робить*, а не те, чим він *є*. Сервер — це комп'ютер, робота якого полягає в тому, щоб **обслуговувати** (serve) інші комп'ютери.

```
Ваш ноутбук:
  - Має екран, клавіатуру, трекпад
  - Створений для ОДНІЄЇ людини, яка сидить перед ним
  - Запускає графічний інтерфейс (вікна, іконки, миша)
  - Ви переглядаєте вебсторінки, пишете документи, дивитеся відео

Сервер:
  - Часто НЕ МАЄ екрана, клавіатури чи миші
  - Створений для обслуговування БАГАТЬОХ користувачів/комп'ютерів одночасно
  - Зазвичай запускає ТІЛЬКИ термінальний інтерфейс (без робочого столу)
  - Він хостить вебсайти, запускає бази даних, обробляє дані
```

### Аналогія з кухнею

Подумайте про це так:

- **Ваш ноутбук** — це домашня кухня. Ви готуєте для себе. Ви їсте в тій самій кімнаті, де готуєте. "Шеф-кухар" і "клієнт" — це одна і та сама особа.

- **Сервер** — це кухня ресторану. Вона існує для того, щоб готувати їжу та відправляти її *назовні* до обіднього залу, повного клієнтів. На кухні немає обідніх столів — це не її робота. Її робота — готувати та подавати.

Коли ви відвідуєте вебсайт, ваш браузер (клієнт в обідньому залі) надсилає запит на сервер (кухня позаду). Сервер готує відповідь (готує страву) і надсилає її назад у ваш браузер (подає страву).

## Ваш ноутбук проти сервера

"Під капотом" вони мають однакові основні частини:

| Компонент | Ваш ноутбук | Типовий сервер |
|-----------|-------------|-----------------|
| CPU | 4-8 ядер | 16-128 ядер |
| RAM | 8-32 ГБ | 64-512 ГБ |
| Накопичувач | 256 ГБ - 2 ТБ | 1 ТБ - 100 ТБ+ |
| Екран | Так | Зазвичай ні |
| Клавіатура | Так | Зазвичай ні |
| Операційна система | macOS/Windows | Linux (майже завжди) |
| Призначення | Одна людина, багато завдань | Багато клієнтів, конкретні завдання |

Сервер — це, по суті, потужний комп'ютер, оптимізований для обслуговування багатьох запитів одночасно, роботи в режимі 24/7 та віддаленого керування.

## Локальний проти віддаленого

Два слова, які ви чутимете постійно:

- **Локальний** (local) = ваш комп'ютер, прямо тут, перед вами
- **Віддалений** (remote) = інший комп'ютер, десь в іншому місці, до якого ви підключаєтеся через мережу

```
Локально:  Ваша кухня. Ви стоїте в ній.
Віддалено: Кухня в іншому місті. Ви телефонуєте їм, щоб дати замовлення.
```

Коли ви вводите `ls` у своєму терміналі прямо зараз, ця команда виконується **локально** — на вашому комп'ютері.

Коли ви підключаєтеся до сервера і вводите `ls`, ця команда виконується **віддалено** — на сервері. Але у вашому терміналі це виглядає абсолютно однаково. У цьому і полягає вся краса.

### Швидка перевірка: локально чи віддалено?

> **Зупиніться та подумайте**: Класифікуйте ці 5 сценаріїв як локальні або віддалені операції.
> 1. Редагування фотографії на робочому столі вашого ноутбука.
> 2. Використання SSH для перевірки логів вебсервера в Лондоні.
> 3. Запуск `pwd` у вашому терміналі відразу після його відкриття.
> 4. Введення `ls` після підключення через `ssh admin@10.0.0.5`.
> 5. Запит вашого браузера на вебсторінку з Вікіпедії.

<details>
<summary>Відповіді</summary>
1. **Локально** (відбувається на машині перед вами)
2. **Віддалено** (ви кажете далекому серверу показати його логи)
3. **Локально** (ви ще не підключилися до іншої машини)
4. **Віддалено** (ваш термінал тепер керує машиною 10.0.0.5)
5. **Віддалено** (сервер Вікіпедії надсилає вам дані сторінки)
</details>

## SSH: Ваш безпечний тунель до віддалених кухонь

**SSH** розшифровується як **S**ecure **Sh**ell (безпечна оболонка). Це програма, яка дозволяє безпечно відкрити термінальну сесію на віддаленому комп'ютері.

Уявіть SSH як **захищену телефонну лінію до віддаленої кухні**. Ви берете слухавку (відкриваєте SSH), набираєте номер (підключаєтеся до сервера) і починаєте давати замовлення (вводити команди). Шеф-кухар на віддаленій кухні виконує їх, і ви чуєте результати через телефон.

Частина "безпечна" (secure) є важливою: все, що ви надсилаєте через SSH, шифрується. Ніхто не може підслухати вашу розмову. Це як мати приватну, зашифровану телефонну лінію.

### Кілька слів про змінні середовища

Іноді ви бачитимете речі на кшталт `$USER` або `$HOME` у командах або документації. Це **змінні середовища** (environment variables) — іменовані контейнери, які ваша система автоматично заповнює корисними значеннями.

- `$USER` містить ваше ім'я користувача
- `$HOME` містить шлях до вашої домашньої директорії

Спробуйте їх:

```bash
echo $USER
echo $HOME
```

Вам не потрібно їх налаштовувати — ваш комп'ютер робить це за вас під час входу в систему. Ми згадуємо про це зараз, тому що команди SSH часто використовують `$USER`, і ви будете бачити змінні середовища протягом усієї вашої кар'єри.

### Спробуйте зараз

Запустіть ці команди у своєму терміналі прямо зараз:

```bash
echo $USER
echo $HOME
```

> **Зупиніться та подумайте**: Якщо ви підключитеся до віддаленого сервера за допомогою `ssh chef@192.168.1.100`, а потім запустите `echo $USER`, що він виведе? Ім'я користувача вашого ноутбука чи `chef`?
>
> <details>
> <summary>Відповідь</summary>
> Він виведе <code>chef</code>! Коли ви підключаєтеся через SSH до сервера, команди виконуються в контексті віддаленого користувача. Змінна <code>$USER</code> на цьому сервері встановлюється на ім'я користувача, під яким ви увійшли.
> </details>

### Команда SSH

Базова команда SSH виглядає так:

```bash
ssh username@ip-address
```

Розберемо її:

- `ssh` — програма, яку ви запускаєте
- `username` — ім'я вашого облікового запису на віддаленому сервері (як ваш бейдж на віддаленій кухні)
- `@` — просто роздільник (символ "at")
- `ip-address` — адреса віддаленого сервера (як номер телефону кухні)

Реальний приклад може виглядати так:

```bash
ssh chef@192.168.1.100
```

Або з доменним ім'ям замість IP-адреси:

```bash
ssh chef@kitchen.example.com
```

### Що відбувається під час підключення

```
1. Ви вводите: ssh chef@kitchen.example.com
2. SSH зв'язується з віддаленим сервером
3. Сервер запитує: "Хто ви? Доведіть це."
4. Ви надаєте доказ (пароль або ключ — про це нижче)
5. Сервер каже: "Ласкаво просимо, шеф. Ось ваш термінал."
6. Ваше запрошення (prompt) змінюється, показуючи назву віддаленого сервера
7. Кожна введена вами команда тепер виконується на ВІДДАЛЕНОМУ сервері
8. Введіть "exit", щоб відключитися та повернутися до локального термінала
```

Запрошення вашого термінала може змінитися з:

```
yourname@your-laptop ~ $
```

на:

```
chef@remote-server ~ $
```

Саме так ви дізнаєтеся, що підключені до іншої машини.

## Паролі проти SSH-ключів

Є два способи підтвердити свою особу на віддаленому сервері:

### Паролі

Простий спосіб. Сервер запитує пароль, ви його вводите.

```bash
ssh chef@kitchen.example.com
# Сервер запитує: chef@kitchen.example.com's password:
# Ви вводите пароль (він не відображатиметься на екрані — це нормально)
```

Паролі працюють, але з ними є проблеми:
- Ви повинні вводити їх щоразу
- Їх можна вгадати або вкрасти
- Вони незручні для автоматизованих систем

### Ключі SSH (Кращий спосіб)

> **Подумайте про це**: Паролі можна вгадати, вкрасти або виманити за допомогою фішингу. Що, якби замість того, щоб повідомляти серверу секретне слово, ви могли б довести свою особу за допомогою чогось, що належить тільки вам — як фізичний ключ, що підходить до певного замка? Саме це роблять ключі SSH. Сервер ніколи не дізнається ваш "пароль" — він лише перевіряє, чи підходить ваш ключ.

Ключі SSH працюють як система замка та ключа:

```
Ви маєте КЛЮЧ (private key) — зберігається на вашому комп'ютері, нікому не передається.
Сервер має ЗАМОК (public key) — він знає, який ключ підходить.

Коли ви підключаєтеся:
1. Ви пред'являєте свій ключ
2. Сервер перевіряє: "Чи підходить цей ключ до мого замка?"
3. Якщо так: "Заходьте!" (пароль не потрібен)
4. Якщо ні: "Доступ заборонено."
```

Це як мати фізичний ключ від дверей віддаленої кухні. Вам не потрібно нікому називати пароль — ви просто відмикаєте двері.

**Генерація ключів SSH** (вам не потрібно робити це прямо зараз, просто знайте як):

```bash
ssh-keygen -t ed25519
```

Це створює два файли:
- `~/.ssh/id_ed25519` — Ваш **приватний ключ** (ключ). НІКОЛИ нікому його не передавайте.
- `~/.ssh/id_ed25519.pub` — Ваш **публічний ключ** (замок). Його можна вільно поширювати.

Ви копіюєте публічний ключ на сервер, і відтоді можете підключатися без пароля.

> **Подумайте про це так**: Приватний ключ — це ключ від вашого будинку. Публічний ключ — це копія замка на ваших дверях. Ви можете дати замок будь-кому і сказати: "Постав це на свої двері". Тепер ваш ключ відкриває і їхні двері теж. Але ніхто не зможе зробити ключ, просто дивлячись на замок.

## Поширені параметри SSH

| Параметр | Що він робить | Приклад |
|--------|-------------|---------|
| `-p` | Підключитися через інший порт (типовий — 22) | `ssh -p 2222 chef@server.com` |
| `-i` | Використовувати конкретний файл ключа | `ssh -i ~/.ssh/mykey chef@server.com` |
| `-v` | Докладний режим (показує, що відбувається — корисно для відладки) | `ssh -v chef@server.com` |

### Міні-вправа: Побудуйте команду

> **Зупиніться та подумайте**: Напишіть точну команду SSH для підключення користувача `admin` до сервера `10.0.0.5`, використовуючи порт `2222` та файл ключа `~/.ssh/work_key`.

<details>
<summary>Відповідь</summary>

```bash
ssh -p 2222 -i ~/.ssh/work_key admin@10.0.0.5
```
*(Примітка: порядок `-p` та `-i` не має значення, головне, щоб вони йшли перед частиною `user@host`).*
</details>

## Життєвий цикл з'єднання

```
Ваш комп'ютер                          Віддалений сервер
    |                                       |
    |  --- ssh chef@server.com ---------->  |
    |                                       |  "Запит на з'єднання отримано"
    |  <--- "Доведіть свою особу" --------  |
    |                                       |
    |  --- надсилає ключ/пароль ---------->  |
    |                                       |  "Особу підтверджено"
    |  <--- "Вітаємо! Ось ваша оболонка" --  |
    |                                       |
    |  --- ls, pwd, nano тощо ----------->  |  (команди виконуються ТУТ, на сервері)
    |  <--- [ ПРОПУСК 1: Вгадайте, що відбувається тут ] --- |
    |                                       |
    |  --- [ ПРОПУСК 2: Як відключитися? ] ------> |
    |                                       |  "До побачення"
    |  (повернення до локального термінала) |
```

> **Зупиніться та подумайте**: Заповніть два пропуски в діаграмі вище. Що сервер надсилає назад після того, як ви запустили команду, і яку команду ви вводите, щоб відключитися?
>
> <details>
> <summary>Відповіді</summary>
> <strong>ПРОПУСК 1:</strong> Сервер надсилає назад результати/вивід ваших команд.<br>
> <strong>ПРОПУСК 2:</strong> Ви вводите <code>exit</code>, щоб відключитися та повернутися до локального термінала.
> </details>

> **Зупиніться та подумайте**: Якщо ви підключені до сервера через SSH і запускаєте `rm -rf /home/yourname/projects`, що станеться? Чи видалить це файли на вашому ноутбуці чи на сервері? Це не питання з підступом — але саме така помилка спричиняла реальні збої. Завжди перевіряйте `hostname` після підключення, щоб підтвердити, що ви перебуваєте саме на тій машині, на якій думаєте.

**Ключове розуміння**: Коли ви підключені через SSH, ваш термінал *виглядає* так само, але кожна команда виконується на віддаленій машині. Якщо ви створюєте файл, він створюється на сервері, а не на вашому ноутбуці. Якщо ви запускаєте `pwd`, він показує директорію сервера, а не вашу.

## Чому це важливо для Kubernetes

Kubernetes працює на серверах — зазвичай на багатьох. Типовий кластер Kubernetes може мати:

- 3 сервери для "control plane" (офіс управління рестораном)
- 10-100+ серверів як "worker nodes" (кухні, які безпосередньо готують)

Ви будете використовувати SSH для підключення до цих серверів, усунення несправностей, перевірки логів та керування конфігураціями. Команди, які ви вивчили в попередніх модулях — `ls`, `cd`, `nano`, `cat` — працюють на цих віддалених серверах точно так само.

## Чи знали ви?

- **SSH був винайдений у 1995 році фінським дослідником** на ім'я Тату Ілонен після того, як його університетська мережа зазнала атаки з перехопленням паролів. До SSH люди використовували `telnet` для підключення до віддалених комп'ютерів, який передавав усе (включаючи паролі) у відкритому тексті, який міг прочитати будь-хто в мережі. SSH зашифрував з'єднання, зробивши його безпечним. Це була настільки гарна ідея, що вона стала світовим стандартом майже миттєво.

- **Порт 22 був призначений особисто.** Коли Тату Ілонен створив SSH, йому потрібен був номер порту. Він написав електронного листа організації, яка керує номерами портів (IANA), і вони просто призначили йому порт 22. Він отримав його протягом дня. Сьогодні порт 22 є одним із найбільш впізнаваних номерів портів в обчислювальній техніці.

- **Міжнародна космічна станція використовує SSH.** Астронавти та центр управління польотами використовують SSH для безпечного керування комп'ютерними системами станції. Той самий інструмент, який ви вивчаєте в цьому модулі, використовується для керування комп'ютерами на орбіті Землі.

## Типові помилки

| Помилка | Чому це проблема | Що робити замість цього |
|---------|-------------------|-------------------|
| Передача вашого приватного ключа | Будь-хто з вашим приватним ключем може отримати доступ до ваших серверів | Ніколи не передавайте `id_ed25519` (файл БЕЗ `.pub`). Передавайте тільки файл `.pub` |
| Забуваєте, що ви на віддаленому сервері | Ви можете видалити файли або змінити налаштування не на тій машині | Завжди звертайте увагу на запрошення термінала та використовуйте `hostname`, щоб перевірити, де ви |
| Введення пароля не в те поле | SSH приховує введення пароля (ні крапок, ні зірочок). Люди думають, що він зламався, і вводять його десь на видноті | Коли SSH запитує пароль, просто вводіть його "наосліп" і натисніть Enter. Він ОТРИМУЄ ваші натискання клавіш |
| Не відключаєтеся після завершення роботи | Залишення відкритих сесій SSH витрачає ресурси сервера і може бути ризиком для безпеки | Вводіть `exit` або натискайте `Ctrl + D`, коли закінчите |
| Паніка при розриві з'єднання | Мережеві перебої трапляються — це не означає, що щось зламалося | Просто підключіться знову за допомогою тієї ж команди SSH. Ваші файли на сервері нікуди не зникли |

## Контрольні запитання

1. **Ви підключилися до сервера через SSH і запустили `hostname` — він показує назву вашого ноутбука. Що сталося?**
   <details>
   <summary>Відповідь</summary>
   Насправді ви не підключені до віддаленого сервера! Або з'єднання SSH не вдалося, або ви вже ввели `exit` і повернулися до свого локального термінала, не усвідомивши цього. Оскільки ви запустили `hostname` і побачили назву свого ноутбука, ви підтвердили, що ваші команди наразі виконуються локально, а не віддалено. Завжди перевіряйте своє запрошення та використовуйте `hostname` для перевірки середовища, щоб уникнути виконання руйнівних команд не на тій машині.
   </details>

2. **Ваш колега просить вас надіслати йому електронною поштою ваш приватний ключ SSH, щоб він міг швидко увійти на сервер, яким ви обоє керуєте. Що ви відповісте і чому?**
   <details>
   <summary>Відповідь</summary>
   Ви повинні відповісти категоричною відмовою. Ваш приватний ключ (наприклад, `id_ed25519`) — це ваш особистий "ключ від будинку", і його ніколи не можна нікому передавати, навіть колегам. Якщо їм потрібен доступ до сервера, вони повинні згенерувати власну пару ключів SSH і дати вам свій *публічний* ключ (`id_ed25519.pub`), який ви потім зможете додати до списку дозволених на сервері. Це гарантує безпеку доступу для кожного і дозволяє вам пізніше відкликати їхній доступ, не змінюючи власний ключ.
   </details>

3. **Яка різниця між приватним і публічним ключами?**
   <details>
   <summary>Відповідь</summary>
   Приватний ключ залишається на вашому комп'ютері і ніколи не повинен передаватися — це як ваш ключ від будинку. Публічний ключ можна вільно поширювати і розміщувати на серверах, до яких ви хочете мати доступ — це як замок. Коли ви підключаєтеся, сервер перевіряє, чи відповідає ваш приватний ключ збереженому публічному ключу. Якщо відповідає — ви всередині. Знання публічного ключа нікому не допоможе видати себе за вас — для цього потрібен приватний ключ.
   </details>

4. **Коли ви підключені до віддаленого сервера через SSH і вводите `touch recipe.txt`, де створюється файл?**
   <details>
   <summary>Відповідь</summary>
   На віддаленому сервері, а не на вашій локальній машині. Коли ви підключені через SSH, кожна команда, яку ви вводите, виконується на віддаленому сервері. Файл `recipe.txt` створюється у файловій системі сервера. Ваша локальна машина не отримує копії. Щоб перевірити, на якій машині ви перебуваєте, подивіться на запрошення термінала або введіть `hostname`.
   </details>

5. **Вищий рівень складності: Вам потрібно автоматизувати розгортання з сервера CI (Continuous Integration) на 50 робочих машин. Ви б використали автентифікацію за паролем чи за ключем? Поясніть свою думку.**
   <details>
   <summary>Відповідь</summary>
   Ви повинні використовувати автентифікацію на основі ключів. Якщо використовувати паролі, автоматизований скрипт або застрягне в очікуванні людини, яка введе пароль 50 разів, або вам доведеться прописувати пароль прямо в скрипті (що є величезним ризиком для безпеки). З ключами SSH сервер CI може безпечно зберігати приватний ключ, а 50 робочих машин можуть бути налаштовані на довіру до його публічного ключа. Це дозволяє автоматизованій системі підключатися миттєво і безпечно без втручання людини, роблячи розгортання швидким і захищеним.
   </details>

## Практична вправа: SSH до localhost

Ми потренуємося використовувати SSH, підключившись до вашого власного комп'ютера. Це може здатися безглуздим ("Я ж і так тут!"), але це точно демонструє, як працює SSH — і досвід ідентичний підключенню до справжнього віддаленого сервера.

### На macOS:

Спочатку увімкніть віддалений вхід (SSH), якщо він ще не увімкнений:

1. Відкрийте **Системні параметри** (System Settings)
2. Перейдіть до **Загальні > Спільний доступ** (General > Sharing)
3. Увімкніть **Віддалений вхід** (Remote Login)

Тепер у вашому терміналі:

```bash
ssh localhost
```

Ви побачите щось на кшталт:

```
The authenticity of host 'localhost (127.0.0.1)' can't be established.
ED25519 key fingerprint is SHA256:AbCdEf123456...
Are you sure you want to continue connecting (yes/no)?
```

Введіть `yes` і натисніть Enter. (Це з'являється лише під час першого підключення до нового сервера. Комп'ютер каже: "Я ніколи раніше не спілкувався з цим сервером — ви впевнені, що він справжній?")

Введіть свій пароль Mac, коли з'явиться відповідний запит.

Тепер ви "підключені". Спробуйте:

```bash
hostname
pwd
ls
echo "Привіт з SSH!"
```

Все виглядає так само, тому що ви підключилися до самих себе через SSH. Але механізм ідентичний підключенню до сервера на іншому кінці світу.

Щоб відключитися:

```bash
exit
```

### На Linux:

У більшості систем Linux SSH увімкнено за замовчуванням. Якщо ні:

```bash
sudo apt install openssh-server    # Debian/Ubuntu
sudo systemctl start sshd          # Запустити сервіс SSH
```

Потім:

```bash
ssh localhost
```

Виконайте ті самі кроки, що й для macOS вище.

### На Windows:

SSH до localhost на Windows налаштувати складніше. Натомість просто потренуйте синтаксис команди:

```
# Ось як виглядало б підключення до реального сервера:
ssh yourname@192.168.1.100

# Ви побачите:
yourname@192.168.1.100's password:

# Після введення пароля:
yourname@remote-server:~$

# Тепер кожна команда виконується на віддаленому сервері.
# Введіть "exit", щоб відключитися.
```

### Що сталося б із реальним сервером

Якби у вас був хмарний сервер (ви отримаєте його, коли почнете трек Kubernetes), досвід був би таким:

```bash
# Підключення до сервера в хмарі
ssh ubuntu@54.123.45.67

# Ви б побачили:
ubuntu@ip-54-123-45-67:~$

# Тепер ви всередині Linux-сервера десь у центрі обробки даних
# Кожна команда виконується там:
ls          # Показує файли на сервері
pwd         # Показує ваш шлях на сервері
nano        # Відкриває nano на сервері
cat /etc/os-release    # Показує операційну систему сервера

# Відключіться, коли закінчите
exit
```

Це виглядало б, відчувалося б і працювало б точно так само, як вправа з localhost. Єдина різниця — фізичне розташування комп'ютера.

**Критерії успіху**: Ви повинні запустити `hostname` у своєму терміналі *перед* підключенням через SSH і записати результат. Потім успішно відкрити з'єднання SSH і знову запустити `hostname` *після* підключення, щоб довести, що ви перебуваєте в іншому середовищі. Ви повинні зафіксувати цю різницю у виводі, запустити принаймні ще одну команду для перевірки файлової системи, а потім правильно відключитися. Це доведе, що ви розумієте, як перевірити, якою машиною ви зараз активно керуєте.
