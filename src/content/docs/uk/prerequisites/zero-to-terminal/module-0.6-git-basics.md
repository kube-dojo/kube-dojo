# Модуль 0.6: Основи Git — Відстежуйте свою роботу

**Складність**: [ПОЧАТКІВЕЦЬ]
**Час на виконання**: 45 хвилин
**Попередні вимоги**: Модуль 0.5 (Редагування файлів)

## Що ви зможете зробити

Після завершення цього модуля ви зможете:
- **Ініціалізувати** локальні репозиторії Git та **налаштувати** ідентифікацію користувачів для точного визначення авторства комітів.
- **Побудувати** логічну історію комітів, вибірково додаючи змінені файли до індексу за допомогою `git add` та `git commit`.
- **Діагностувати** неочікувані стани репозиторію, аналізуючи вивід команд `git status`, `git log` та `git diff`.
- **Синхронізувати** локальні репозиторії з віддаленими серверами за допомогою `git push` та `git pull`.
- **Сформулювати** файл `.gitignore` для запобігання відстеженню конфіденційних даних або згенерованих артефактів.

## Чому це важливо

Середня за розміром фінансово-технологічна компанія використовувала спільний мережевий диск для зберігання файлів конфігурації своєї інфраструктури. Інженери підключали диск, відкривали файли у своїх локальних редакторах, вносили зміни та зберігали їх назад у мережу. Одного п'ятничного дня молодшому інженеру доручили оновити тестове серевовище. Помилково він відкрив файл `payment-processor-deployment.yaml` для продуктового середовища замість тестового варіанту. Він змінив тег образу контейнера на експериментальну збірку, зберіг файл і застосував конфігурацію до кластера.

Протягом декількох хвилин система обробки платежів почала втрачати транзакції. Коли старші інженери поспішили розібратися, вони подивилися на конфігураційний файл на мережевому диску, але він показував лише поточний, непрацюючий стан. Оскільки не було системи контролю версій, не існувало історичного запису про те, як виглядав файл десять хвилин тому, хто його змінив і чому. Команді довелося вручну відновлювати продуктовий конфіг з пам'яті та застарілої документації. Подальша затримка у відновленні сервісу зупинила всі транзакції клієнтів на шість годин, що призвело до мільйонів доларів втраченого доходу та серйозної шкоди репутації.

Контроль версій запобігає саме такому сценарію. Git, галузевий стандарт контролю версій, діє як незнищенна машина часу для вашого коду та конфігурації. Він фіксує кожну зміну, точно ідентифікує того, хто її зробив, і дозволяє миттєво повернутися до будь-якого попереднього стану. У сучасному хмарно-орієнтованому (cloud-native) світі інфраструктура визначається як код. Якщо ви не можете відстежувати, переглядати та скасовувати свій інфраструктурний код з абсолютною впевненістю, ви працюєте в очікуванні катастрофи. Оволодіння Git не є необов'язковим для platform-інженерів; це фундаментальна навичка, на якій базується вся надійна автоматизація та співпраця.

Без Git не можуть існувати сучасні конвеєри безперервної інтеграції та безперервного розгортання (CI/CD). Вся ідея автоматизованого розгортання покладається на єдине надійне джерело істини, яке запускає автоматизацію щоразу, коли виявляється нова, схвалена зміна. Якщо ви не розумієте Git, ви не зможете зрозуміти сучасну доставку програмного забезпечення.

## Розділ 1: Концепція контролю версій та архітектура Git

Перш ніж вводити будь-які команди, ви повинні зрозуміти, як Git сприймає ваші файли. Багато новачків мають проблеми з Git, тому що ставляться до нього як до простого інструменту резервного копіювання (як Dropbox або Google Drive). Git працює інакше. Він не просто безперервно синхронізує файли у фоновому режимі; він робить навмисні, незмінні знімки (snapshots) вашого проєкту в певні моменти часу, але тільки тоді, коли ви явно даєте йому таку команду.

Уявіть Git як відеогру з точками збереження вручну. Ви можете грати в гру, робити помилки, отримувати шкоду та заходити в глухі кути. Доки ви явно створюєте "точку збереження" перед небезпечною битвою з босом, ви завжди можете перезавантажити цей точний стан, якщо справи підуть кепсько. Git працює так само, але для ваших текстових файлів.

### Централізований проти розподіленого контролю версій

Історично системи контролю версій, такі як Subversion (SVN) або Team Foundation Server (TFS), були централізованими. Існував один головний сервер, який зберігав історію. Якщо ви хотіли переглянути історію або зробити коміт, ви повинні були бути підключені до інтернету, щоб зв'язатися з цим сервером. Якщо сервер виходив з ладу, ніхто не міг працювати.

Git — це **розподілена** система контролю версій. Ковили ви використовуєте Git, ви не просто завантажуєте останні файли; ви завантажуєте всю історію проєкту. Ваш локальний ноутбук стає повноцінним репозиторієм. Ви можете переглядати історію, порівнювати версії та робити коміти, перебуваючи повністю офлайн в літаку. Інтернет потрібен лише тоді, коли ви хочете синхронізувати свою історію з чиєюсь іншою.

### Три дерева Git

Git керує вашими файлами, переміщуючи їх через три окремі логічні зони, які часто називають "деревами". Розуміння цього конвеєра є ключем до діагностики майже будь-якої проблеми в Git.

```text
+---------------------+       +---------------------+       +---------------------+
|                     |       |                     |       |                     |
|  Робоча директорія  | ----> |   Зона очікування   | ----> |     Репозиторій     |
| (Ваші локальні файли) |     |  (Staging Area)     |       | (Збережена історія) |
|                     |       |                     |       |                     |
+---------------------+       +---------------------+       +---------------------+
           |                             |                             |
           |   1. Зміна файлів           |                             |
           |---------------------------->|                             |
           |                             |   2. Групування змін        |
           |                             |---------------------------->|
           |                             |                             |
           |                                                           |
           |<----------------------------------------------------------|
                               3. Відновлення старих версій
```

1. **Робоча директорія (Working Directory)**: Це ваш поточний робочий простір на комп'ютері. Вона містить реальні файли, які ви редагуєте, видаляєте або створюєте. Коли ви відкриваєте файл у Vim або VS Code, ви змінюєте Робочу директорію. Git бачить ці зміни, але ще не зберіг їх.
2. **Зона очікування (Staging Area або Index)**: Це вирішальний проміжний крок, унікальний для Git. Уявіть це як зону підготовки або сцену перед фотосесією. Перш ніж зробити фінальний знімок, ви обираєте, які саме змінені файли розмістити на сцені. Ви можете додати одні файли, залишивши інші позаду. Це дозволяє створювати дуже сфокусовані, логічні коміти, навіть якщо ви змінили п'ятдесят файлів одночасно.
3. **Репозиторій (Repository або Історія комітів)**: Це постійна база даних, де Git зберігає ваші знімки (які називаються комітами). Як тільки файли закомічені сюди, вони надійно зафіксовані в історії з ім'ям автора, міткою часу та описовим повідомленням. Коміт математично запечатаний; його неможливо таємно змінити, не змінивши його унікальний ідентифікатор.

### Зупиніться та подумайте
> **Зупиніться та подумайте**: Ви щойно завершили довгий сеанс налагодження. Ви виправили помилку з'єднання з базою даних у `db.py`, але поки шукали помилку, ви також додали тимчасові інструкції виводу (print) у файли `auth.py` та `api.py`, які ви не хочете зберігати назавжди. Як архітектура Трьох дерев Git дозволяє створити чисту історію в цьому сценарії, не втрачаючи тимчасовий налагоджувальний код у робочій директорії?
>
> *Перевірка передбачення: Архітектура Трьох дерев відокремлює ваші робочі файли від того, що зберігається. Ви можете використовувати Зону очікування (Staging Area), щоб вибірково додати лише `db.py` для наступного коміту. Тимчасові інструкції print в `auth.py` та `api.py` залишаються в безпеці у вашій Робочій директорії, щоб ви могли продовжувати ними користуватися або видалити пізніше, ніколи не забруднюючи постійну історію Репозиторію.*

## Розділ 2: Підготовка сцени: Встановлення та налаштування

Git за своєю суттю є інструментом командного рядка. Хоча існують графічні інтерфейси, вивчення термінальних команд є обов'язковим для platform-інженерії, оскільки ви часто будете використовувати Git на віддалених серверах без графічних можливостей. Більшість сучасних дистрибутивів Linux та macOS постачаються з попередньо встановленим Git, або його можна легко додати за допомогою стандартного менеджера пакетів, такого як `apt`, `yum` або `brew`.

Щоб перевірити встановлення, відкрийте термінал і перевірте версію:

```bash
git --version
```

Очікуваний вивід:
```text
git version 2.39.2
```

### Налаштування ідентифікації

Оскільки Git розроблений для співпраці потенційно тисяч розробників, він відмовляється створювати знімки, доки не дізнається точно, хто їх робить. Кожен коміт потребує імені автора та адреси електронної пошти. Це критично важливо для підзвітності — якщо зміна інфраструктури призведе до падіння продакшну, команді потрібно знати, до кого звернутися за роз'ясненням причин зміни.

Ви налаштовуєте це за допомогою команди `git config`. Прапор `--global` застосовує ці налаштування до всіх репозиторіїв на вашому поточному комп'ютері, записуючи їх у прихований файл у вашій домашній директорії (`~/.gitconfig`).

```bash
# Встановіть своє ім'я (використовуйте справжнє ім'я, воно з'явиться в історії)
git config --global user.name "Alex Chen"

# Встановіть свою електронну пошту
git config --global user.email "alex.chen@example.com"
```

Ви можете перевірити свою конфігурацію в будь-який час, попросивши Git вивести всі його поточні налаштування:

```bash
git config --list
```

### Ініціалізація репозиторію: git init

Щоб наказати Git почати відстеження проєкту, ви повинні ініціалізувати репозиторій у кореневій директорії вашого проєкту. Давайте створимо нову директорію для гіпотетичного Kubernetes-проєкту та ініціалізуємо Git у ній.

```bash
# Створіть нову директорію
mkdir k8s-webapp
cd k8s-webapp

# Скажіть Git почати відстеження цієї директорії
git init
```

Очікуваний вивід:
```text
Initialized empty Git repository in /home/alex/k8s-webapp/.git/
```

Що саме зробила команда `git init`? Вона не сканувала ваш жорсткий диск магічним чином. Вона просто створила приховану директорію з назвою `.git` всередині вашої папки `k8s-webapp`. Ця прихована директорія `.git` і є тим самим "Репозиторієм" з нашої моделі Трьох дерев. Саме тут Git зберігає всі внутрішні об'єкти бази даних, стиснений вміст файлів, граф історії комітів та локальну конфігурацію для цього конкретного проєкту.

Якщо ви видалите директорію `.git`, ви видалите всю історію версій проєкту, хоча ваші поточні робочі файли залишаться недоторканими на диску.

```bash
# Виведіть список усіх файлів, включаючи приховані
ls -la
```

Очікуваний вивід:
```text
total 12
drwxr-xr-x 3 alex alex 4096 Oct 12 10:00 .
drwxr-xr-x 5 alex alex 4096 Oct 12 09:59 ..
drwxr-xr-x 7 alex alex 4096 Oct 12 10:00 .git
```

Після успішної ініціалізації репозиторію та налаштування ідентифікації ви заклали необхідний фундамент. Далі ми розберемося, як наповнити цей репозиторій, свідомо переміщуючи файли через життєвий цикл Git для створення ваших перших збережених знімків.

## Розділ 3: Життєвий цикл знімка: Add, Commit та Status

Тепер, коли ми маємо активний репозиторій, давайте пройдемо через щоденний робочий процес відстеження файлів. Команда, яку ви будете використовувати частіше за будь-яку іншу — сотні разів на день — це `git status`. Вона діє як ваш компас, повідомляючи вам точно, де перебувають ваші файли в Трьох деревах.

### Крок 1: Зміна робочої директорії

Створимо наш перший файл — базову конфігурацію простору імен (namespace) Kubernetes.

```bash
cat << 'EOF' > namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: webapp-prod
EOF
```

Запустіть `git status`, щоб побачити, що Git думає про цей щойно створений файл:

```bash
git status
```

Вивід:
```text
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	namespace.yaml

nothing added to commit but untracked files present (use "git add" to track)
```

Git розпізнає, що файл існує в Робочій директорії, але позначає його як "Untracked" (невідстежуваний). Це означає, що Git ніколи раніше не бачив цього файлу. Він не стежить за змінами в ньому і не буде автоматично зберігати його. Git відстежує лише те, що ви явно наказуєте йому відстежувати.

### Крок 2: Переміщення до Зони очікування (git add)

Щоб сказати Git, що ми хочемо включити цей файл у наш найперший знімок, ми повинні перемістити його до Зони очікування (зони підготовки) за допомогою команди `git add`.

```bash
git add namespace.yaml
```

Знову запустіть `git status`, щоб побачити зміну стану:

```bash
git status
```

Вивід:
```text
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   namespace.yaml
```

Файл перемістився. Тепер він класифікується як "Changes to be committed" (зміни, що будуть закомічені). Він знаходиться в зоні очікування, готовий до фіксації.

### Крок 3: Створення знімка (git commit)

Щоб назавжди зберегти підготовлені зміни в Репозиторій, ми використовуємо `git commit`. Кожен коміт потребує повідомлення (commit message), яке пояснює, *чому* було зроблено зміну. Написання хороших повідомлень до комітів — це основна професійна навичка. Гарне повідомлення пояснює намір і контекст, а не просто те, які рядки змінилися.

```bash
git commit -m "feat: add production namespace definition"
```

Вивід:
```text
[main (root-commit) a1b2c3d] feat: add production namespace definition
 1 file changed, 4 insertions(+)
 create mode 100644 namespace.yaml
```

Давайте перевіримо `git status` ще раз, щоб побачити фінальний результат нашого робочого процесу:

```bash
git status
```

Вивід:
```text
On branch main
nothing to commit, working tree clean
```

Ваша Робоча директорія тепер описується як "clean" (чиста). Це означає, що кожен окремий файл на вашому диску точно відповідає останньому знімку, збереженому в базі даних `.git`. Очікуваних змін немає.

### Повчальна історія: Випадковий коміт секрету

Молодший розробник тестував додаток локально, для якого був потрібен ключ доступу AWS. Для зручності він прописав ключ прямо у файл `deployment.yaml`, просто щоб перевірити, чи запустяться поди. Все запрацювало. У захваті він виконав `git add .` (команду, яка без розбору додає всі змінені файли в усій директорії) і потім запустив `git commit -m "fix deployment"`.

Потім він відправив код у публічний репозиторій GitHub. Протягом 120 секунд автоматизовані боти безпеки, що сканують публічні репозиторії GitHub, знайшли відкритий ключ AWS. Боти миттєво запустили сотні величезних серверів для майнінгу криптовалюти в різних регіонах AWS, використовуючи скомпрометовані дані. Коли розробник прокинувся наступного ранку, компанія отримала рахунок від хмарного провайдера на 65 000 доларів.

**Урок**: Ніколи сліпо не використовуйте `git add .`, якщо ви не впевнені на сто відсотків, що саме ви змінили. Завжди запускайте `git status` та `git diff` перед додаванням до індексу, щоб переконатися, що ви випадково не комітите паролі, API-ключі, приватні SSH-ключі або тимчасові файли налагодження.

Тепер, коли ви знаєте, як безпечно створювати знімки, вам неминуче знадобиться переглядати минулі знімки або вивчати точні модифікації окремих файлів з часом. У наступному розділі ми заглибимося в команди, які дозволяють аналізувати вашу історію комітів і досліджувати точні відмінності у файлах.

## Розділ 4: Подорож у часі: Log та Diff

Коли ви зробили кілька комітів, вам потрібні надійні способи перегляду часової шкали історії та розуміння того, що саме змінилося між різними моментами часу.

Давайте внесемо ще одну зміну в наш проєкт. Ми оновимо файл простору імен, щоб додати мітку (label) — це звичайне завдання в Kubernetes для цілей організації.

```bash
cat << 'EOF' > namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: webapp-prod
  labels:
    environment: production
EOF
```

### Перегляд того, що змінилося: git diff

Перш ніж додавати файл до індексу або комітити його, ви завжди повинні перевіряти, які саме рядки ви змінили. Пам'ять може підвести; diff об'єктивний. Команда `git diff` порівнює вашу поточну Робочу директорію з останнім знімком, який ви зробили.

```bash
git diff
```

Вивід:
```text
diff --git a/namespace.yaml b/namespace.yaml
index e46b825..8394c41 100644
--- a/namespace.yaml
+++ b/namespace.yaml
@@ -2,3 +2,5 @@ apiVersion: v1
 kind: Namespace
 metadata:
   name: webapp-prod
+  labels:
+    environment: production
```

**Як розшифрувати вивід diff:**
- Заголовки `--- a/namespace.yaml` та `+++ b/namespace.yaml` показують дві версії файлів, що порівнюються (стара та нова).
- `@@ -2,3 +2,5 @@` — це заголовок фрагмента. Він надає системі контекст про те, де приблизно у файлі відбулися зміни.
- Рядки, що починаються з пробілу, — це незмінені рядки контексту. Git показує їх, щоб допомогти вам зорієнтуватися.
- Рядки, що починаються з `+` (зазвичай підсвічуються зеленим), — це нові доповнення.
- Рядки, що починаються з `-` (зазвичай підсвічуються червоним), — це видалення. Якщо ви змінили рядок, Git представляє це як видалення старого рядка та додавання нового.

### Зупиніться та подумайте: Читання Diff
> **Зупиніться та подумайте**: Уявіть, що ви запустили `git diff` для файлу розгортання (deployment) і побачили наступний вивід:
>
> ```text
> @@ -10,3 +10,3 @@
>  spec:
>    replicas: 3
> -  image: nginx:1.14
> +  image: nginx:1.24
> ```
>
> Перш ніж читати далі, що саме зробив інженер у цьому файлі? Будьте точними.
>
> *Перевірка передбачення: Інженер не додав абсолютно новий структурний елемент. Він змінив існуючий рядок. Він видалив рядок, що вказував версію контейнера Nginx 1.14, і замінив його рядком із версією 1.24. Це означає оновлення версії образу контейнера.*

Тепер, коли ми переконалися, що наші зміни правильні та не містять секретів, давайте додамо до індексу та закомітимо нашу мітку:

```bash
git add namespace.yaml
git commit -m "chore: add environment label to namespace"
```

### Перегляд історії: git log

Щоб побачити часову шкалу ваших знімків, використовуйте команду `git log`. Вона відкриває програму перегляду (зазвичай `less`), яка показує вашу історію у зворотному хронологічному порядку (найновіші спочатку).

```bash
git log
```

Вивід:
```text
commit 8f9e0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f (HEAD -> main)
Author: Alex Chen <alex.chen@example.com>
Date:   Wed Oct 12 10:45:12 2023 -0400

    chore: add environment label to namespace

commit a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8g9h
Author: Alex Chen <alex.chen@example.com>
Date:   Wed Oct 12 10:15:30 2023 -0400

    feat: add production namespace definition
```

Зверніть увагу на довгий рядок літер і цифр (наприклад, `8f9e0a1b2...`). Це хеш коміту (Commit Hash, зокрема SHA-1 хеш). Це математично згенерований унікальний ідентифікатор для цього конкретного знімка. Він охоплює вміст файлів, автора, дату та хеш попереднього коміту. Ви використовуєте цей хеш, якщо вам коли-небудь знадобиться повернути проєкт до цієї точної точки в часі.

Для більш компактного вигляду, що особливо корисно при дослідженні репозиторію з тисячами комітів за кілька років, використовуйте прапор `--oneline`:

```bash
git log --oneline
```

Вивід:
```text
8f9e0a1 chore: add environment label to namespace
a1b2c3d feat: add production namespace definition
```

Ви також можете використовувати `git log -p`, щоб побачити реальні зміни (diffs), внесені кожним окремим комітом в історії, що дозволяє побачити не тільки те, *що* коміт стався, але й те, *які саме рядки* він змінив.

Оволодіння здатністю орієнтуватися у вашій локальній часовій шкалі дає величезну впевненість при експериментах з інфраструктурним кодом. Однак сучасна інженерія — це командна робота; наступний розділ розповість про те, як безпечно ділитися вашою локальною історію з віддаленими серверами для безперешкодної співпраці з іншими.

## Розділ 5: Співпраця зі світом: Remotes, Push та Pull

До цього моменту все, що ми робили, існувало виключно на жорсткому диску вашого локального ноутбука. Якщо ваш комп'ютер вийде з ладу або якщо ви проллєте на нього чашку кави, вся історія вашого проєкту назавжди зникне. Крім того, ніхто інший у вашій команді не може бачити вашу роботу.

Щоб співпрацювати з іншими та створювати резервні копії своєї роботи, ви повинні підключити свій локальний репозиторій до віддаленого сервера, такого як GitHub, GitLab або Bitbucket.

### Що таке Remote?

"Remote" (віддалений репозиторій) — це просто ще один репозиторій Git, розміщений у мережі або інтернеті. Оскільки Git розподілений, кожен має повну самостійну копію історії. "Pushing" та "Pulling" — це явні дії, які ми робимо для синхронізації нашої локальної часової шкали з віддаленою.

```text
+-----------------------+                    +-----------------------+
|                       |                    |                       |
|  Локальний репозиторій |                    | Віддалений репозиторій|
|     (Ваш ноутбук)     |                    |    (GitHub/GitLab)    |
|                       |                    |                       |
|  коміт C (HEAD)       | ==== git push ===> |  коміт C              |
|  коміт B              |                    |  коміт B              |
|  коміт A              | <=== git pull ==== |  коміт A              |
|                       |                    |                       |
+-----------------------+                    +-----------------------+
```

### Підключення до віддаленого репозиторію

Коли ви створюєте порожній репозиторій на платформі на кшталт GitHub, платформа надає вам URL-адресу для підключення (зазвичай HTTPS або SSH). Ви кажете своїй локальній інсталяції Git підключитися до цієї URL-адреси за допомогою команди `git remote add`.

За галузевою угодою, основний віддалений сервер за замовчуванням майже завжди називається `origin`.

```bash
# Приклад команди (не запускайте, якщо у вас немає реальної URL-адреси репозиторію)
git remote add origin https://github.com/yourusername/k8s-webapp.git
```

Якщо ви приєднуєтеся до компанії і хочете завантажити існуючий проєкт, ви не використовуєте `git init` та `git remote add`. Замість цього ви використовуєте `git clone <url>`. Клонування автоматично ініціалізує локальний репозиторій, додає віддалений сервер як `origin` і завантажує всю історію та файли за один крок.

### Відправка змін (git push)

Щоб завантажити ваші локально створені коміти на віддалений сервер, ви їх "відправляєте" (push). Ви повинні вказати назву віддаленого репозиторію (зазвичай `origin`) та назву гілки, яку ви відправляєте (часто `main` або `master`).

```bash
# Відправте вашу гілку main до origin вперше
git push -u origin main
```

Прапор `-u` означає "upstream". Його потрібно використовувати лише при першій відправці нової гілки. Він створює постійний зв'язок відстеження між вашою локальною гілкою `main` та віддаленою гілкою `main`. Надалі ви зможете просто вводити `git push` без аргументів, і Git знатиме точно, куди відправляти дані.

### Отримання змін (git pull)

Якщо колега вносить зміни, комітить їх локально і відправляє на GitHub, ваш локальний репозиторій *не* оновиться автоматично. Git ніколи не змінить ваші локальні файли без вашого явного дозволу. Ви повинні самі звернутися до сервера, завантажити нові коміти та інтегрувати їх у свою локальну історію за допомогою `git pull`.

```bash
# Отримайте зміни з віддаленого репозиторію та злийте їх у вашу локальну гілку
git pull origin main
```

Насправді `git pull` — це макрос, який запускає дві окремі команди послідовно: `git fetch` (яка безпечно завантажує нові коміти з віддаленого репозиторію без зміни ваших робочих файлів) та `git merge` (яка намагається безперешкодно об'єднати завантажені зміни з вашою поточною робочою директорією).

### Зупиніться та подумайте: Синхронізація з віддаленим репозиторієм
> **Зупиніться та подумайте**: Ви провели ранок, працюючи офлайн, і зробили два локальні коміти. Тим часом ваш колега відправив три коміти в ту саму гілку на віддалений сервер. Що станеться, якщо ви сліпо запустіть `git push origin main`, коли знову підключитеся до інтернету?
>
> *Перевірка передбачення: Відправка буде відхилена віддаленим сервером. Git розпізнає, що віддалена гілка має коміти, яких немає у вашій локальній гілці, що запобігає ненавмисному перезапису роботи вашого колеги. Ви повинні спочатку запустити `git pull`, щоб отримати та інтегрувати їхні зміни у вашу локальну історію, перш ніж ви зможете успішно відправити свою об'єднану часову шкалу.*

## Розділ 6: Ігнорування шуму: .gitignore

У будь-якому реальному проєкті програмного забезпечення або інфраструктури є численні файли, які ви **ніколи** не захочете комітити в систему контролю версій. До них відносяться:
- Компільовані бінарні файли, виконувані файли або артефакти збірки (наприклад, `.exe`, `.jar`, директорії `/dist/`).
- Лог-файли, що генеруються вашим додатком під час тестування.
- Приховані файли операційної системи (наприклад, `.DS_Store`, що генерується macOS Finder).
- **Секрети, API-ключі, паролі до баз даних та локальні змінні оточення** (наприклад, файли `.env`).

Якщо ви випадково закомітите їх, ви збільшите розмір репозиторію або, що ще гірше, спричините серйозне порушення безпеки. Щоб сказати Git вдавати, що цих файлів взагалі не існує, ви створюєте звичайний текстовий файл з назвою `.gitignore` у кореневій папці вашого проєкту.

Давайте створимо `.gitignore`, адаптований для нашого хмарно-орієнтованого середовища.

```bash
cat << 'EOF' > .gitignore
# Ігнорувати файли, згенеровані операційною системою
.DS_Store
Thumbs.db

# Ігнорувати локальні файли секретів та облікових даних
.env
secret-keys.yaml
kubeconfig-local

# Ігнорувати файли стану terraform (якщо ми додамо Infrastructure as Code пізніше)
*.tfstate
*.tfstate.backup
.terraform/
EOF
```

Git читає цей файл зверху вниз. Будь-який файл, що відповідає шаблону, вказаному в `.gitignore`, ніколи не з'явиться в `git status` як невідстежуваний. Це унеможливлює випадкове додавання його до індексу за допомогою команд із символами підстановки, наприклад `git add .`.

### Зупиніться та подумайте: Зіставлення шаблонів
> **Зупиніться та подумайте**: Враховуючи файл `.gitignore` вище, який із наступних трьох щойно створених файлів все одно відображатиметься як "Untracked" при запуску `git status`? 1) `main.tfstate`, 2) `secret-keys.yaml`, 3) `secret-keys.txt`.
>
> *Перевірка передбачення: Тільки `secret-keys.txt` відображатиметься як невідстежуваний. `main.tfstate` ігнорується правилом `*.tfstate`, а `secret-keys.yaml` явно ігнорується за назвою. `secret-keys.txt` не відповідає жодному шаблону ігнорування, тому Git продовжить позначати його як невідстежуваний файл.*

### Зупиніться та подумайте: Пізнє ігнорування
> **Зупиніться та подумайте**: У вас є файл з назвою `database-creds.txt`, який ви створили минулого тижня. Ви закомітили його в Git кілька днів тому. Сьогодні ви усвідомили свою помилку і додали `database-creds.txt` у свій файл `.gitignore`. Ви змінюєте файл з обліковими даними і запускаєте `git status`. Чи проігнорує Git зміни?
>
> *Перевірка передбачення: Ні, Git не проігнорує зміни. Файл `.gitignore` запобігає додаванню до бази даних тільки **невідстежуваних** файлів. Як тільки файл почав відстежуватися (був закомічений), Git продовжуватиме відстежувати його незалежно від того, що написано в `.gitignore`. Ви повинні явно видалити його з відстеження за допомогою команди `git rm --cached database-creds.txt`, перш ніж правило ігнорування почне діяти.*

## Чи знали ви?

1. **Git був створений за два тижні.** У 2005 році спільнота розробників ядра Linux раптово втратила безкоштовну ліцензію на пропрієтарну систему контролю версій. Лінус Торвальдс, творець Linux, потребував негайної заміни. Незадоволений існуючими варіантами, він написав початкову версію Git всього за 14 днів, а масивна база коду ядра Linux перейшла на неї через два місяці.
2. **Назва — це самопринизлива образа.** Торвальдс, відомий своїм різким гумором, назвав систему "Git" (британський сленг для впертої, неприємної або некомпетентної людини). Він відомий своїм жартом на конференції: "Я егоїстичний виродок, і я називаю всі свої проєкти на свою честь. Спочатку Linux, тепер Git".
3. **Git не відстежує порожні директорії.** Через архітектуру бази даних, яка зіставляє шляхи безпосередньо з вмістом файлів, Git відстежує лише файли. Якщо ви створите порожню директорію і запустіть `git status`, Git повністю її проігнорує. Розробники обходять це обмеження, розміщуючи всередині директорії прихований порожній файл (часто за домовленістю з назвою `.gitkeep`), щоб змусити Git відстежувати існування папки.
4. **Колосальна стійкість до колізій.** Хеші, що ідентифікують ваші коміти (рядки SHA-1), є 40-символьними шістнадцятковими числами. Математична ймовірність того, що два різних знімки згенерують однаковий хеш (колізія хешів), астрономічно низька. Статистично набагато імовірніше, що у вас влучить блискавка, поки ви одночасно виграєте в лотерею, ніж ви зіткнетеся з випадковою колізією хешів Git у своєму репозиторії.

| **Випадковий коміт пароля/секрету** | Використання без розбору команди `git add .` без попередньої перевірки `git status`, випадкове додавання файлу `.env` до зони очікування. | Якщо коміт не відправлений на сервер: `git reset HEAD~1`, щоб скасувати коміт локально. Якщо відправлений — **секрет скомпрометовано**. Ви повинні негайно відкликати/змінити облікові дані в AWS/GCP. Не просто видаляйте їх у новому коміті; історія залишається назавжди. |
| **"fatal: refusing to merge unrelated histories"** | Ви ініціалізували репозиторій локально і створили окремий репозиторій на GitHub з файлом README за замовчуванням, а потім спробували отримати зміни (pull). Git вважає їх двома абсолютно різними, непов'язаними проєктами. | Запустіть `git pull origin main --allow-unrelated-histories`, щоб примусово вказати Git об'єднати дві окремі часові шкали в одну. |
| **"Updates were rejected because the remote contains work that you do not have locally"** | Колега відправив нові коміти на сервер GitHub, поки ви працювали офлайн. Git відмовляється дозволити вам відправку змін, щоб ви не перезаписали їхню роботу. | Спочатку запустіть `git pull`, щоб завантажити та інтегрувати їхні зміни у вашу локальну гілку. Виправте можливі конфлікти злиття, а потім запустіть `git push`. |
| **Порожні або безглузді повідомлення комітів** | Поспіх. Використання розпливчастих повідомлень на кшталт `git commit -m "update"` або `git commit -m "fixed stuff"`. | Використовуйте `git commit --amend -m "new better message"`, якщо ви ще не відправили зміни на сервер. Виробіть професійну звичку писати "Чому", а не просто "Що". |
| **Забули додати файли до індексу перед комітом** | Запуск `git commit -m "message"`, коли ваші зміни все ще знаходяться в Робочій директорії, повністю минаючи Зону очікування. | Запустіть `git status`, щоб побачити, що не додано до індексу. Виконайте `git add <файл>`, щоб перемістити зміни в індекс, а потім повторіть команду `git commit`. |
| **Коміт величезних бінарних файлів** | Випадкове додавання артефактів збірки, дампів пам'яті бази даних або великих відео в репозиторій. Git розроблений для відстеження тексту, а не великих бінарних файлів. | Видаліть файл із бази даних відстеження за допомогою `git rm --cached <файл>`, закомітьте видалення та негайно додайте тип файлу до вашого `.gitignore`. |

## Контрольні запитання

<details>
<summary>1. Ваша команда щойно розгорнула нову конфігурацію Kubernetes, і кластер негайно впав. Вам потрібно швидко дізнатися, хто вніс останню зміну і яким було їхнє повідомлення до коміту. Яку команду ви запустите?</summary>
Вам слід запустити `git log` або `git log --oneline`. Ця команда відображає історію комітів у зворотному хронологічному порядку, показуючи автора, мітку часу та повідомлення до коміту для кожного знімка. Аналізуючи цей вивід, ви зможете миттєво визначити, хто вніс останні зміни, і прочитати їхнє пояснення, чому ця зміна була необхідною. Така швидка видимість історії — саме те, чому контроль версій є критично важливим під час інцидентів на продуктивному середовищі.
</details>

<details>
<summary>2. Ви змінили файл `service.yaml` на своєму ноутбуці, щоб відкрити новий порт. Ви вводите `git commit -m "expose port 8080"`, але Git повертає повідомлення "nothing added to commit but untracked files present". Що ви забули зробити?</summary>
Ви забули перемістити файл із Робочої директорії до Зони очікування за допомогою команди `git add`. Архітектура Git вимагає від вас явно вибирати, які файли мають бути включені в наступний знімок. Оскільки ви оминули Зону очікування, Git побачив ваш змінений файл, але відмовився автоматично включати його в репозиторій. Ви повинні запустити `git add service.yaml`, перед тим як намагатися зробити коміт знову.
</details>

<details>
<summary>3. Ви збираєтеся додати до індексу `configMap.yaml`, але не пам'ятаєте, чи встановили ви пароль бази даних для тестування, чи залишили там пароль для продакшну. Яку команду слід запустити, щоб перевірити точні рядки, які ви змінили, перед додаванням до індексу?</summary>
Вам слід запустити `git diff`, щоб переглянути точні покрокові зміни. Ця команда порівнює вашу поточну Робочу директорію з останнім збереженим знімком у репозиторії. Вона відобразить видалені рядки червоним кольором, а додані — зеленим, що дозволить вам об'єктивно перевірити вміст файлу. Використання цієї команди запобігає випадковому потраплянню секретів продакшну в постійний граф історії.
</details>

<details>
<summary>4. Ви щойно приєдналися до нового проєкту. Ви клонуєте репозиторій, але коли запускаєте `git push`, термінал запитує ваше ім'я користувача та електронну пошту і каже запустити `git config`. Чому це необхідно?</summary>
Git вимагає ідентифікацію автора для кожного коміту, щоб забезпечити підзвітність і простежуваність у середовищах спільної розробки. Без налаштування ваших `user.name` та `user.email` Git відмовляється створювати знімки, оскільки не може приписати зміни конкретній людині. Ця ідентифікаційна інформація назавжди вшивається в криптографічний хеш коміту. У разі збою інфраструктури ця атрибуція дозволяє команді негайно дізнатися, до кого звернутися за контекстом щодо зміни.
</details>

<details>
<summary>5. Ви створили файл з назвою `aws-credentials.json` на своїй локальній машині для тестування скрипта. Ви ніколи не хочете, щоб цей файл потрапив у репозиторій компанії. Що саме ви повинні зробити, щоб він ігнорувався назавжди?</summary>
Ви повинні створити звичайний текстовий файл з назвою `.gitignore` у корені вашого репозиторію, якщо він ще не існує. Всередині цього файлу потрібно додати точну назву файлу `aws-credentials.json` з нового рядка. Git читає цей конфігураційний файл зверху вниз і автоматично відфільтровує всі відповідні файли зі свого радара відстеження. Це гарантує, що файл облікових даних ніколи випадково не буде доданий до індексу або закомічений, запобігаючи серйозному порушенню безпеки.
</details>

<details>
<summary>6. Молодший розробник намагається "скинути" зламаний проєкт, щоб почати спочатку. Він відкриває провідник, видаляє приховану папку `.git`, а потім вводить `git status` у своєму терміналі, очікуючи, що Git зрозуміє, що проєкт було скинуто, і покаже порожню історію. Що насправді відбувається і чому?</summary>
Коли розробник вводить `git status`, термінал поверне помилку "fatal: not a git repository", тому що він повністю знищив систему контролю версій для цієї папки. Прихована директорія `.git` — це не просто конфігураційний файл; це вся фактична база даних репозиторію, що містить всі коміти, гілки та історичні знімки. Видаливши її, він не скинув історію — він назавжди стер її локально, перетворивши свій проєкт назад на звичайну паку, яка не відстежується операційною системою. Робочі файли на диску залишаються недоторканими, але Git більше ними не керує.
</details>

<details>
<summary>7. Ви та колега одночасно змінюєте той самий файл `nginx-deployment.yaml`. Колега відправляє зміну, яка встановлює `replicas: 5`, поки ви перебуваєте офлайн. Ви локально змінюєте той же рядок на `replicas: 2` і намагаєтеся відправити зміни. Git відхиляє вашу відправку, тому ви запускаєте `git pull`. Що відбувається далі і як це вирішити?</summary>
Git спробує автоматично об'єднати зміни, але оскільки ви обоє змінили той же самий рядок, він зупиниться і оголосить конфлікт злиття (merge conflict). Він не може безпечно вгадати, чи має бути у розгортанні 5 реплік, чи 2, тому залишає рішення людині. Щоб вирішити конфлікт, ви повинні відкрити файл у своєму редакторі та знайти маркери конфлікту (які виглядають як `<<<<<<< HEAD` та `>>>>>>>`). Після ручного редагування файлу до бажаного стану та видалення маркерів, ви повинні використати `git add` та `git commit`, щоб завершити злиття перед відправкою.
</details>

## Практична вправа

У цій вправі ви створите локальний репозиторій з нуля, симулюєте стандартний інженерний процес, зробивши кілька логічних комітів, і простежите за отриманою історією. Ми будемо використовувати фіктивні файли конфігурації Kubernetes для симуляції реального процесу роботи з інфраструктурою.

- [ ] Директорія створена, перехід здійснено.
- [ ] Репозиторій Git ініціалізовано.

### Завдання 6: Бонусне завдання — Розширені правила ігнорування
Ви працюєте над новим додатком, який генерує численні лог-файли, що закінчуються на `.log`, у різних директоріях. Ви хочете, щоб Git ігнорував їх усі для економії місця, але ви повинні гарантувати, що один конкретний файл з назвою `audit-trail.log` у кореневій директорії завжди відстежується з міркувань відповідності стандартам.
Створіть файл `.gitignore`, який реалізує таку конфігурацію. Перевірте своє рішення, створивши тестові файли (наприклад, `app.log`, `database.log` та `audit-trail.log`), і перевіривши `git status`, щоб переконатися, що тільки `audit-trail.log` готовий до відстеження.
- [ ] Файл `.gitignore` створено з відповідними правилами підстановки та виключення.
- [ ] Створено тестові файли для перевірки правил.
- [ ] `git status` підтверджує, що тільки потрібний файл не ігнорується.

<details>
<summary>Рішення: Завдання 6</summary>

```bash
# Створіть файл .gitignore
cat << 'EOF' > .gitignore
*.log
!audit-trail.log
EOF

# Створіть тестові файли
touch app.log
touch database.log
touch audit-trail.log

# Перевірте статус
git status
```
Git покаже `.gitignore` та `audit-trail.log` як невідстежувані файли. Інші файли `.log` будуть успішно проігноровані.
</details>
