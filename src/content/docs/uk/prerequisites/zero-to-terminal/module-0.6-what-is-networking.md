## Чому цей модуль важливий

Kubernetes — це система для запуску застосунків на **кількох комп'ютерах**, які спілкуються між собою через **мережу**. Якщо ви не розумієте, що таке мережа, що означає IP-адреса чи що робить порт — Kubernetes здаватиметься магією, і не тією приємною.

Цей модуль дає вам ментальну модель. До кінця ви зрозумієте, як комп'ютери знаходять один одного, як вони спілкуються, і ви використаєте справжні мережеві команди з терміналу.

## Що таке мережа?

**Мережа** — це два або більше комп'ютерів, з'єднаних разом, щоб вони могли обмінюватись інформацією.

Ось і все. Ось уся концепція.

Ваш домашній Wi-Fi? Це мережа. Інтернет? Це мережа мереж — мільйони комп'ютерів, усі з'єднані. Коли ви відвідуєте google.com, ваш комп'ютер надсилає повідомлення через мережу до комп'ютера Google, який надсилає повідомлення назад.

> Аналогія з кухнею ресторану: Мережа — це як **система внутрішнього зв'язку** у великій мережі ресторанів. Кожна кухня (комп'ютер) має свій переговорний пристрій (мережеве з'єднання). Будь-яка кухня може зателефонувати будь-якій іншій, щоб поділитися рецептами, запросити постачання або координувати замовлення. Система зв'язку, що з'єднує всі кухні? Це мережа.

### Локальна мережа та інтернет

| Тип | Що це | Приклад |
|-----|-------|---------|
| **Локальна мережа (LAN)** | Комп'ютери поблизу, з'єднані напряму | Ваш ноутбук і принтер у домашньому Wi-Fi |
| **Інтернет (WAN)** | Комп'ютери де завгодно у світі, з'єднані через інфраструктуру | Ваш ноутбук спілкується з сервером у Токіо |

У Kubernetes ви працюватимете з обома: локальними мережами всередині кластера (де контейнери спілкуються між собою) та інтернет-з'єднаннями із зовнішнього світу до ваших застосунків.

## IP-адреси: Як комп'ютери знаходять один одного

Кожен комп'ютер у мережі має **IP-адресу** — унікальний номер, що його ідентифікує, як поштова адреса ідентифікує будівлю.

```
192.168.1.42
```

Це IP-адреса. Чотири числа, розділені крапками, кожне від 0 до 255.

### Аналогія з поштовою адресою

| Концепція | Мережа | Реальний світ |
|-----------|--------|---------------|
| IP-адреса | `192.168.1.42` | вул. Хрещатик, 123 |
| Що робить | Ідентифікує один конкретний комп'ютер | Ідентифікує одну конкретну будівлю |
| Хто призначає | Ваш роутер (для локальної) або ваш провайдер (для інтернету) | Міська адміністрація |

### Типи IP-адрес

**Приватні IP-адреси** — використовуються всередині локальної мережі:

```
192.168.x.x    (найпоширеніші для домашніх мереж)
10.x.x.x       (поширені в офісах і центрах обробки даних)
172.16-31.x.x  (менш поширені, але дійсні)
```

Вони працюють лише у вашій локальній мережі. Це як номери квартир — вони мають сенс усередині будинку, але хтось ззовні не знатиме, який саме будинок ви маєте на увазі.

**Публічні IP-адреси** — ваша адреса в інтернеті:

Ваш роутер має публічну IP-адресу, яку бачить решта інтернету. Усі пристрої за вашим роутером ділять одну публічну адресу.

### Як дізнатися свою IP-адресу

**На macOS:**

```bash
$ ipconfig getifaddr en0
192.168.1.42
```

**На Linux:**

```bash
$ hostname -I
192.168.1.42
```

**На будь-якій системі (ваша публічна IP):**

```bash
$ curl -s ifconfig.me
203.0.113.55
```

> Не хвилюйтесь, якщо вивід відрізняється від прикладів. Числа будуть різними — це очікувано, бо кожен комп'ютер отримує свою унікальну адресу.

## Порти: Багато дверей за однією адресою

Отже, кожен комп'ютер має IP-адресу. Але комп'ютер одночасно запускає багато програм — веббраузер, поштовий клієнт, месенджер. Як комп'ютер знає, яка програма має отримати вхідне повідомлення?

Саме для цього існують **порти**.

**Порт** — це число (від 0 до 65535), що ідентифікує конкретну програму або сервіс на комп'ютері.

### Аналогія з багатоповерхівкою

Якщо IP-адреса — це **поштова адреса** (будівля), то порт — це **номер квартири** (яка саме квартира в будинку).

```
IP-адреса:  192.168.1.42     = вул. Хрещатик, 123
Порт:       80               = Квартира 80

Повна адреса: 192.168.1.42:80 = вул. Хрещатик, 123, кв. 80
```

Двокрапка `:` розділяє IP-адресу та номер порту.

### Поширені порти

| Порт | Сервіс | Що він робить |
|------|--------|---------------|
| 80 | HTTP | Звичайний вебтрафік (коли ви відвідуєте вебсайт) |
| 443 | HTTPS | Захищений вебтрафік (замок у вашому браузері) |
| 22 | SSH | Захищений віддалений доступ до терміналу |
| 53 | DNS | Пошук доменних імен |
| 6443 | Kubernetes API | Як `kubectl` спілкується з вашим кластером |

> У Kubernetes ви зустрічатимете порти скрізь. Кожен сервіс має порт. Кожен контейнер відкриває порти. Розуміння цієї концепції зараз заощадить вам величезну плутанину пізніше.

## DNS: Телефонна книга інтернету

Коли ви вводите `google.com` у браузері, ваш комп'ютер насправді не знає, де `google.com`. Комп'ютери розуміють лише IP-адреси (числа). Як же це працює?

**DNS** — **Domain Name System** (система доменних імен) — перекладає зрозумілі людям імена в IP-адреси.

```
Ви вводите:     google.com
DNS повертає:   142.250.80.46
Ваш комп'ютер з'єднується з: 142.250.80.46
```

### Аналогія з телефонною книгою

| Без DNS | З DNS |
|---------|-------|
| «Зателефонуй на 142.250.80.46» | «Зателефонуй у Google» |
| Потрібно запам'ятовувати номери | Достатньо пам'ятати імена |
| Як у старі часи запам'ятовування номерів телефонів | Як список контактів у вашому телефоні |

DNS — причина, чому вам не потрібно вводити `142.250.80.46` щоразу, коли хочете щось пошукати. Ви вводите `google.com`, і DNS шукає номер за вас.

### Як працює DNS (спрощено)

```
1. Ви вводите: google.com
2. Ваш комп'ютер питає роутер: "Яка IP для google.com?"
3. Роутер питає DNS-сервер: "Яка IP для google.com?"
4. DNS-сервер відповідає: "142.250.80.46"
5. Ваш комп'ютер з'єднується з 142.250.80.46
```

Все це відбувається за мілісекунди. Ви цього навіть не помічаєте.

## Збираємо все разом: Надсилання листа

Використаймо одну велику аналогію, щоб з'єднати все:

**Відвідування вебсайту — це як надсилання листа.**

```
1. Ви хочете надіслати листа до "Google HQ"
   (Ви вводите google.com)

2. Ви шукаєте адресу в телефонній книзі
   (DNS перекладає google.com → 142.250.80.46)

3. Ви пишете поштову адресу на конверті
   (IP-адреса: 142.250.80.46)

4. Ви пишете номер квартири/відділу
   (Порт: 443 для HTTPS)

5. Ви кидаєте листа в поштову скриньку
   (Ваш комп'ютер надсилає запит через мережу)

6. Google отримує й надсилає відповідь
   (Вебсторінка повертається до вашого браузера)
```

Ось так просто — і ви розумієте основи мереж.

## Ваші перші мережеві команди

Забруднімо руки. Відкрийте термінал і спробуйте ці команди.

### `ping` — «Ти тут?»

`ping` надсилає крихітне повідомлення іншому комп'ютеру й чекає на відповідь. Це як постукати у двері, щоб перевірити, чи хтось вдома.

```bash
$ ping -c 4 google.com
PING google.com (142.250.80.46): 56 data bytes
64 bytes from 142.250.80.46: icmp_seq=0 ttl=118 time=11.4 ms
64 bytes from 142.250.80.46: icmp_seq=1 ttl=118 time=10.8 ms
64 bytes from 142.250.80.46: icmp_seq=2 ttl=118 time=11.2 ms
64 bytes from 142.250.80.46: icmp_seq=3 ttl=118 time=10.9 ms
```

- `-c 4` означає «надіслати 4 пінги й зупинитися». Без цього ping працюватиме нескінченно (на Linux/macOS), поки ви не натиснете **Ctrl+C**.
- `time=11.4 ms` означає, що повний шлях туди й назад зайняв 11,4 мілісекунди. Ось як швидко світло (ну, електричні сигнали) може дістатися до сервера Google й повернутися.

> Якщо ви бачите `Request timeout`, сервер або не існує, або блокує пінги, або є проблеми з мережею.

### `curl` — «Дай мені ту вебсторінку»

`curl` отримує вміст за URL-адресою. Уявіть це як веббраузер у терміналі — але він показує сирий текст замість гарно відрендереної сторінки.

```bash
$ curl -s example.com
```

Ви побачите сирий HTML:

```html
<!doctype html>
<html>
<head>
    <title>Example Domain</title>
...
```

Прапорець `-s` означає «silent» (тихий режим) — ховає індикатор прогресу, щоб ви бачили лише вміст.

**Отримання лише заголовків** (метадані відповіді):

```bash
$ curl -I example.com
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
...
```

`200 OK` означає, що сервер отримав ваш запит і все добре. Ви дізнаєтесь більше про HTTP-коди стану пізніше, але найважливіші:

| Код | Значення | Аналогія |
|-----|----------|----------|
| 200 | OK — ось те, що ви просили | «Замовлення готове! Ось ваша страва.» |
| 404 | Не знайдено — такої сторінки не існує | «Вибачте, такої страви немає в меню.» |
| 500 | Помилка сервера — щось зламалося на їхньому боці | «Кухня горить.» |

### `nslookup` або `dig` — «Знайди адресу»

Ці команди вручну просять DNS перекласти ім'я в IP-адресу:

```bash
$ nslookup google.com
Server:    192.168.1.1
Address:   192.168.1.1#53

Non-authoritative answer:
Name:   google.com
Address: 142.250.80.46
```

Це підтверджує, що `google.com` перекладається в `142.250.80.46` (ваш результат може відрізнятися — Google має багато серверів по всьому світу).

## Чи знали ви?

> 1. **Інтернет тримається на фізичних кабелях під океаном.** Понад 550 підводних кабелів пролягають по дну океану, передаючи 99% міжконтинентальних даних. Коли ви відвідуєте вебсайт, розміщений на іншому континенті, ваш запит буквально подорожує кабелем по морському дну. Ці кабелі приблизно завтовшки з садовий шланг і простягаються на тисячі кілометрів.
>
> 2. **Порт 80 для HTTP був обраний доволі довільно.** Тім Бернерс-Лі, винахідник Всесвітньої мережі, обрав порт 80 у 1991 році просто тому, що він не був зайнятий іншим сервісом. Порт 443 для HTTPS з'явився пізніше, коли додали шифрування.
>
> 3. **DNS винайшли, бо запам'ятовувати IP-адреси було надто складно.** До появи DNS (1983 рік) існував буквально один текстовий файл під назвою `hosts.txt`, що містив список кожного комп'ютера в інтернеті та його адресу. Хтось підтримував його вручну. Коли інтернет розрісся, це стало неможливим, тому Пол Мокапетріс винайшов DNS для автоматизації процесу.

## Типові помилки

| Помилка | Що трапляється | Виправлення |
|---------|---------------|-------------|
| Забули `-c` з ping | Ping працює нескінченно | Натисніть **Ctrl+C**, щоб зупинити, або завжди використовуйте `ping -c 4` |
| Пінгуєте сайт, що блокує ping | `Request timeout` — виглядає, ніби сайт не працює | Сайт може бути в порядку — багато серверів блокують ping. Використовуйте `curl` натомість |
| Плутаєте IP-адресу й порт | З'єднуєтесь не з тим | IP = який комп'ютер. Порт = який сервіс на цьому комп'ютері. `192.168.1.42:80` |
| Думаєте, що локальна IP — це публічна | Плутанина при спілкуванні з іншими | `192.168.x.x` — лише локальна. Використовуйте `curl ifconfig.me` для публічної IP |
| Забуваєте `http://` з curl | Іноді curl вгадує неправильно або видає помилку | Будьте точними: `curl http://example.com` або `curl https://example.com` |
| Думаєте, що збій DNS означає, що сайт не працює | Не можете дістатися до сайту | Сайт може бути в порядку, але DNS не може визначити ім'я. Спробуйте використати IP напряму |

## Тест

**Запитання 1**: Що таке IP-адреса?

<details>
<summary>Показати відповідь</summary>

Унікальне число, що ідентифікує комп'ютер у мережі, як поштова адреса ідентифікує будівлю. Приклад: `192.168.1.42`

</details>

**Запитання 2**: Якщо IP-адреса — це поштова адреса, то що таке порт?

<details>
<summary>Показати відповідь</summary>

Порт — це **номер квартири** (або номер дверей). Він ідентифікує конкретний сервіс або програму на цьому комп'ютері. Наприклад, порт 80 — для вебтрафіку, порт 22 — для SSH.

</details>

**Запитання 3**: Що робить DNS?

<details>
<summary>Показати відповідь</summary>

DNS перекладає зрозумілі людям доменні імена (як `google.com`) в IP-адреси (як `142.250.80.46`), які комп'ютери використовують для пошуку один одного. Це телефонна книга інтернету.

</details>

**Запитання 4**: Яку команду ви використаєте, щоб перевірити, чи доступний сервер?

<details>
<summary>Показати відповідь</summary>

```bash
$ ping -c 4 servername.com
```

`ping` надсилає невелике повідомлення й чекає на відповідь. Якщо ви отримуєте відповіді — сервер доступний. Використовуйте `-c 4`, щоб обмежити до 4 спроб.

</details>

**Запитання 5**: Ви бачите `curl: (6) Could not resolve host: fakename.xyz`. Що це означає?

<details>
<summary>Показати відповідь</summary>

DNS не зміг перекласти `fakename.xyz` в IP-адресу. Це означає, що доменне ім'я або не існує, або написано з помилкою, або є проблема з DNS у вашій мережі.

</details>

## Практична вправа: Дослідження мережі

### Мета

Використати мережеві команди для дослідження з'єднань, пошуку адрес та отримання вебвмісту.

### Кроки

1. **Пропінгуйте вебсайт:**

```bash
$ ping -c 4 google.com
```

Зверніть увагу на IP-адресу, в яку він визначається, та час відповіді.

2. **Пропінгуйте інший сайт і порівняйте:**

```bash
$ ping -c 4 cloudflare.com
```

Час відповіді швидший чи повільніший? Різниця залежить від того, наскільки далеко їхні сервери від вас.

3. **Знайдіть свою локальну IP-адресу:**

На macOS:
```bash
$ ipconfig getifaddr en0
```

На Linux:
```bash
$ hostname -I
```

Запишіть свою локальну IP. Вона має починатися з `192.168.`, `10.` або `172.`.

4. **Знайдіть свою публічну IP-адресу:**

```bash
$ curl -s ifconfig.me
```

Це IP-адреса, яку бачить решта інтернету. Вона відрізнятиметься від локальної.

5. **Знайдіть доменне ім'я:**

```bash
$ nslookup github.com
```

Знайдіть IP-адресу для `github.com` у виводі.

6. **Отримайте вебсторінку:**

```bash
$ curl -s example.com
```

Ви маєте побачити сирий HTML сторінки example.com.

7. **Перевірте заголовки відповіді:**

```bash
$ curl -I example.com
```

Шукайте `HTTP/1.1 200 OK` — це означає успіх!

8. **Спробуйте сторінку, яка не існує:**

```bash
$ curl -I example.com/this-page-does-not-exist
```

Шукайте `404` у відповіді — це означає «не знайдено».

9. **Збережіть вебсторінку у файл** (поєднання мережевих та файлових навичок!):

```bash
$ curl -s example.com > ~/kubedojo-practice/my-first-webpage.html
$ cat ~/kubedojo-practice/my-first-webpage.html
```

Ви щойно завантажили вебсторінку й зберегли її як файл!
