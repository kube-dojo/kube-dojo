## Чому цей модуль важливий

Kubernetes створений для розподілених контейнеризованих застосунків — як правило, мікросервісів. Розуміння еволюції від монолітів до мікросервісів допомагає:

1. Знати, чому існують певні можливості Kubernetes
2. Приймати кращі архітектурні рішення
3. Розуміти, коли мікросервіси доречні (а коли — ні)
4. Говорити мовою сучасної архітектури програмного забезпечення

## Моноліт

### Що таке моноліт?

Монолітний застосунок — це єдина одиниця розгортання, що містить усю функціональність:

```
┌─────────────────────────────────────────────────────────────┐
│              MONOLITHIC APPLICATION                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                 MyApp.jar (or .exe)                 │   │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │   │
│  │  │  User   │ │ Product │ │  Order  │ │ Payment │   │   │
│  │  │ Module  │ │ Module  │ │ Module  │ │ Module  │   │   │
│  │  └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘   │   │
│  │       │           │           │           │         │   │
│  │       └───────────┴───────────┴───────────┘         │   │
│  │                       │                             │   │
│  │              ┌────────┴────────┐                    │   │
│  │              │    Database     │                    │   │
│  │              │   (shared)      │                    │   │
│  │              └─────────────────┘                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Deploy: One unit                                          │
│  Scale: All or nothing                                     │
│  Database: Shared by all modules                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Переваги моноліту

```
✅ Простий для початкової розробки
✅ Легко тестувати (один застосунок)
✅ Просте розгортання (один артефакт)
✅ Легко зневаджувати (один процес)
✅ Без мережевої затримки між компонентами
✅ ACID-транзакції просто реалізувати
```

### Виклики моноліту (у масштабі)

```
❌ Зміни потребують повного перерозгортання
❌ Довгі цикли збірки/тестування
❌ Масштабування означає масштабування всього
❌ Вибір технологій впливає на весь застосунок
❌ Один баг може покласти все
❌ Координація команди стає складною
❌ Кодова база стає некерованою
```

## Мікросервісний підхід

### Що таке мікросервіси?

Мікросервіси декомпозують застосунок на малі, незалежні сервіси:

```
┌─────────────────────────────────────────────────────────────┐
│              MICROSERVICES ARCHITECTURE                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌───────────┐   ┌───────────┐   ┌───────────┐            │
│  │   User    │   │  Product  │   │   Order   │            │
│  │  Service  │   │  Service  │   │  Service  │            │
│  │  ┌─────┐  │   │  ┌─────┐  │   │  ┌─────┐  │            │
│  │  │ DB  │  │   │  │ DB  │  │   │  │ DB  │  │            │
│  │  └─────┘  │   │  └─────┘  │   │  └─────┘  │            │
│  └─────┬─────┘   └─────┬─────┘   └─────┬─────┘            │
│        │               │               │                    │
│        └───────────────┼───────────────┘                    │
│                        │                                    │
│                ┌───────┴───────┐                           │
│                │  API Gateway  │                           │
│                └───────────────┘                           │
│                        │                                    │
│                    Clients                                  │
│                                                             │
│  Deploy: Each service independently                        │
│  Scale: Per service based on need                          │
│  Database: Each service owns its data                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Переваги мікросервісів

```
✅ Незалежне розгортання
✅ Масштабування окремих сервісів
✅ Різноманіття технологій (правильний інструмент для завдання)
✅ Ізоляція збоїв (один сервіс упав ≠ усі впали)
✅ Автономія команд
✅ Менші, зосереджені кодові бази
```

### Виклики мікросервісів

```
❌ Складність мережі
❌ Збої розподілених систем
❌ Проблеми з узгодженістю даних
❌ Тестування складніше
❌ Операційна складність
❌ Зневадження між сервісами
❌ Потреба в надійній інфраструктурі
```

## Коли що обирати

### Моноліт часто кращий, коли:

```
✅ Мала команда (< 10 розробників)
✅ Новий продукт / стадія стартапу
✅ Простий домен
✅ Стислі терміни
✅ Невідомі вимоги
✅ Єдине середовище розгортання
```

### Мікросервіси мають сенс, коли:

```
✅ Великі, кілька команд
✅ Потреба в незалежному масштабуванні
✅ Різні технологічні вимоги для різних компонентів
✅ Високі вимоги до доступності
✅ Потрібні часті релізи
✅ Чіткі межі домену
```

### Реальність

```
┌─────────────────────────────────────────────────────────────┐
│              THE TYPICAL JOURNEY                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Stage 1: Start with Monolith                              │
│  ├── Fast development                                      │
│  ├── Simple operations                                     │
│  └── Learn the domain                                      │
│                                                             │
│  Stage 2: Grow (Still Monolith)                            │
│  ├── Team grows                                            │
│  ├── Features accumulate                                   │
│  └── Pain points emerge                                    │
│                                                             │
│  Stage 3: Strategic Decomposition                          │
│  ├── Extract services where it hurts                       │
│  ├── Clear boundaries                                      │
│  └── Incremental migration                                 │
│                                                             │
│  DON'T: Start with microservices for a new product        │
│  DON'T: Decompose without clear boundaries                │
│  DON'T: Microservices for small teams                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Як Kubernetes забезпечує роботу мікросервісів

Kubernetes розв'язує операційні виклики мікросервісів:

| Виклик | Рішення Kubernetes |
|--------|---------------------|
| Виявлення сервісів | Сервіси, DNS |
| Балансування навантаження | Сервіси, Ingress |
| Масштабування | Деплойменти, HPA |
| Конфігурація | ConfigMaps, Secrets |
| Моніторинг стану | Проби (Probes) |
| Плавні оновлення | Деплойменти |
| Стійкість до збоїв | ReplicaSets, самовідновлення |
| Керування ресурсами | Requests/Limits |

```
Without Kubernetes:
├── Manual service registration
├── Custom load balancer configuration
├── Script-based deployments
├── Hope things restart when they crash
└── Configuration files everywhere

With Kubernetes:
├── Automatic service discovery
├── Built-in load balancing
├── Declarative deployments
├── Self-healing by default
└── Centralized configuration
```

## Патерни мікросервісів

### API Gateway

```
                    ┌───────────────┐
                    │  API Gateway  │
                    └───────┬───────┘
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
      ┌─────────┐     ┌─────────┐     ┌─────────┐
      │ Service │     │ Service │     │ Service │
      │    A    │     │    B    │     │    C    │
      └─────────┘     └─────────┘     └─────────┘

Purpose:
- Single entry point
- Authentication/Authorization
- Rate limiting
- Request routing

In K8s: Ingress Controller or dedicated gateway
```

### Service Mesh

```
      ┌─────────────────────────────────────────┐
      │             Service Mesh                │
      │  ┌────────┐        ┌────────┐          │
      │  │Svc A   │◄──────►│Svc B   │          │
      │  │┌─────┐ │        │┌─────┐ │          │
      │  ││Proxy│ │        ││Proxy│ │          │
      │  │└─────┘ │        │└─────┘ │          │
      │  └────────┘        └────────┘          │
      └─────────────────────────────────────────┘

Purpose:
- Traffic management
- Security (mTLS)
- Observability

Examples: Istio, Linkerd
```

### Патерн Sidecar

```
      ┌─────────────────────────────────────────┐
      │                 Pod                     │
      │  ┌─────────────┐  ┌─────────────┐      │
      │  │   Main      │  │   Sidecar   │      │
      │  │ Application │◄►│   (Proxy)   │      │
      │  └─────────────┘  └─────────────┘      │
      └─────────────────────────────────────────┘

Purpose:
- Add functionality without changing app
- Logging, monitoring, security

K8s: Multi-container pods
```

## Візуалізація: патерни комунікації

```
┌─────────────────────────────────────────────────────────────┐
│              MICROSERVICES COMMUNICATION                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  SYNCHRONOUS (Request/Response)                            │
│  ┌─────────┐  HTTP/gRPC   ┌─────────┐                     │
│  │Service A│─────────────►│Service B│                     │
│  └─────────┘◄─────────────└─────────┘                     │
│              Response                                       │
│  Use: When you need immediate response                     │
│  Risk: Tight coupling, cascading failures                  │

│  ASYNCHRONOUS (Events/Messages)                            │
│  ┌─────────┐    Event    ┌─────────┐   ┌─────────┐       │

│  └─────────┘             └─────────┘   └─────────┘       │
│  Use: Decoupled systems, eventual consistency             │
│  Benefit: Loose coupling, resilience                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Чи знали ви?

- **Amazon був піонером мікросервісів.** Їхній наказ 2002 року вимагав, щоб усі команди спілкувалися через API. Це зрештою стало AWS.

- **Netflix має понад 1000 мікросервісів.** Вони також створили багато інструментів, які тепер є частиною cloud native екосистеми (Zuul, Eureka, Hystrix).

- **«Правило двох піц»** каже, що команди мають бути досить малими, щоб їх можна було нагодувати двома піцами. Це узгоджується з принципом володіння мікросервісами.

- **Моноліти теж масштабуються.** Shopify обробляє величезний трафік за допомогою монолітного застосунку на Rails. Вибір архітектури залежить від контексту.

## Поширені хибні уявлення

| Хибне уявлення | Реальність |
|----------------|------------|
| «Мікросервіси завжди кращі» | Вони додають складності. Часто надмірні для малих команд/продуктів. |
| «Моноліти не масштабуються» | Масштабуються! Shopify, Stack Overflow це доводять. Інші компроміси. |
| «Мікросервіси = використання Kubernetes» | На K8s можна запускати й моноліти. K8s — це інфраструктура, не архітектура. |
| «Кожному мікросервісу потрібна власна база даних» | Поширений патерн, але не обов'язковий. Спільні бази даних іноді прийнятні. |

## Тест

1. **Яка основна перевага мікросервісів над монолітами?**
   <details>
   <summary>Відповідь</summary>
   Незалежне розгортання та масштабування. Кожний сервіс може розроблятися, розгортатися та масштабуватися незалежно окремими командами. Це забезпечує швидші цикли релізів і краще використання ресурсів.
   </details>

2. **Коли моноліт був би кращим вибором, ніж мікросервіси?**
   <details>
   <summary>Відповідь</summary>
   Малі команди, нові продукти, прості домени, невідомі вимоги, стислі терміни. Мікросервіси додають операційну складність, яка не виправдана без масштабу.
   </details>

3. **Як Kubernetes допомагає з мікросервісами?**
   <details>
   <summary>Відповідь</summary>
   K8s забезпечує виявлення сервісів (DNS), балансування навантаження, масштабування, керування конфігурацією, моніторинг стану, плавні оновлення та самовідновлення — усі ті виклики, що стають критичними при великій кількості сервісів.
   </details>

4. **Що таке service mesh?**
   <details>
   <summary>Відповідь</summary>
   Інфраструктурний рівень для комунікації між сервісами. Забезпечує керування трафіком, безпеку (mTLS) та спостережуваність. Реалізується через sidecar-проксі. Приклади: Istio, Linkerd.
   </details>

## Вправа на роздуми

Цей модуль охоплює архітектурні концепції, для яких немає вправ у командному рядку. Натомість поміркуйте над цими запитаннями:

**1. Подумайте про застосунки, які ви використовували або створювали:**
- Були вони монолітами чи мікросервісами?
- Які ознаки на це вказували? (Частота розгортання, структура команди, патерни масштабування)

**2. Уявіть гіпотетичний інтернет-магазин:**
- Які сервіси можна було б виділити? (Користувачі, Продукти, Замовлення, Платежі, Інвентар)
- Які з них потребували б незалежного масштабування? (Пошук продуктів? Обробка платежів?)
- Якими даними володів би кожний сервіс?

**3. Оцініть цей сценарій:**
> Стартап із 3 осіб створює новий продукт. Вони розглядають мікросервіси, «щоб бути сучасними».
- Чи це гарна ідея? (Ймовірно, ні)
- Що б ви порекомендували? (Спочатку моноліт, декомпозиція коли з'являться больові точки)
- Чому?

**4. Дослідіть одну компанію:**
- Пошукайте, як Netflix, Amazon або Spotify підходять до мікросервісів
- З якими викликами вони зіткнулися?
- Чи підійшов би їхній підхід для меншої компанії?

Ці запитання готують вас до прийняття архітектурних рішень у кар'єрі, а не лише до складання іспитів.

## Підсумок

**Моноліти**:
- Єдина одиниця розгортання
- Простіші для розробки й експлуатації на початку
- Масштабуються цілком
- Виклики з'являються зі зростанням

**Мікросервіси**:
- Незалежні сервіси
- Масштабування та розгортання окремо
- Гнучкість у виборі технологій
- Операційна складність

**Ключові висновки**:
- Починайте просто (часто з моноліту)
- Декомпозуйте, коли з'являються больові точки
- Kubernetes забезпечує роботу патернів мікросервісів
- Архітектура має відповідати потребам команди та продукту
