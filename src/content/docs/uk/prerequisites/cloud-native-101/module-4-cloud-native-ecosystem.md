## Чому цей модуль важливий

Kubernetes не існує ізольовано. Він — центр величезної екосистеми проєктів, інструментів і практик. Розуміння цієї екосистеми допомагає:

1. Знати, які інструменти існують для різних завдань
2. Розуміти описи вакансій та обговорення в команді
3. Планувати свій навчальний шлях
4. Не винаходити велосипед

Ви не вивчатимете ці інструменти тут — лише дізнаєтеся, що вони існують і для чого призначені.

## Категорії інструментів

### 1. Оркестрація (ядро)

| Інструмент | Що він робить |
|------------|---------------|
| **Kubernetes** | Оркестрація контейнерів (центр усього) |
| **Helm** | Пакетний менеджер для K8s (як apt/yum для K8s) |
| **Kustomize** | Конфігурація K8s без шаблонів |

### 2. Середовище виконання контейнерів

| Інструмент | Що він робить |
|------------|---------------|
| **containerd** | Галузевий стандарт середовища виконання контейнерів |
| **CRI-O** | Легке середовище виконання для K8s |

### 3. Мережа

| Інструмент | Що він робить |
|------------|---------------|
| **Cilium** | CNI з мережею та безпекою на базі eBPF |
| **Calico** | Популярний CNI для мережевих політик |
| **Flannel** | Проста оверлейна мережа |
| **Istio** | Service mesh (керування трафіком, безпека) |
| **Linkerd** | Легкий service mesh |
| **Envoy** | Сервісний проксі (основа багатьох service mesh) |

### 4. Спостережуваність <!-- VERIFY: спостережуваність (= observability) -->

| Інструмент | Що він робить |
|------------|---------------|
| **Prometheus** | Збір метрик та сповіщення |
| **Grafana** | Візуалізація та інформаційні панелі |
| **Jaeger** | Розподілене трасування |
| **Fluentd/Fluent Bit** | Збір і пересилання логів |
| **Loki** | Агрегація логів (у стилі Prometheus) |
| **OpenTelemetry** | Уніфікований фреймворк спостережуваності |

### 5. CI/CD та GitOps

| Інструмент | Що він робить |
|------------|---------------|
| **ArgoCD** | GitOps-безперервна доставка |
| **Flux** | GitOps-інструментарій |
| **Tekton** | CI/CD-конвеєри, нативні для K8s |

### 6. Безпека

| Інструмент | Що він робить |
|------------|---------------|
| **Falco** | Моніторинг безпеки під час виконання |
| **Trivy** | Сканування вразливостей контейнерів |
| **OPA/Gatekeeper** | Застосування політик |
| **cert-manager** | Керування сертифікатами |

### 7. Сховище

| Інструмент | Що він робить |
|------------|---------------|
| **Rook** | Оркестрація сховища (Ceph на K8s) |
| **Longhorn** | Розподілене блочне сховище |
| **Velero** | Резервне копіювання та аварійне відновлення |

## Що вам дійсно потрібно знати

Для сертифікації та більшості вакансій зосередьтеся на:

### Обов'язково знати
- **Kubernetes** — сама платформа
- **Helm** — пакетне керування
- **Kustomize** — керування конфігурацією
- **kubectl** — інструмент командного рядка

### Варто знати (концептуально)
- **Prometheus/Grafana** — моніторинг
- **Концепції service mesh** — керування трафіком
- **Концепції CNI** — як працює мережа подів
- **Container runtime** — containerd, CRI

### Добре знати про (без глибоких знань)
- **ArgoCD/Flux** — GitOps
- **Istio/Linkerd** — реалізації service mesh
- **OPA/Gatekeeper** — політики
- **Falco/Trivy** — сканування безпеки

## Чи знали ви?

- **Ландшафт CNCF налічує понад 1000 проєктів.** Вивчити їх усі неможливо. Зосередьтеся на тому, що потрібно для вашої роботи.

- **Більшість компаній використовують ~10–20 проєктів CNCF.** Не сотні. Спеціалізація перемагає широту.

- **Сам Kubernetes — це приблизно 2 мільйони рядків коду.** Плюс мільйони в екосистемних проєктах. Ось чому сертифікації зосереджені на практичному використанні, а не на внутрішній будові.

- **Нові проєкти приєднуються до CNCF щомісяця.** Ландшафт постійно змінюється. Базові навички K8s залишаються стабільними; інструменти навколо нього змінюються.

```
CNCF Project Stages:

┌──────────┐
│ SANDBOX  │ ─── Early stage, experimental
└────┬─────┘     May not be production ready
     │
     ▼
┌──────────┐
│INCUBATING│ ─── Growing adoption
└────┬─────┘     Community forming
     │
     ▼
┌──────────┐
│ GRADUATED│ ─── Production proven
└──────────┘     Widely adopted, stable
```

Для продакшену надавайте перевагу проєктам зі статусом **Graduated**. Для навчання та експериментів досліджуйте **Incubating** і навіть **Sandbox**.

## Тест

1. **Що таке CNCF?**
   <details>
   <summary>Відповідь</summary>
   Cloud Native Computing Foundation — вендорно-нейтральна організація, яка опікується хмарно-нативними проєктами з відкритим кодом, зокрема Kubernetes, та керує ними. Вона є частиною Linux Foundation.
   </details>

2. **Яка різниця між Helm і Kustomize?**
   <details>
   <summary>Відповідь</summary>
   Helm — це пакетний менеджер із шаблонами: ви визначаєте змінні та шаблони, які рендеряться. Kustomize працює без шаблонів: ви пишете валідний YAML і накладаєте патчі. Обидва керують конфігурацією Kubernetes; підходи різні.
   </details>

3. **Що означає статус «Graduated» для проєкту CNCF?**
   <details>
   <summary>Відповідь</summary>
   Graduated-проєкти перевірені в продакшені та широко прийняті. Вони продемонстрували стабільність, широке впровадження та стійке керування спільнотою. Приклади: Kubernetes, Prometheus, Helm.
   </details>

4. **Навіщо використовувати Prometheus і Grafana разом?**
   <details>
   <summary>Відповідь</summary>
   Prometheus збирає та зберігає метрики, забезпечує сповіщення. Grafana надає візуалізацію та інформаційні панелі. Prometheus — це джерело даних; Grafana робить їх зрозумілими для людини.
   </details>

## Підсумок

Екосистема cloud native включає:

**Основна оркестрація**: Kubernetes, Helm, Kustomize
**Спостережуваність**: Prometheus, Grafana, Jaeger, Fluentd
**Мережа**: Cilium, Calico, Istio, Envoy
**Безпека**: Falco, Trivy, OPA
**CI/CD**: ArgoCD, Flux, Tekton

Ключові моменти:
- CNCF налічує понад 1000 проєктів
- Вам не потрібно вивчати їх усі
- Зосередьтеся на тому, що потрібно для сертифікації або роботи
- Graduated-проєкти найстабільніші
- Kubernetes — фундамент; усе інше будується на ньому
