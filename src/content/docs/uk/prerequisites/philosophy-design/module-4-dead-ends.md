## Чому цей модуль важливий

У 2018 році один ритейлер із Fortune 500 поставив 2,3 мільйона доларів на Apache Mesos як свою контейнерну платформу. Вони найняли команду з 8 спеціалістів з Mesos, побудували власні фреймворки та розгорнули 200+ сервісів. До 2020 року Mesos фактично помер — Apache Foundation відправив його на <!-- VERIFY: горище -->горище. Усю платформу довелося перебудувати на Kubernetes. Два роки роботи — викинуто. Експертиза інженерів з Mesos? За одну ніч знецінилася на ринку праці.

У технологіях знати, чого **НЕ** варто вчити, так само важливо, як знати, що вчити. Цей модуль — екскурсія технологічним цвинтарем — не щоб глузувати з мертвих, а щоб допомогти вам розпізнати тривожні ознаки й ніколи не поставити свою кар'єру на наступний Mesos.

## Цвинтар оркестрації контейнерів

### Docker Swarm

**Чим був**: Рідне рішення Docker для оркестрації.

**Статус**: Фактично застарілий. Docker Desktop видалив режим Swarm у 2022 році.

```
Timeline:
2015: Docker Swarm launched as K8s competitor
2017: Docker adds K8s support (admission of defeat)
2019: Docker Enterprise sold to Mirantis
2020: Mirantis announces Swarm deprecation timeline
2022: Swarm removed from Docker Desktop
```

**Чому загинув**:
- Обмежений набір функцій порівняно з K8s
- Єдиний виробник (Docker Inc.) проти спільнотного K8s
- Не зміг наздогнати зростання екосистеми K8s
- Корпоративні клієнти вимагали K8s

**Не вивчайте**: Концепції Swarm, мережу Swarm, визначення сервісів Swarm

**Виняток**: Docker Compose досі корисний для локальної розробки (не для оркестрації)

---

### Apache Mesos + Marathon

**Чим був**: Дворівневий планувальник ресурсів (Mesos) з оркестрацією контейнерів (Marathon).

**Статус**: Marathon покинутий. Mesos у режимі підтримки.

```
Timeline:
2009: Mesos created at UC Berkeley
2013: Marathon launched for containers
2016: Peak adoption (Twitter, Airbnb, Apple)
2020: Twitter announces migration to K8s
2021: Marathon declared unmaintained
2022: Mesos usage effectively zero for new projects
```

**Чому загинув**:
- Складність дворівневого планування
- Marathon ніколи не зрівнявся з функціями K8s
- Екосистема так і не сформувалася
- Ключові користувачі (Twitter) перейшли на K8s

**Не вивчайте**: Архітектуру Mesos, конфігурації Marathon, фреймворки Mesosphere/D2iQ

---

### Cloud Foundry (для оркестрації)

**Чим був**: Platform-as-a-Service, що з'явився до K8s.

**Статус**: Перейшов на роботу поверх K8s. Оригінальний підхід застарілий.

```
Timeline:
2011: Cloud Foundry launched by VMware
2015: CF Foundation formed
2019: CF begins running on K8s (KubeCF)
2022: CF pivots fully to K8s-based architecture
```

**Чому перейшов**:
- K8s став стандартом інфраструктури
- Не зміг конкурувати з екосистемою K8s
- «Якщо не можеш перемогти — працюй на їхній платформі»

**Не вивчайте**: BOSH (інструмент розгортання CF), Diego (оркестратор CF), специфічні концепції CF

## Цвинтар середовищ виконання контейнерів

### Docker як середовище виконання Kubernetes

**Чим був**: Оригінальне середовище виконання контейнерів для Kubernetes.

**Статус**: Видалений з Kubernetes у версії 1.24 (травень 2022).

```
Timeline:
2014-2016: Docker is THE way to run containers in K8s
2016: CRI (Container Runtime Interface) introduced
2017: containerd becomes CNCF project
2020: K8s announces dockershim deprecation
2022: K8s 1.24 removes dockershim completely
```

**Чому видалили**:
- Docker — це повноцінна платформа, а K8s потребував лише середовище виконання
- containerd (середовище виконання Docker) працює з K8s безпосередньо
- Стандартизація CRI зробила накладні витрати Docker непотрібними
- Покращення безпеки та продуктивності

**Не вивчайте**: Docker-специфічні конфігурації K8s, усунення проблем dockershim

**Досі актуально**: Docker для збирання образів. `docker build` і Dockerfile — це нормально.


## Цвинтар керування конфігурацією

### Chef/Puppet/Ansible для Kubernetes

**Чим вони були**: Інструменти керування конфігурацією для управління серверами.

**Статус**: Хибна парадигма для Kubernetes.

**Чому не підходять**:

| Традиційне КК | Рідний підхід Kubernetes |
|----------------|--------------------------|
| Змінювані сервери | Незмінні контейнери |
| SSH до серверів | Зміни через API |
| Збіжність до стану | Оголошення бажаного стану |
| Агент на кожному сервері | Агенти не потрібні |
| Імперативні скрипти | Декларативний YAML |

```
Traditional (Chef/Puppet/Ansible):
1. SSH to server
2. Check current state
3. Apply changes to reach desired state
4. Hope nothing drifted

Kubernetes:
1. Define desired state in YAML
2. kubectl apply
3. K8s continuously reconciles
4. Self-healing, no SSH
```

**Не вивчайте**: Використання Ansible для керування ресурсами K8s, рецепти Chef для K8s, маніфести Puppet для контейнерів

**Виняток**: Ansible/Terraform для розгортання самого кластера K8s (інфраструктура), а не для керування навантаженнями

---

### Docker Compose для продакшну

**Чим є**: Мультиконтейнерна оркестрація для Docker.

**Статус**: Чудово для розробки, неправильно для продакшну.

```
Docker Compose is perfect for:
- Local development
- CI/CD test environments
- Demo setups

Docker Compose is wrong for:
- Production orchestration
- Multi-node deployments
- High availability requirements
```

**Чому не масштабується**:
- Лише одна нода
- Немає самозцілення
- Немає поступових оновлень
- Немає вбудованого виявлення сервісів
- Немає керування секретами

**Не вивчайте**: Розгортання Compose у продакшн, Compose у режимі Swarm

## Чому ці технології загинули

Загальні патерни технологічних тупиків:

### Патерн 1: Єдиний виробник vs. Спільнота

```
Swarm:    Docker Inc. контролював → Обмежене впровадження
K8s:      Нейтральний CNCF → Загальноіндустріальне впровадження
Урок:     Спільнотне управління перемагає для інфраструктури
```

### Патерн 2: Складність без користі

```
Mesos:    Потужний, але складний → Обмежена екосистема
K8s:      Складний, але цінний → Масивна екосистема
Урок:     Складність прийнятна лише з пропорційною користю
```

### Патерн 3: Хибний рівень абстракції

```
Chef/Puppet: Рівень сервера → Не підходить для контейнерів
K8s:         Рівень контейнера → Ідеальне поєднання
Урок:        Зміна парадигми потребує нових інструментів
```

### Патерн 4: Ефекти екосистеми

```
Once K8s hit critical mass:
- Cloud providers built managed services
- Tool vendors targeted K8s
- Talent learned K8s
- Alternatives became unviable

Lesson: Network effects are powerful. Sometimes the best tech loses.
```

## Чи знали ви?

- **Mesos забезпечував усю інфраструктуру Twitter** на піку. У них було 100+ людей у команді Mesos. І вони все одно мігрували на Kubernetes.

- **Docker Inc. ледь не збанкрутувала.** Компанія, що започаткувала контейнерну революцію, не змогла забезпечити стійку бізнес-модель. Mirantis придбала Docker Enterprise.

- **Kubernetes мало не назвали «Seven of Nine»** (відсилання до Star Trek, як-от Borg). Обрали Kubernetes, бо це звучало професійніше.

- **Chef і Puppet досі прибуткові** для традиційного керування серверами. Вони просто не є правильним інструментом для хмарних навантажень.

## Як оцінювати нові технології

Коли з'являються нові технології, запитайте:

1. **Чи вирішує це справжню проблему?** Чи створює складність заради складності?

2. **Це спільнотний чи одновиробничий проєкт?** Спільнотне управління масштабується краще.

3. **Чи узгоджується з хмарними принципами?** Декларативний, керований через API, незмінний?

4. **Чи є момент екосистеми?** Інструменти, інтеграції, кадри?

5. **Чи впроваджують його великі хмарні провайдери?** Вони мають сильні сигнали щодо <!-- VERIFY: життєздатності -->життєздатності технологій.

## Тест

1. **Чому Docker було видалено як середовище виконання Kubernetes у версії 1.24?**
   <details>
   <summary>Відповідь</summary>
   Docker — це повноцінна платформа, тоді як K8s потребував лише компонент середовища виконання. containerd (базове середовище виконання Docker) працює з K8s напряму через CRI, усуваючи зайві накладні витрати Docker. Це покращило продуктивність і безпеку.
   </details>

2. **Чому Chef/Puppet/Ansible не підходять для моделі Kubernetes?**
   <details>
   <summary>Відповідь</summary>
   Вони розроблені для змінюваних серверів (зайди через SSH, застосуй зміни). Kubernetes використовує незмінні контейнери та декларативну конфігурацію через API. Парадигма принципово інша — K8s безперервно узгоджує бажаний стан, а не збігає при застосуванні.
   </details>

3. **Чому Mesos/Marathon програли Kubernetes, попри використання Twitter у масштабі?**
   <details>
   <summary>Відповідь</summary>
   Mesos був складним (дворівневе планування), Marathon ніколи не зрівнявся з функціями K8s, а екосистема так і не сформувалася. K8s мав нейтральне управління CNCF, підтримку хмарних провайдерів і масивне зростання екосистеми. Навіть Twitter мігрував на K8s.
   </details>

4. **Чи досі Docker релевантний? У якому сенсі?**
   <details>
   <summary>Відповідь</summary>
   Так, для збирання образів контейнерів. `docker build` і Dockerfile досі є основним способом створення образів контейнерів. Docker як середовище виконання для Kubernetes застарілий, але Docker для розробки та збирання образів залишається актуальним.
   </details>

## Вправа на рефлексію

Цей модуль навчає того, чого НЕ варто вчити — що не менш цінно, ніж знати, що вчити:

**1. Оцінка вкладених зусиль:**
- Чи інвестували ви час у технологію, яка потім стала застарілою?
- Як ви зрозуміли, що час рухатися далі?
- Що б ви зробили інакше?

**2. Розпізнавання патернів:**
- Модуль описує патерни технологій-невдах (єдиний виробник, хибна абстракція тощо)
- Чи можете ви визначити поточні технології з подібними тривожними ознаками?
- Як би ви оцінили «гарячий новий інструмент»?

**3. Питання Docker:**
- Чому люди досі кажуть «Docker», маючи на увазі «контейнери»?
- Чи варто досі знати Docker? (Так, для збирання образів)
- Який урок щодо розрізнення інструментів і концепцій?

**4. Наслідки для кар'єри:**
- Якби у вас була експертиза з Mesos у 2018 році, що б ви зробили?
- Як будувати навички, що залишаються цінними навіть при зміні конкретних інструментів?
- Чи «експерт з Kubernetes» — надійніша ставка, ніж «експерт з Docker Swarm»? Чому?

**5. Захист на майбутнє:**
- Що мало б змінитися, щоб Kubernetes став «тупиком»?
- Чи є нові технології, що можуть кинути виклик K8s?
- Як би ви помітили їх завчасно?

Здатність оцінювати технологічні ставки — це кар'єрна навичка, що виходить за межі будь-якого конкретного інструменту.

## Підсумок

Знати, чого уникати, економить час:

**Мертва оркестрація**:
- Docker Swarm (застарілий)
- Mesos/Marathon (покинутий)
- Cloud Foundry оригінальна модель (перейшов на K8s)

**Мертве середовище виконання**:
- Docker як середовище виконання K8s (видалений у K8s 1.24)

**Хибна парадигма**:
- Chef/Puppet/Ansible для навантажень K8s
- Docker Compose для продакшну

**Патерни невдач**:
- Контроль єдиного виробника
- Складність без пропорційної користі
- Хибний рівень абстракції
- Програш ефектам екосистеми

Зосередьте свій час на поточних, підтримуваних, широко впроваджених технологіях із сильним спільнотним управлінням.
