## Чому цей модуль важливий

Чергова інженерка в SaaS-компанії отримала виклик о 2-й ночі — критичний Под безперервно перезапускався. Напівсонна, вона зайшла на ноду через SSH і вручну перезапустила контейнер. Проблему вирішено, можна спати далі. Та от Kubernetes негайно зупинив її вручну запущений контейнер і замінив його новим — який теж почав перезапускатися. Вона перезапустила знову. Kubernetes знову зупинив. Упродовж 45 хвилин вона билася із системою, не усвідомлюючи, що **Kubernetes не зламався — вона мислила імперативно в декларативному світі.**

Щойно вона оновила YAML Деплойменту, виправивши справжню проблему конфігурації, і виконала `kubectl apply`, система зцілилася за 12 секунд. Ось різниця між імперативним і декларативним мисленням — і це найважливіша концепція в Kubernetes.

Якщо ви розумієте декларативне мислення, Kubernetes стає зрозумілим. Якщо ні — ви боротиметеся із системою замість того, щоб нею користуватися.

## Два підходи

### Імперативний: Скажіть системі, що робити

```bash
# Imperative approach
ssh server1
docker run -d nginx
docker run -d nginx
docker run -d nginx
# Check if they're running
docker ps
# If one dies, start another
# If traffic increases, run more
# If server fails, SSH somewhere else and repeat
```

Ви — цикл керування. Ви спостерігаєте, вирішуєте й дієте.

### Декларативний: Скажіть системі, чого ви хочете

```yaml
# Declarative approach
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx
```

```bash
kubectl apply -f nginx-deployment.yaml
# Done. Kubernetes handles the rest.
```

Kubernetes — це цикл керування. Він спостерігає, вирішує й діє — безперервно.

## Чому декларативний підхід перемагає

### 1. Самозцілення

```yaml
# You say: "I want 3 replicas"
spec:
  replicas: 3
```

Що відбувається, коли:
- Контейнер аварійно завершується? → K8s запускає новий
- Нода виходить з ладу? → K8s перепланує Поди на інші ноди
- Хтось вручну видаляє Под? → K8s створює його заново

Ви не писали жодної з цих логік. Ви просто оголосили, чого хочете.

### 2. Ідемпотентність

```bash
# Run this 100 times
kubectl apply -f deployment.yaml

# Result: Same state every time
# No duplicates, no conflicts, no "already exists" errors
```

Декларативні операції є **ідемпотентними** — повторне застосування тієї самої конфігурації дає однаковий результат.

### 3. Контроль версій і GitOps

```bash
# Your infrastructure is code
git log --oneline
a1b2c3d feat: scale web to 5 replicas
d4e5f6g fix: increase memory limit
g7h8i9j feat: add health checks

# Roll back infrastructure with git
git revert a1b2c3d
kubectl apply -f .
```

Декларативну конфігурацію можна версіонувати, рецензувати й аудитувати.

### 4. Виявлення дрейфу

```
Imperative world:
- Someone SSH'd in and made changes
- Documentation doesn't match reality
- "It works on my machine"
- Fear of touching production

Declarative world:
- Git is the source of truth
- K8s continuously enforces that truth
- Changes require PR review
- Confidence in deployments
```

## Імперативна пастка

Kubernetes має імперативні команди:

```bash
# These work, but...
kubectl run nginx --image=nginx
kubectl scale deployment nginx --replicas=5
kubectl set image deployment/nginx nginx=nginx:1.19
```

**Чому вони небезпечні:**

1. **Немає журналу аудиту**: Хто виконав цю команду? Коли?
2. **Немає рецензування**: Зміни обходять процес PR/рецензії
3. **Дрейф**: Стан системи не відповідає жодному файлу
4. **Невідтворюваність**: «Які команди ми виконували для налаштування?»

**Коли імперативний підхід допустимий:**
- Навчання та експерименти
- Діагностика (тимчасові зміни)
- Аварійні ситуації (з негайним оновленням декларативної конфігурації)

**Найкраща практика:**
```bash
# Generate YAML, don't apply directly
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml > deployment.yaml

# Review, commit, then apply
kubectl apply -f deployment.yaml
```

## Чи знали ви?

- **Цикл узгодження виконується кожні 10 секунд** за замовчуванням. Контролери постійно перевіряють, чи реальність відповідає бажаному стану.

- **Кожен ресурс Kubernetes є декларативним.** Поди, Сервіси, ConfigMap — усе це лише оголошення бажаного стану, які контролери узгоджують.

- **GitOps народився з цієї моделі.** ArgoCD і Flux розширюють декларативне мислення: Git стає джерелом істини, а K8s узгоджує стан із Git.

- **Terraform використовує ту саму модель.** Декларативна революція виходить за межі K8s. Сучасні інструменти для інфраструктури майже повсюдно декларативні.

## Типові помилки

| Помилка | Чому це шкодить | Рішення |
|---------|-----------------|---------|
| Використання імперативних команд у продакшні | Немає журналу аудиту, ризик дрейфу | Завжди використовуйте `kubectl apply -f` |
| Редагування живих ресурсів через `kubectl edit` | Зміни не потрапляють у Git | Оновлюйте YAML-файли, застосовуйте з Git |
| Нерозуміння безперервності змін | Здивування, коли K8s «скасовує» ручні зміни | Прийміть модель — змінюйте оголошення |
| Боротьба з циклом узгодження | Розчарування, обхідні шляхи | Працюйте разом із системою, а не проти неї |

## Зміна мислення

### Старе мислення (Імперативне)
```
"I need to deploy my app"
→ SSH to server
→ Pull code
→ Build
→ Start process
→ Configure nginx
→ Update firewall
→ Test
→ Document what I did
```

### Нове мислення (Декларативне)
```
"I need to deploy my app"
→ Define desired state in YAML
→ Commit to Git
→ kubectl apply (or let GitOps do it)
→ K8s handles the rest
→ Git IS the documentation
```

## Тест

1. **Що таке цикл узгодження?**
   <details>
   <summary>Відповідь</summary>
   Безперервний процес, у якому Kubernetes порівнює бажаний стан (ваш YAML) із поточним станом (те, що фактично працює) і вживає заходів для їхнього вирівнювання. Цей цикл працює завжди, а не лише коли ви застосовуєте зміни.
   </details>

2. **Чому імперативні команди небезпечні в продакшні?**
   <details>
   <summary>Відповідь</summary>
   Вони створюють дрейф між тим, що зберігається в Git, і тим, що працює. Немає журналу аудиту, процесу рецензування, а зміни неможливо відтворити. Стан системи не відповідає жодному файлу.
   </details>

3. **Що означає «ідемпотентний» і чому це важливо?**
   <details>
   <summary>Відповідь</summary>
   Операція є ідемпотентною, якщо її багаторазове виконання дає той самий результат, що й одноразове. `kubectl apply` є ідемпотентним — застосування одного й того самого YAML 100 разів дає той самий стан. Це робить автоматизацію безпечною і передбачуваною.
   </details>

4. **Що станеться, якщо вручну видалити Под із Деплойменту?**
   <details>
   <summary>Відповідь</summary>
   Kubernetes негайно створить новий Под на заміну. Контролер Деплойменту бачить, що поточний стан (2 Поди) не відповідає бажаному стану (3 Поди), і узгоджує їх, створюючи ще один Под.
   </details>

## Вправа на рефлексію

Цей модуль є концептуальним — поміркуйте над цими питаннями:

**1. Аналогія з термостатом:**
- Які інші системи у вашому житті працюють декларативно? (Круїз-контроль? Автояскравість?)
- Чому їх легше використовувати, ніж ручні аналоги?

**2. Ваш поточний робочий процес:**
- Як ви розгортаєте програмне забезпечення сьогодні?
- Це більше імперативний чи декларативний підхід?
- Що б змінилося, якби ви перейшли на інший підхід?

**3. Наслідки самозцілення:**
- Якщо Kubernetes «скасовує» ручні зміни, це можливість чи вада?
- Як це змінює ваш підхід до діагностики?

**4. Git та інфраструктура:**
- Чому зберігання інфраструктури як YAML у Git є потужним?
- На які запитання можна відповісти за допомогою `git log`, які раніше були недоступні?

**5. Зміна ментальної моделі:**
- Оператор запитує «які команди мені виконати?» Декларативний мислитель запитує «якого стану я хочу?»
- Яке питання веде до більш підтримуваних систем? Чому?

Ця зміна мислення — від «як» до «чого» — є найважливішою концепцією у всьому навчальному плані.

## Підсумок

Декларативна модель — це фундамент Kubernetes:

- **Бажаний стан**: Ви визначаєте, чого хочете (YAML)
- **Узгодження**: K8s безперервно приводить реальність у відповідність до бажаного
- **Самозцілення**: Збої автоматично виправляються
- **Ідемпотентність**: Застосовуйте ту саму конфігурацію багаторазово — результат буде однаковим
- **Контроль версій**: Ваша інфраструктура — це код у Git

Це не просто технічний вибір — це філософія, що уможливлює надійну, масштабовану та перевіряєму інфраструктуру.
