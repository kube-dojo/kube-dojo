## Чому цей модуль важливий

Була 3-тя година ночі в «Чорну п'ятницю» 2014 року. Розробник у стартапі електронної комерції, що швидко зростав, дивився на 47 вікон термінала, кожне з яких було підключене через SSH до іншого сервера. Їхній блискавичний розпродаж став вірусним — трафік у 10 разів перевищив очікуваний. Контейнери Docker падали швидше, ніж він встигав їх перезапускати. «Масштабуємо сервер 23... ні, зачекайте, 23 переповнений, спробуємо 31... 31 лежить, коли це сталося?» О 4-й ранку сайт вже не працював. У понеділок генеральний директор питав, чому вони втратили 340 000 доларів на продажах.

Цей кошмар розробника — саме те, заради чого існує Kubernetes. Не як академічна вправа, не як марнославний проєкт Google — а тому, що **ручне управління контейнерами в масштабі — це проблема, яка з'їсть вас живцем.**

Kubernetes переміг не випадково. Розуміння воєн оркестрації допомагає оцінити, чому існують певні патерни і чому альтернативи зазнали невдачі.

## Проблема: Оркестрація контейнерів

До 2014 року контейнери довели свою цінність. Docker зробив їх доступними. Але виникла нова проблема:

**Як запускати тисячі контейнерів на сотнях машин?**

Ручне управління не масштабується. Вам потрібні:
- Автоматизоване планування (де має працювати цей контейнер?)
- Самовідновлення (що стається, коли контейнер вмирає?)
- Масштабування (як впоратися з більшим трафіком?)
- Мережева взаємодія (як контейнери знаходять один одного?)
- Оновлення (як розгортати без простоїв?)

Це і є **оркестрація контейнерів**.

## Претенденти

### Docker Swarm

**Простий вибір**

Відповідь компанії Docker Inc. на оркестрацію. Вбудована в сам Docker.

```
Плюси:
- Простий у налаштуванні
- Нативна інтеграція з Docker
- Знайомий синтаксис Docker Compose
- «Просто працює» для невеликих розгортань

Мінуси:
- Обмежений набір функцій
- Погана підтримка мультихмарності
- Прив'язка до постачальника (vendor lock-in) Docker Inc.
- Обмеження масштабування
```

**Що сталося**: Docker Inc. поставила все на Swarm. Коли Kubernetes переміг, Docker Inc. зрештою змінила курс (її придбала компанія Mirantis у 2019 році). Swarm зараз фактично вважається застарілим.

### Apache Mesos + Marathon

**Вибір для підприємств**

Створений в Каліфорнійському університеті в Берклі, використовувався Twitter та Airbnb. Marathon забезпечував рівень оркестрації контейнерів поверх Mesos.

```
Плюси:
- Перевірений у бойових умовах у величезних масштабах (Twitter)
- Гнучкий (запускає контейнери ТА інші робочі навантаження)
- Дворівнева архітектура планування
- Перевірений у продакшені

Мінуси:
- Складний в експлуатації
- Крута крива навчання
- Менша екосистема
- Вимагав окремих компонентів (Marathon, Chronos)
```

**Що сталося**: Twitter відмовився від Mesos у 2020 році, перейшовши на Kubernetes. Компанія Mesosphere змінила назву на D2iQ і тепер продає... Kubernetes. Marathon покинуто.

### Kubernetes

**Вибір Google**

Внутрішня система Google «Borg» оркеструвала контейнери понад десять років. Kubernetes став наступником Borg з відкритим вихідним кодом, переданим до новоствореної організації CNCF.

```
Плюси:
- Десятирічний досвід Google
- Декларативна модель (бажаний стан)
- Величезна екосистема
- Хмарно-нативна (cloud-native) основа
- Сильне управління з боку спільноти

Мінуси:
- Складний
- Крута крива навчання
- «Занадто великий» для простих розгортань
```

**Що сталося**: Він переміг. Беззаперечно.

## Чому Kubernetes переміг

### 1. Декларативна модель

Kubernetes запропонував фундаментально інший підхід:

```
Імперативний підхід (Swarm/Традиційний):
"Запусти 3 контейнери nginx на server-1"
"Якщо один помре, запусти інший"
"Якщо трафік зросте, запусти ще 2"

Декларативний підхід (Kubernetes):
"Я хочу, щоб працювали 3 репліки nginx. Завжди."
(Kubernetes сам розбирається з рештою)
```

Цей зсув є фундаментальним. Ви описуєте *чого ви хочете*, а не *як цього досягти*. Kubernetes безперервно узгоджує реальність із вашим бажаним станом.

### 2. Досвід Google

Google понад десять років запускав контейнери в масштабі за допомогою Borg. Kubernetes втілив уроки, винесені з:
- Мільярдів розгортань контейнерів на тиждень
- Збоїв на кожному можливому рівні
- Того, що насправді працює в масштабі

Це не було першою спробою стартапу — це була система третього покоління від Google з відкритим вихідним кодом.

### 3. Ефект екосистеми

Kubernetes прийняв розумні архітектурні рішення:
- **Розширюваність**: Користувацькі визначення ресурсів (CRD) дозволяють будь-кому розширювати K8s
- **Модульність**: Середовища виконання контейнерів, мережева взаємодія, сховища — все підключається як модулі
- **Пріоритет API (API-first)**: Все є API, що відкриває можливості для створення інструментів

Це спричинило вибуховий ріст проєктів екосистеми. Сьогодні ландшафт CNCF налічує понад 1000 проєктів, побудованих на базі Kubernetes або навколо нього.

### 4. Адопція хмарними провайдерами

До 2017 року всі основні хмарні провайдери пропонували керований Kubernetes:
- Google Kubernetes Engine (GKE)
- Amazon Elastic Kubernetes Service (EKS)
- Azure Kubernetes Service (AKS)

Це було безпрецедентно. Конкуренти зазвичай не впроваджують технології один одного. Але Kubernetes був:
- З відкритим вихідним кодом (жоден постачальник ним не володіє)
- Під управлінням CNCF (нейтральної організації)
- Стандартом, що формувався (його неможливо було ігнорувати)

### 5. Спільнота та управління

Організація CNCF (Cloud Native Computing Foundation) забезпечила:
- Нейтральне управління (не контролюється Google)
- Незалежну від постачальників сертифікацію
- Чіткі правила для контрибуторів
- Захист торговельних марок

Це означало, що компанії могли інвестувати в Kubernetes, не боячись прив'язки до постачальника.

## Хронологія

```
2013: Docker запускається, контейнери стають масовими
2014: Google анонсує Kubernetes (червень)
      Анонсовано Docker Swarm
      Mesos/Marathon набирають популярність

2015: Випущено Kubernetes 1.0 (липень)
      Створено CNCF, Kubernetes передано туди
      Починаються «Війни оркестрації»

2016: Pokemon Go працює на Kubernetes (масове підтвердження)
      Всі основні хмари анонсують підтримку K8s

2017: Docker Inc. додає підтримку Kubernetes (капітуляція)
      Kubernetes стає стандартом де-факто

2018-2019: Відмова від Swarm, Mesos покинуто
           Домінування Kubernetes стає абсолютним

2020+: Фокус зміщується з «чи варто використовувати K8s?» на
       «як краще використовувати K8s?»
```

## Чи знали ви?

- **Kubernetes означає «стерничий» грецькою мовою.** Логотип — це штурвал корабля. Сім спиць символізують сімох перших розробників.

- **Borg був названий на честь раси з «Зоряного шляху» (Star Trek).** Внутрішній попередник Kubernetes у Google. Інші проєкти Google наслідували цю тенденцію: Omega (ще одна відсилка до «Зоряного шляху»).

- **Запуск Pokemon Go став віхою для K8s.** Коли гра вийшла у 2016 році, трафік перевищив очікуваний у 50 разів. Kubernetes автоматично масштабував бекенд. Це стало «доказом», який переконав багато великих підприємств.

- **Docker намагався купити Kubernetes.** До того як Kubernetes отримав відкритий вихідний код, Docker звертався до Google з пропозицією придбати його. Google відхилив пропозицію і натомість передав його CNCF.

## Типові хибні уявлення

| Хибне уявлення | Реальність |
|---------------|---------|
| «K8s переміг завдяки Google» | Google допоміг, але нейтральне управління було ключовим. AWS не прийняла б продукт, контрольований Google. |
| «Swarm програв через Docker» | Swarm програв, бо не міг зрівнятися з K8s за можливостями чи екосистемою. Проблеми компанії лише прискорили це. |
| «K8s — це завжди правильний вибір» | Для простих розгортань K8s може бути надмірним. Docker Compose досі має виправдані сценарії використання. |
| «Mesos мав гіршу технологію» | Mesos був потужним, але занадто складним. Технологія сама по собі не перемагає — екосистема та простота мають значення. |

## Чому ця історія важлива для навчання

Розуміння того, чому Kubernetes переміг, допоможе вам:

1. **Оцінити декларативний дизайн**: Цикл узгодження не є випадковим — це ключова інновація.

2. **Зрозуміти екосистему**: Знання того, чому існують CRD, допоможе вам ефективно їх використовувати.

3. **Уникати глухих кутів**: Ви не витрачатимете час на застарілі підходи.

4. **Говорити однією мовою**: На співбесідах та в роботі цей контекст демонструє глибше розуміння.

## Тест

1. **У чому полягає фундаментальна різниця між підходами Kubernetes і Docker Swarm?**
   <details>
   <summary>Відповідь</summary>
   Kubernetes використовує декларативну модель (ви визначаєте бажаний стан, K8s узгоджує його), тоді як Swarm був більш імперативним (ви віддаєте команди для внесення змін). Декларативна модель із безперервним узгодженням — це ключова інновація Kubernetes.
   </details>

2. **Чому всі основні хмарні провайдери запровадили Kubernetes, незважаючи на те, що його створив Google?**
   <details>
   <summary>Відповідь</summary>
   Kubernetes був переданий CNCF, нейтральній організації. Це означало, що жоден постачальник не контролював його. Хмарні провайдери могли інвестувати в Kubernetes та пропонувати його, не надаючи переваги конкуренту.
   </details>

3. **Яким був внутрішній попередник Kubernetes у Google?**
   <details>
   <summary>Відповідь</summary>
   Borg. Він керував виробничими навантаженнями Google понад десятиліття. Kubernetes врахував уроки, отримані з експлуатації Borg у величезних масштабах.
   </details>

4. **Чому декларативна модель вважається кращою для інфраструктури?**
   <details>
   <summary>Відповідь</summary>
   Декларативні системи здатні до самовідновлення. Ви визначаєте, чого хочете, і система постійно працює над підтримкою цього стану. Якщо щось ламається, воно автоматично відновлюється. Імперативні системи вимагають зовнішнього моніторингу та втручання.
   </details>

## Вправа для роздумів

Цей модуль концептуальний — CLI не потрібен. Поміркуйте над такими питаннями:

**1. Патерни впровадження технологій:**
- Як ви думаєте, чому хмарні провайдери прийняли Kubernetes, хоча його створив Google?
- Яку роль відіграла нейтральність CNCF?

**2. Декларативне осяяння (insight):**
- Чому вказівка «я хочу 3 репліки» ефективніша за «запусти 3 контейнери»?
- Що відбувається в кожній з моделей, коли трапляється збій?

**3. Динаміка екосистеми:**
- Чому наявність розширюваності (CRD) мала більше значення, ніж вбудованість усіх функцій?
- Чи можете ви згадати інші технології, які перемогли завдяки екосистемі, а не характеристикам?

**4. Уроки на майбутнє:**
- Що потрібно для того, щоб щось замінило Kubernetes?
- Які сигнали вказуватимуть на те, що K8s стає «глухим кутом»?

Ці питання допоможуть вам критично мислити щодо вибору технологій — навичка, яка переживе будь-який конкретний інструмент.

## Підсумок

Kubernetes переміг у війнах оркестрації через такі причини:
- **Декларативна модель**: Визначаєте бажаний стан, K8s розбирається з рештою
- **Досвід Google**: Десятирічний досвід роботи з продакшеном на прикладі Borg
- **Екосистема**: Розширювана архітектура сприяла створенню масштабної екосистеми інструментів
- **Управління**: Нейтральність CNCF уможливила впровадження в межах усієї індустрії
- **Підтримка хмар**: Усі основні провайдери запропонували керований K8s

Альтернативи зазнали невдачі не тому, що були поганими — вони програли, бо Kubernetes мав кращу позицію для впровадження в усій індустрії.
