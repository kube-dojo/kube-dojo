### Gemini API-key auth is now PRIMARY; OAuth is fallback

User direction (verbatim 2026-05-04): *"we have 1000 calls a day with api then we have to fall back to oauth"*.

- Old `GEMINI_API_KEY` (prefix `AIzaSy…EPl4`) returned `API_KEY_INVALID` from `generativelanguage.googleapis.com`. dispatch.py's old "API-key first / OAuth fallback" was a no-op the entire time — every call went to OAuth via `cloudcode-pa.googleapis.com` because the gemini CLI is hardwired to OAuth via `~/.gemini/settings.json` `selectedType: "oauth-personal"`.
- New API key generated by user via `aistudio.google.com/apikey` → added to **`~/.bash_secrets`** (existing pattern, sourced from both `.zshrc` and `.bash_profile`, multi-project home for personal API keys). Smoke-tested against `gemini-3-flash-preview`, `gemini-3.1-pro-preview`, `gemini-2.5-pro`, `gemini-2.5-flash` — all four return ok via REST.
- **`scripts/dispatch.py` adds `dispatch_gemini_rest()`** (urllib POST to `generativelanguage.googleapis.com/v1beta` using `GEMINI_API_KEY`). `dispatch_gemini_with_retry()` reordered: REST first if `GEMINI_API_KEY` is set and `mcp` not requested → fall back to gemini CLI subprocess (OAuth) on 429/5xx/network error. The `dispatch_gemini` (CLI subprocess) function's API-key branch is now largely dead code; docstring TODO marker added; cleanup deferred to a follow-up PR.
- Bug fix in same commit: when `dispatch_gemini_with_retry(review=True)` is called with the default `model=GEMINI_DEFAULT_MODEL`, the retry now promotes to `GEMINI_REVIEW_MODEL` before routing to REST (the REST path's own `if model is None` swap never triggered because the retry's default arg already substituted Flash). Caught by Gemini cross-family review on #882.

### Quality lesson learned — Gemini-drafted P1 frequently echoes concepts of preserved P2

5 Must-Fix findings on PR #882 from Gemini's own cross-family review:

- 4× **P1/P2 redundancy** — the Gemini drafter (gemini-3.1-pro-preview) often packed too much concept into the new P1, defining the module's core topic — which then read jarringly against the preserved P2 that defines the same topic. Examples: module-1.2-cis-benchmarks listed CIS audit vectors in P1 then P2 reintroduced CIS; module-5.3-static-analysis defined static analysis at end of P1 then P2 immediately defined static analysis again.
- 2× **marketing-tone phrases** — Gemini drifted toward "this critical component", "rigorous, defense-in-depth approach", "robust, customizable rules engine", "deep, granular signal", "deeply hidden threat indicators". The drafter ignored an explicit prompt rule banning these. The reviewer (also Gemini) caught all of them.

Mitigation for sweep #3 prompt: tighten the drafter brief — "P1 ends BEFORE the topic is introduced; P2 carries the topic introduction (which is preserved verbatim). Your P1 establishes the vulnerability/principle context only — do NOT define the module's tooling or concept in P1." Add explicit anti-marketing-phrase examples.
