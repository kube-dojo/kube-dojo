## Decisions and contract changes

### Hard rule — incident anecdotes are one-per-module, curriculum-wide

User direction (verbatim 2026-05-04): *"one example can be only once, if we cannot find other we can refer it where it appeared first."* Plus, for fabricated/templated stories (`GlobalTradeX`, `fintech + Black Friday + $X.X million`, etc.): *"write concept-led WTMM section instead. ... I want nice proper solution, which is still enjoyable to read but no bullshit, humans are very receptive for that. you cannot fool a human."*

This locks the policy that every named real-world incident appears in **at most ONE module** across the entire EN curriculum. The one module that owns the incident is its **canonical**. Non-canonical modules either (a) replace with a different real, primary-source-verifiable incident from the catalog, (b) cross-reference the canonical with `<!-- incident-xref: SLUG -->` HTML markers, or (c) drop the dramatic opener and write a concept-led WTMM section. Fabricated companies and templated holiday-outage stories are NEVER acceptable — they get option (c).

### CI guardrail — `scripts/check_incident_reuse.py`

New durable guardrail: scans every module file, fails if any tracked incident appears in more than ONE module unless the second mention carries an `<!-- incident-xref: SLUG -->` HTML comment within 200 chars. Imports the `INCIDENTS` regex catalog from `audit_incident_reuse.py` so both stay in sync. Wires alongside `check_site_health.py` and `check_reader_aids.py` once the user merges the PR.

Current state: **98 violations across 19 incidents** (down from 121 pre-sweep). Prereqs subtotal: **0**. Each future per-track sweep PR drops the count further; the final sweep gets it to 0 and the guardrail becomes a CI-required check.

### Operational pattern locked — Gemini-first sweep workflow

Per user direction 2026-05-04: *"try gemini for phase 2 first instead of codex-desktop for now, if you are not happy with gemini we can try codex-desktop, also wait for the catalog."*

The Phase 2 (per-track sweep) pattern is now:

1. Orchestrator (Claude) creates per-track worktree based on the most-recent incident-dedup branch (initially `main` after #880 merges).
2. Orchestrator builds a single Gemini drafting prompt enumerating each file's current incident, the chosen replacement (slug from catalog) or option (b)/(c), and the module's bridge sentence.
3. `scripts/dispatch.py gemini` returns drafted opening paragraphs as text.
4. Orchestrator integrates each draft via `Edit`, preserves paragraphs 2 and 3 of the WTMM verbatim, runs the guardrail + `npm run build` locally.
5. Orchestrator commits, pushes, opens PR, and dispatches a second fresh-context Gemini call as cross-family adversary reviewer.
6. Orchestrator addresses review findings, re-runs guardrail + build, comments on PR.

Same-family caveat: the drafter and the reviewer are both Gemini. This is a partial-priors risk. During this session, Gemini-as-reviewer caught four substantive issues in #879 (regex false positives × 2, primary source quality, prompt template gap) and one stylistic nit in #880 — strong evidence that the gemini-first path is producing real review value. When codex returns ~11:00 2026-05-05, codex review takes over for any remaining sweep PRs as the cleanest cross-family signal.
